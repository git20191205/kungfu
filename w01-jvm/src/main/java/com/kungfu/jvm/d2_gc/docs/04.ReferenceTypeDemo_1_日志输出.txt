========================================
  四种引用类型实战演示
========================================

=== 一、强引用（Strong Reference）===
  场景：日常写的 Object obj = new Object() 就是强引用

  创建 2MB 强引用对象
  GC 后对象是否存活: true
  置 null + GC 后才被回收

  → 强引用的风险：忘记置 null → 内存泄漏
    经典场景：static List 不断 add 却不 remove

=== 三、弱引用（Weak Reference）===
  场景：WeakHashMap、ThreadLocal 的 key

  创建 1MB 弱引用对象
  GC 前是否存活: true
  GC 后是否存活: false ← 一定是 false！

  --- WeakHashMap 实战场景 ---
  WeakHashMap 存入数据，size: 1
  key 还在 + GC 后，size: 1
  key = null + GC 后，size: 0 ← 自动清理了！

  → 弱引用 vs 软引用：软引用「内存不够才回收」，弱引用「GC 就回收」
  → WeakHashMap：key 没有外部强引用时，条目自动清理，防止内存泄漏
  → ThreadLocal 的 key 也是弱引用，但 value 不是 → 所以要手动 remove()！

=== 四、虚引用（Phantom Reference）===
  场景：追踪对象被 GC 回收的时机，DirectByteBuffer 堆外内存回收

  phantomRef.get(): null ← 永远是 null
  GC 后 ReferenceQueue 中是否收到通知: true
  收到的引用是不是 phantomRef: true

  → 虚引用唯一的用途：对象被回收时收到通知
  → 经典应用：DirectByteBuffer 的 Cleaner 机制
    NIO 分配堆外内存 → 用虚引用追踪 → 对象被 GC 时触发堆外内存释放
  → 日常开发很少直接用，但面试会考原理

=== 二、软引用（Soft Reference）===
  场景：图片缓存、计算结果缓存 — 内存够就留着，不够就扔

  创建 2MB 软引用对象（模拟缓存图片）
  缓存是否可用: true
  内存充足 + GC 后，缓存是否还在: true

  开始疯狂分配内存，制造内存压力...
  内存紧张后，缓存是否还在: true

  → 软引用 = 「内存够就缓存，不够就丢弃」
  → 比手动管理缓存过期更优雅，JVM 自动帮你决定何时清理
  → 实战：Android 图片加载框架、本地计算缓存

========================================
  四种引用对比速记：
  ┌──────────┬────────────┬──────────────────────┐
  │ 引用类型  │ GC 回收时机  │ 典型场景              │
  ├──────────┼────────────┼──────────────────────┤
  │ 强引用    │ 不回收       │ Object o = new O()   │
  │ 软引用    │ 内存不够时   │ 图片缓存、本地缓存     │
  │ 弱引用    │ GC 就回收    │ WeakHashMap/ThreadLocal│
  │ 虚引用    │ GC 就回收    │ 堆外内存回收通知       │
  └──────────┴────────────┴──────────────────────┘

  面试追问：ThreadLocal 为什么会内存泄漏？
  → ThreadLocalMap 的 key 是弱引用（GC 会回收）
  → 但 value 是强引用（GC 不会回收）
  → key 被回收后 value 变成孤儿 → 内存泄漏
  → 解决方案：用完必须调用 threadLocal.remove()
