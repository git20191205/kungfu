  关键知识点

  为什么 int 和 long 循环耗时差不多？

  这里只有一个线程在跑，没有 GC 并发发生，所以看不出安全点的影响。安全点的坑是这样暴露的：

  线程 A: for (int i=0; i<巨大数; i++) { 计算... }   ← 没有安全点，一直跑
  线程 B: 正常业务
  GC 线程: 需要 STW → 线程 B 秒到安全点 → 等线程 A...等...等...

  结果： GC 的 STW 时间不是取决于 GC 本身，而是取决于线程 A 什么时候跑完循环到达安全点。这在线上就表现为「某次 GC 停顿异常地长」。

  面试这样答

  ▎ Q: 什么是 SafePoint？

  ▎ 安全点是代码中特定的位置（方法调用、循环回跳、异常跳转），线程必须跑到安全点才能响应 GC 的 STW 请求。JVM 用主动式中断：设一个标志位，线程在安全点轮询标志，看到后自己暂停。

  ▎ Q: 实际遇到过 SafePoint 的问题吗？

  ▎ int 类型的可数循环可能不被 JIT 放置安全点。如果循环耗时很长，GC 要等这个线程跑完才能 STW，导致停顿异常。解决方案：改用 long 循环、拆分循环、或在循环内加 Thread.sleep(0)（它是一个安全点）。
