  3. GC 演进的三个阶段

  阶段一：纯 Young GC（0→56000 任务）
  - 大量 Young GC，每次 8224K→32K，回收非常干净
  - 老年代缓慢增长：1KB→5KB→8KB...
  - 每次 YGC 约 0.3-0.5ms，程序运行流畅

  阶段二：Young GC + 偶尔 Full GC（56000→约200000任务）
  - 老年代逐渐填满（任务对象的 Runnable/byte[] 被队列持有，晋升到老年代）
  - 第一次 Full GC（第462行）：ParOldGen 20329K→3926K，回收了 80%！
  - 但每次 Full GC 后回收量越来越少：3926K→5565K→7075K→8799K...

  阶段三：Full GC 风暴（约800000+ 任务）
  ParOldGen: 20479K→20479K（老年代 100% 满，回收不了）
  PSYoungGen: 8186K→1415K（新生代还能勉强回收一些）
  每次 Full GC 耗时从 56ms 飙到 350-370ms，user 时间高达 6-7 秒（多核并行）

  5. 为什么 Full GC 能回收一部分但不 OOM？

  任务队列中的任务是被线程池消费的：
  - 线程池有 3 个线程，每个任务执行 15 秒
  - 执行完的任务会被从队列移除 → 对应的 byte[] 可回收
  - 生产速度 >> 消费速度，但消费一直在进行

  所以每次 Full GC 能回收刚消费完的几个任务（约 1-2MB），刚好不触发 GC overhead limit：
  - 回收了 >2% → 不满足 overhead 条件
  - 分配能成功（勉强） → 不触发 Java heap space

  6. 老年代存活量的缓慢爬升

  观察 Full GC 后老年代的残留：
  第462行:  3926K (3.8MB)
  第737行:  5565K (5.4MB)
  第1189行: 8799K (8.6MB)
  第1878行: 13162K (12.9MB)
  第3400行: 20479K (20MB，满了) → 回收后 ~21KB
  老年代在缓慢但持续地增长，队列中的任务对象越积越多。

  7. 性能灾难数据

  后期的 Full GC：
  - 每次 370ms 实际时间，6-7 秒 CPU 时间（20 个 GC 线程并行）
  - 每 500 个任务（约 1 秒）触发一次 Full GC
  - GC 时间占比接近 30-50% — CPU 大量用于 GC 而非业务

  8. 生产环境的危害

  这是最危险的 OOM 模式：
  1. 不会快速失败 — 监控看不到 OOM，不会触发告警
  2. 慢性性能退化 — 接口响应时间从 10ms 涨到 500ms+
  3. CPU 飙高 — 大量 CPU 用于 Full GC（user=7s 就是明证）
  4. 最终堆满进程假死 — 99% 的堆被队列占用，几乎无法处理新请求

  这就是为什么阿里巴巴开发手册禁止使用 Executors.newFixedThreadPool() — 底层是 LinkedBlockingQueue(Integer.MAX_VALUE)，即无界队列。

  ❯ 不是你还是没有说这个案例是演示啥的

  ● 这个案例演示的是：Executors.newFixedThreadPool() 的无界队列陷阱。

    生产中的真实场景：
    1. 用 Executors.newFixedThreadPool(N) 创建线程池
    2. 底层是 new LinkedBlockingQueue<>()，队列容量 = Integer.MAX_VALUE（约 21 亿）
    3. 请求高峰时，任务提交速度 > 线程消费速度
    4. 任务在队列中无限堆积，每个任务携带请求数据（body、附件等）
    5. 队列吃光堆内存 → OOM

    核心教训：永远不要用 Executors 工厂方法，要手动创建 ThreadPoolExecutor，指定有界队列 + 拒绝策略。

    但当前代码用的是 ArrayList 模拟队列，没有真正的线程池，只是简化演示"无限往队列塞数据"这个本质。
