GCTuningMethodologyDemo 日志分析

整体结果：Demo 在 30MB 小堆下运行 2000 次模拟请求，触发 47 次 Young GC + 3 次 Full GC，
          GC 频率过高，验证了「堆太小 → GC 频繁」的典型问题。

逐步分析：

  │ 阶段             │ 预期结果                     │ 状态   │

  │ 基线采集         │ GC 次数 = 0                  │ ✅     │
  │ 负载执行         │ 产生大量临时对象 + 缓存数据  │ ✅     │

重点观察 1 — Young GC 频率分析：

  日志原文：
    Young GC 频率: 1127.3 次/分钟 ← ★ 过高！建议扩大新生代

  分析：
    - 30MB 堆，NewRatio=2 → 新生代约 10MB，Eden 约 8MB
    - 每次请求产生 5KB 临时对象（5 × 1KB）
    - 2000 次请求 = 10MB 临时对象
    - Eden 仅 8MB，几乎每轮都会触发 Young GC
    - 健康阈值：≤ 10次/分钟，实际超标 100 倍以上

    优化方案：
    - 增大堆到 100MB，NewRatio=1 → 新生代约 50MB，Eden 约 40MB
    - 预期 Young GC 降至个位数

重点观察 2 — Full GC 出现原因：

  日志原文：
    Full GC 次数: 3 次 ← ★ 有 Full GC！需关注

  分析：
    - 老年代约 20MB（30MB 堆 × 2/3）
    - 缓存 20 个热点数据，每个 10KB = 200KB（不大）
    - Full GC 根因：Young GC 过于频繁，Survivor 区太小，
      临时对象来不及回收就被晋升到老年代
    - 这是典型的「对象过早晋升」问题

    关键指标链：
    Eden 太小 → YGC 频繁 → Survivor 容不下 → 对象直接晋升老年代
    → 老年代快速填满 → 触发 Full GC

重点观察 3 — GC 开销评估：

  日志原文：
    GC 总开销: 4.52% ✓

  分析：
    - GC 开销 = (GC 总耗时 / 业务总耗时) × 100%
    - (68ms + 45ms) / 2501ms = 4.52%
    - 虽然在 5% 以下，但这是因为业务逻辑简单
    - 真实生产环境中如果 GC 频率这么高，
      接口 P99 延迟会显著上升（GC 停顿叠加到请求链路上）
    - 建议生产环境 GC 开销控制在 1% 以下

  验证方法：用 -Xms100m -Xmx100m -XX:NewRatio=1 重新运行 Demo

结论：
  1. GC 调优的核心是「量化」—— 没有指标对比，无法判断调优是否有效
  2. 30MB 小堆下 Young GC 频率严重超标，是因为 Eden 区太小
  3. Full GC 的根因是对象过早晋升，而非真正的老年代数据量大
  4. 调优方向明确：扩大堆 + 增大新生代比例
  5. 下一步：学习 D5 MemoryLeakDetectionDemo，区分「堆太小」和「内存泄漏」
