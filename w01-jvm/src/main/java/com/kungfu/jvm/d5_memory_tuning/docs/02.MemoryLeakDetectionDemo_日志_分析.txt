MemoryLeakDetectionDemo 日志分析

整体结果：Demo 演示了 3 种经典内存泄漏模式，每种模式都展示了泄漏过程、
          GC 后的内存状态、泄漏特征和修复方案。全部符合预期。

逐步分析：

重点观察 1 — static 集合泄漏的内存增长曲线：

  日志原文：
    第 1 轮: 缓存 200 条, 堆使用 4.2 MB
    第 2 轮: 缓存 400 条, 堆使用 5.1 MB
    ...
    第 5 轮: 缓存 1000 条, 堆使用 7.6 MB
    手动 GC 后: 缓存仍有 1000 条, 堆使用 7.1 MB

  分析：
    - 每轮增加 200 条 × 0.5KB = 100KB
    - 堆使用量呈线性上升：4.2 → 5.1 → 5.9 → 6.8 → 7.6 MB
    - GC 后从 7.6 降到 7.1，仅回收了 0.5MB 临时对象
    - 缓存数据 (1000 × 0.5KB = 500KB) 完全无法回收
    - 这就是泄漏的经典特征：Full GC 后堆使用率不回落

    生产环境中这种模式更隐蔽：
    - 每天增长几十 MB，可能运行一周才触发 OOM
    - 需要通过长期监控堆使用率趋势来发现

重点观察 2 — WeakHashMap 的自动回收效果：

  日志原文：
    写入 1000 条后: WeakHashMap size = 1000
    GC 后: WeakHashMap size = 0 ← key 无强引用时自动回收

  分析：
    - WeakHashMap 的 key 用 WeakReference 包装
    - 当 key 对象没有其他强引用时，GC 会回收 key
    - key 被回收后，对应的 entry 在下次操作时被清理
    - 注意：必须用 new String() 创建 key，字面量字符串在常量池中有强引用
    - WeakHashMap 适合「缓存」场景，不适合需要精确控制的场景

    使用限制：
    - 不支持并发（需要 Collections.synchronizedMap 包装）
    - 无法控制淘汰策略（只能基于可达性）
    - 生产建议：用 Caffeine Cache 替代（支持 LRU + TTL + 并发）

重点观察 3 — 监听器泄漏的引用链：

  日志原文：
    第 5 批: 注册 100 个监听器, 堆使用 7.8 MB
    → 虽然组件 "销毁" 了，但 LISTENERS 列表仍持有引用

  分析：
    引用链：
    GC Root (static field LISTENERS)
      → ArrayList
        → HeavyComponent[0] (50KB)
        → HeavyComponent[1] (50KB)
        → ... (共 100 个)

    总泄漏量：100 × 50KB = 5MB
    即使 HeavyComponent 在业务层面已经 "不用了"，
    但 LISTENERS 持有引用 → GC 无法回收

    这在 Spring 中容易出现的场景：
    - 手动 addApplicationListener 但没有 remove
    - 自定义 EventBus 的 register/unregister 不对称
    - 解决：Spring 的 @EventListener 注解自动管理生命周期

重点观察 4 — ThreadLocal 泄漏的根本原因：

  日志原文：
    线程池的线程不会被销毁 → Thread 对象一直存活
    Thread 持有 ThreadLocalMap → Map 中的 Entry 不会回收

  分析：
    ThreadLocal 内部结构：
    Thread
      └── threadLocals (ThreadLocal.ThreadLocalMap)
            └── Entry[] table
                  └── Entry extends WeakReference<ThreadLocal<?>>
                        ├── referent (key) → WeakReference → 可能被 GC
                        └── value → 强引用 → 不会被 GC！

    泄漏路径：
    1. ThreadLocal 变量被 GC 回收 → Entry.key 变成 null
    2. 但 Entry.value 仍被 Thread 强引用
    3. 线程池中线程不销毁 → value 永远无法回收
    4. 这就是所谓的 "key 为 null 的废弃 Entry"

    ThreadLocal.remove() 做了什么？
    - 从 ThreadLocalMap 中删除当前 ThreadLocal 对应的 Entry
    - 同时清理其他 key 为 null 的废弃 Entry（启发式清理）
    - 所以 remove() 不仅清理自己，还顺带清理其他泄漏

结论：
  1. 三种泄漏模式覆盖了生产中 80% 的内存泄漏场景
  2. 核心检测手段：jmap -histo 对比 + MAT Leak Suspects
  3. 泄漏的本质：对象的引用关系超过了其业务生命周期
  4. 防御措施：使用容量限制的缓存、在 finally 中清理资源、代码审查
  5. 下一步：D6 学习 MAT 工具的具体使用方法和非堆 OOM 排查
