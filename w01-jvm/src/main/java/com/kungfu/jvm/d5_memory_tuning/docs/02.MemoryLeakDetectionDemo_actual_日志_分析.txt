MemoryLeakDetectionDemo 实际运行日志分析

运行环境：JDK 1.8.0_441, Windows 11, -Xms50m -Xmx50m

整体结果：3 种泄漏模式全部演示成功，WeakHashMap 对比效果明显。

与编造日志的关键差异：

  ┌────────────────────┬──────────────┬──────────────┐
  │ 数据点             │ 编造值       │ 实际值       │
  ├────────────────────┼──────────────┼──────────────┤
  │ 模式一 第1轮堆使用 │ 4.2 MB       │ 1.3 MB       │
  │ 模式一 第5轮堆使用 │ 7.6 MB       │ 2.3 MB       │
  │ GC后堆使用         │ 7.1 MB       │ 1.6 MB       │
  │ 模式二 第1批堆使用 │ 3.8 MB       │ 1.7 MB       │
  │ 模式二 第5批堆使用 │ 7.8 MB       │ 5.6 MB       │
  │ WeakHashMap GC后   │ 0            │ 0            │
  └────────────────────┴──────────────┴──────────────┘

重点观察 1 — 堆使用量比编造的低很多：

  模式一 static 集合泄漏：
  - 编造：4.2 → 7.6 MB（每轮 +0.8 MB）
  - 实际：1.3 → 2.3 MB（每轮 +0.25 MB）

  原因：
  - 1000 条 × 512B = 500KB 缓存数据
  - HashMap 结构开销大约 50%（Node 对象 + 空槽位）
  - 总占用约 750KB ≈ 0.7MB
  - 编造时严重高估了 JVM 基线内存开销（实际 50MB 堆下基线 ≈ 1MB）

重点观察 2 — 模式二监听器泄漏的线性增长：

  实际数据完美展示了线性泄漏：
    第 1 批: 1.7 MB
    第 2 批: 2.7 MB（+1.0）
    第 3 批: 3.7 MB（+1.0）
    第 4 批: 4.7 MB（+1.0）
    第 5 批: 5.6 MB（+0.9）

  每批 20 个组件 × 50KB = 1MB → 完美匹配！
  这种「每批增长固定量、GC 后不回落」就是泄漏的标志性特征

重点观察 3 — WeakHashMap 行为完全符合预期：

  写入 1000 条后: size = 1000
  GC 后: size = 0

  实际和编造一致。WeakHashMap 的行为是确定性的：
  - 用 new String() 创建 key → 无外部强引用
  - System.gc() 触发 → WeakReference 被回收
  - 下次 get/put 时清理废弃 Entry

重点观察 4 — GC 后堆使用量的恢复：

  模式一 GC 后：1.6 MB（低于第 1 轮的 1.3 MB + 泄漏量）

  说明 GC 回收了循环过程中的临时对象，但保留了 LEAKY_CACHE 中的数据。
  如果运行更多轮（如 100 轮），堆使用会持续上升直到 OOM。

结论：
  1. 编造的堆使用量偏高约 3 倍，主要是高估了 JVM 基线开销
  2. 模式二的每批 +1MB 线性增长与理论值（20 × 50KB）完美匹配
  3. WeakHashMap 行为是确定性的，编造和实际一致
  4. Demo 的核心教学价值不受数字偏差影响 — 模式和趋势是对的
