重点观察 1 -- JDK 动态代理方法拦截与代理对象特征：

  日志原文：
    [JDK Proxy] 前置: pay 开始
    [PayService] 执行支付: orderId=ORD-001, amount=99.9
    [JDK Proxy] 后置: pay 结束
  返回: PAY-SUCCESS

    代理类名: com.kungfu.spring.d18_aop.$Proxy0
    是否为接口实例: true
    是否为实现类实例: false

  分析：
  - JDK 动态代理通过 InvocationHandler 在 pay 方法前后插入了拦截逻辑
  - 业务方法正常执行，传入 orderId=ORD-001, amount=99.9，返回 PAY-SUCCESS
  - 代理类名为 $Proxy0，这是 JDK 运行时自动生成的类名（$Proxy + 序号）
  - 代理对象是接口实例（true），但不是实现类实例（false）
  - 核心特征：代理对象实现了目标接口，但与目标实现类无继承关系

重点观察 2 -- CGLIB 代理方法拦截与代理对象特征：

  日志原文：
    [CGLIB Proxy] 前置: send 开始
    [NotificationService] 发送通知: 订单已发货
    [CGLIB Proxy] 后置: send 结束
  返回: SENT

    代理类名: com.kungfu.spring.d18_aop.ProxyPatternDemo$NotificationService$$EnhancerByCGLIB$$5f126743
    是否为原始类实例: true
    父类: NotificationService

  分析：
  - CGLIB 通过 MethodInterceptor 在 send 方法前后插入了拦截逻辑
  - 业务方法正常执行，发送通知内容为"订单已发货"，返回 SENT
  - 代理类名包含 $$EnhancerByCGLIB$$ 标识和哈希值 5f126743
  - 代理对象是原始类实例（true），因为 CGLIB 通过继承目标类生成子类
  - 父类为 NotificationService，直接证实了"基于继承"的代理模式

重点观察 3 -- 代理类命名规律：

  JDK Proxy:  com.kungfu.spring.d18_aop.$Proxy0
  CGLIB:      ...ProxyPatternDemo$NotificationService$$EnhancerByCGLIB$$5f126743

  命名规律：
    JDK Proxy  -> $Proxy + 序号（从 0 开始递增，全局计数）
    CGLIB      -> 原始类全限定名 + $$EnhancerByCGLIB$$ + 哈希值

  通过类名即可判断使用了哪种代理：
    包含 "$Proxy"                    -> JDK 动态代理
    包含 "$$EnhancerByCGLIB$$"       -> 原生 CGLIB
    包含 "$$EnhancerBySpringCGLIB$$" -> Spring 内嵌 CGLIB（见 Demo03）

重点观察 4 -- instanceof 行为差异与实际影响：

  JDK Proxy:
    是否为接口实例: true      （代理类实现了目标接口）
    是否为实现类实例: false    （代理类与实现类没有继承关系）

  CGLIB:
    是否为原始类实例: true     （代理类是目标类的子类）
    父类: NotificationService  （继承关系确认）

  实际影响：
  JDK Proxy 代理的对象只能按接口类型注入，不能按实现类注入。
  CGLIB 代理的对象可以按目标类类型注入。

  典型问题场景：
    @Autowired
    private PayServiceImpl payService;   // JDK Proxy 时报错！类型不匹配
    @Autowired
    private PayService payService;       // 正确写法，按接口注入

  这也是 SpringBoot 2.0+ 默认切换到 CGLIB 的原因之一。

重点观察 5 -- Spring 代理选择策略（决策树）：

  日志原文：
    proxyTargetClass = true ?
    ├── 是 -> 使用 CGLIB（SpringBoot 2.0+ 默认）
    └── 否
         ├── 目标类实现了接口？
         │   ├── 是 -> 使用 JDK Proxy
         │   └── 否 -> 使用 CGLIB
         └── 目标类是 final？
              └── 是 -> 无法代理，抛出异常

  配置方式：
    @EnableAspectJAutoProxy(proxyTargetClass = true)   // 注解方式
    spring.aop.proxy-target-class=true                 // SpringBoot 配置（默认 true）

  分析：
  - SpringBoot 2.0+ 默认 proxyTargetClass=true，即默认使用 CGLIB
  - 原因：避免 JDK Proxy 的 BeanNotOfRequiredTypeException
  - Spring 内嵌了 CGLIB（spring-core 包含），无需额外依赖
  - 限制：final 类无法被 CGLIB 代理（无法继承），final 方法无法被重写

重点观察 6 -- JDK Proxy vs CGLIB 完整对比验证：

  日志中的对比表（8 项维度）：
  ┌──────────────┬─────────────────────────┬───────────────────────────┐
  │ 对比项       │ JDK 动态代理            │ CGLIB 代理                │
  ├──────────────┼─────────────────────────┼───────────────────────────┤
  │ 实现方式     │ 基于接口（Proxy类）     │ 基于继承（ASM字节码）     │
  │ 要求         │ 目标类必须有接口        │ 目标类不能是 final         │
  │ 代理对象关系 │ 实现相同接口            │ 目标类的子类              │
  │ 拦截方式     │ InvocationHandler       │ MethodInterceptor         │
  │ 依赖         │ JDK 内置                │ 需要引入 cglib 库         │
  │ final 方法   │ 不影响（接口无 final）  │ final 方法无法代理        │
  │ 创建速度     │ 较快                    │ 较慢（字节码生成）        │
  │ 调用速度     │ 反射调用                │ FastClass 直接调用        │
  └──────────────┴─────────────────────────┴───────────────────────────┘

  本 Demo 的运行结果逐一验证了上表中的理论差异。

结论：
  JDK Proxy 和 CGLIB 是 Spring AOP 的两大代理引擎。
  JDK Proxy 基于接口实现，代理类名为 $Proxy0，代理对象是接口实例但不是实现类实例；
  CGLIB 基于继承，代理类名含 $$EnhancerByCGLIB$$ 标识，代理对象是目标类的子类实例。
  SpringBoot 2.0+ 默认选择 CGLIB 是为了解决 JDK Proxy 的类型兼容性问题。
  面试速记："有接口可以 JDK Proxy，无接口必须 CGLIB，Boot 默认 CGLIB"。
