01.MiniCircularDepDemo 日志分析
========================================

整体结果：手写三级缓存成功解决 A↔B 循环依赖，最终 a.serviceB.serviceA == a 为 true，证明引用一致。

逐步分析：

┌──────┬──────────────────────────────────────┬───────────────────────────────────────────────┬──────────────────┐
│ 步骤 │ 操作                                 │ 缓存状态变化                                  │ 状态             │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│  1   │ getBean("serviceA")                  │ 三级缓存为空，进入 createBean                 │ 开始             │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│  2   │ new ServiceA()                       │ 实例化完成，A 是半成品（字段=null）            │ ✅ 实例化        │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│  3   │ singletonFactories.put("serviceA")   │ 三级缓存: {serviceA: ObjectFactory}           │ ✅ 放入三级      │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│  4   │ populate(A) → getBean("serviceB")    │ A 需要 B，递归进入 getBean                    │ 递归↓            │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│  5   │ new ServiceB()                       │ 实例化完成，B 是半成品（字段=null）            │ ✅ 实例化        │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│  6   │ singletonFactories.put("serviceB")   │ 三级缓存: {serviceA: OF, serviceB: OF}        │ ✅ 放入三级      │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│  7   │ populate(B) → getBean("serviceA")    │ B 需要 A，再次递归 getBean("serviceA")        │ 递归↓↓           │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│  8   │ getSingleton("serviceA") 命中三级    │ 执行 ObjectFactory → 升级到二级缓存            │ ⚠ 缓存命中！     │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│  9   │ B.serviceA = 早期引用A               │ B 的属性填充完成                               │ ✅ B 注入完成    │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│ 10   │ singletonObjects.put("serviceB")     │ B 升入一级缓存（成品）                         │ ✅ B 完成        │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│ 11   │ A.serviceB = 成品B                   │ A 的属性填充完成                               │ ✅ A 注入完成    │
├──────┼──────────────────────────────────────┼───────────────────────────────────────────────┼──────────────────┤
│ 12   │ singletonObjects.put("serviceA")     │ A 升入一级缓存（成品）                         │ ✅ A 完成        │
└──────┴──────────────────────────────────────┴───────────────────────────────────────────────┴──────────────────┘

重点观察 1 — 三级缓存结构定义（日志原文）：

  日志开头打印了完整的三级缓存结构：
    "一级缓存：singletonObjects      <beanName, 完整的Bean>"
    "二级缓存：earlySingletonObjects <beanName, 半成品Bean>"
    "三级缓存：singletonFactories    <beanName, ObjectFactory>"

  查找流程也有明确记录：
    "查一级缓存 → 有则返回"
    "查二级缓存 → 有则返回"
    "查三级缓存 → 执行 ObjectFactory.getObject() → 升级到二级缓存"

  这正是 DefaultSingletonBeanRegistry 中 getSingleton(beanName, allowEarlyReference) 的逻辑。
  三级之间是"逐级降级查找、命中后升级"的关系。

重点观察 2 — 循环依赖解决的关键转折点：

  日志关键行：
    "[创建开始] serviceB"
    "[3.属性填充] 开始注入 serviceB 的依赖字段"
    "  → 需要注入 serviceA，递归 getBean("serviceA")"
    "  ↳ 三级缓存命中: 执行 ObjectFactory 生成 serviceA"
    "    [ObjectFactory] 生成早期引用（可能被包装为AOP代理）"
    "  ↳ 已从三级缓存移动到二级缓存"

  这是整个循环依赖解决过程的"递归出口"。
  当 B 在属性填充时需要 A，A 已在三级缓存中暴露了 ObjectFactory。
  执行 ObjectFactory 后，A 的早期引用从三级缓存升级到二级缓存，B 成功拿到 A 的引用。

重点观察 3 — "serviceB=null" 揭示半成品本质：

  日志关键行：
    "[serviceA] 从缓存获取: ServiceA@4f47d241[serviceB=null]"

  此时 B 拿到的 A 实例，其 serviceB 字段仍然是 null。
  这就是"半成品"的含义 —— 实例化完成但属性填充尚未完成。

  但这不影响最终正确性，因为 B 拿到的是 A 的对象引用（内存地址）。
  后续步骤中 A.serviceB = B 完成填充后，B 手中持有的 A 引用自动"看到"新值。
  这是 Java 引用语义的关键特性。

重点观察 4 — 创建完成后的最终验证：

  日志关键行：
    "[5.放入一级缓存] singletonObjects.put("serviceB", bean)"
    "[创建完成] serviceB = ServiceB@4c3e4790[serviceA=ServiceA@4f47d241]"
    "[5.放入一级缓存] singletonObjects.put("serviceA", bean)"
    "[创建完成] serviceA = ServiceA@4f47d241[serviceB=ServiceB@4c3e4790]"

  最终结果验证：
    "a         = ServiceA@4f47d241[serviceB=ServiceB@4c3e4790]"
    "a.serviceB = ServiceB@4c3e4790[serviceA=ServiceA@4f47d241]"
    "a.serviceB.serviceA == a ? true"

  注意对象地址的一致性：
    - ServiceA 始终是 @4f47d241，ServiceB 始终是 @4c3e4790
    - B 持有的 A 引用（@4f47d241）与最终的 A 是同一个对象
    - 说明从三级缓存拿到的半成品引用，和最终一级缓存中的成品，是同一个实例

重点观察 5 — 构造器注入为何不行（日志原文）：

  日志明确给出了对比分析：
    字段注入：
      "1. new ServiceA()                 ← 实例化成功"
      "2. 放入三级缓存                    ← 此时可被循环依赖方获取"

    构造器注入：
      "1. new ServiceA(serviceB)         ← 需要 B 才能实例化"
      "   └── new ServiceB(serviceA)      ← 需要 A 才能实例化"
      "       └── A 还没实例化完成，无法放入三级缓存"
      "→ 抛出 BeanCurrentlyInCreationException"

  结论原文：
    "循环依赖的前提是「至少一方的实例化可以独立于另一方」"

重点观察 6 — 缓存升级路径对比：

  serviceA 的缓存变化：
    步骤3  → 三级缓存（ObjectFactory）
    步骤8  → 二级缓存（ObjectFactory 执行后，早期引用）
    步骤12 → 一级缓存（属性填充 + 初始化全部完成，成品 Bean）

  serviceB 的缓存变化：
    步骤6  → 三级缓存（ObjectFactory）
    步骤10 → 一级缓存（直接从三级跳到一级）

  关键发现：serviceB 没有经过二级缓存！
  只有被循环依赖方提前请求过早期引用的 Bean（即 serviceA）才会出现在二级缓存中。
  serviceB 在属性填充阶段没有被其他正在创建的 Bean 请求过，所以直接进入一级。

结论：

  三级缓存的核心思路：
    "先让实例化完成的半成品暴露给其他 Bean，等属性填充完再升级为成品。"

  这个策略依赖两个前提：
    1. Java 对象的引用语义 —— B 拿到 A 的引用后，A 后续的字段变化 B 都能看到
    2. 实例化和属性填充是分离的 —— new A() 不需要依赖 B（构造器注入不满足此条件）

  三级缓存 vs 二级缓存：
    如果没有 AOP，二级缓存（半成品 + 成品）就够了。
    三级缓存（ObjectFactory）的意义在于：当存在 AOP 时，
    需要在循环依赖被发现的那个时刻才决定是否创建代理对象，
    而不是在实例化时就提前创建代理（违反 Spring 设计原则：代理应在初始化后创建）。

  面试速记原文总结：
    "singletonObjects → 成品 Bean（已完成所有初始化）"
    "earlySingletonObjects → 半成品 Bean（已实例化，未填充属性）"
    "singletonFactories → ObjectFactory（产生早期引用，可能是AOP代理）"
    "构造器循环依赖无法解决：实例化未完成，无法放入三级缓存"
    "prototype 循环依赖无法解决：prototype 不走缓存"
