TransactionPropagationDemo 日志分析
========================================

整体结果: REQUIRED vs REQUIRES_NEW 行为差异验证通过 -- REQUIRED 一损俱损，REQUIRES_NEW 独善其身。

=== 逐步分析 ===

  ┌────────────────────────────────────┬──────────────────────────────┬──────┐
  | 场景                               | 结果                          | 状态 |
  ├────────────────────────────────────┼──────────────────────────────┼──────┤
  | 场景 1：REQUIRED + 外部抛异常       | Alice=1000.0, Bob=1000.0     | ✅   |
  | 场景 2：REQUIRES_NEW + 外部抛异常   | Alice=1000.0, Bob=1010.0     | ✅   |
  └────────────────────────────────────┴──────────────────────────────┴──────┘

=== 重点观察 1：场景 1 REQUIRED -- 外部异常导致全部回滚 ===

  日志原文：

    【场景 1】REQUIRED + 外部抛异常
      期望：Alice 和 Bob 的变更都回滚

      捕获: 外部方法异常
      [REQUIRED 异常后] Alice=1000.0, Bob=1000.0

  分析：
  - 初始状态 "[初始状态] Alice=1000.0, Bob=1000.0"
  - 外部方法执行 Alice -10，内部方法（REQUIRED）执行 Bob +10
  - REQUIRED 让内部方法加入了外部事务（共享同一连接和事务）
  - 外部方法抛异常 → 整个事务回滚
  - 结果 "Alice=1000.0, Bob=1000.0" -- 两者都恢复到初始值
  - 虽然内部方法执行成功（Bob +10 的 SQL 已执行），但因为共享事务，外部回滚带走了一切

  执行时间线：
    1. 外部方法开启事务 T1
    2. UPDATE Alice -10（在 T1 内）
    3. 内部方法（REQUIRED）→ 加入 T1（不新建事务）
    4. UPDATE Bob +10（在 T1 内）
    5. 内部方法正常返回（T1 未结束，不提交）
    6. 外部方法抛异常 → T1 整体 ROLLBACK
    7. Alice -10 和 Bob +10 全部撤销

=== 重点观察 2：场景 2 REQUIRES_NEW -- Bob 的变更独立保留 ===

  日志原文：

    【场景 2】REQUIRES_NEW + 外部抛异常
      期望：Alice 回滚（外部），Bob 保留（内部独立提交）

      捕获: 外部方法异常
      [REQUIRES_NEW 异常后] Alice=1000.0, Bob=1010.0

  分析：
  - Alice=1000.0 说明外部事务的 Alice -10 被回滚了
  - Bob=1010.0（而非 1000.0）说明内部事务的 Bob +10 被独立提交并保留了！
  - 这就是 REQUIRES_NEW 的核心价值：内部事务独立于外部事务

  执行时间线：
    1. 外部方法开启事务 T1（连接 C1）
    2. UPDATE Alice -10（在 T1 内）
    3. 内部方法（REQUIRES_NEW）→ 挂起 T1，新建事务 T2（连接 C2）
    4. UPDATE Bob +10（在 T2 内）
    5. 内部方法正常返回 → T2 COMMIT（Bob +10 已持久化到数据库！）
    6. 恢复 T1
    7. 外部方法抛异常 → T1 ROLLBACK（只撤销 Alice -10）
    8. T2 的提交不可逆，Bob=1010.0 保留

=== 重点观察 3：Bob=1010.0 -- 这个 +10 是 REQUIRES_NEW 的铁证 ===

  对比两个场景的 Bob 余额：
    场景 1（REQUIRED）：   Bob=1000.0  → Bob +10 被回滚
    场景 2（REQUIRES_NEW）：Bob=1010.0  → Bob +10 被保留

  同样的操作（Bob +10），同样的外部异常，结果截然不同。
  唯一区别就是内部方法的传播行为：REQUIRED vs REQUIRES_NEW。

  典型应用场景（需要 REQUIRES_NEW）：
  1. 审计日志 -- 即使业务操作失败回滚，审计记录也必须保留
  2. 操作记录 -- 记录用户操作轨迹，不能因业务回滚而丢失
  3. 消息发送标记 -- 标记消息已发送，避免因回滚导致重复发送
  4. 序列号生成 -- 获取自增序列号后不能因回滚而"浪费"号段

=== 重点观察 4：7 种传播行为对比表（日志中已给出） ===

  日志原文中的对比表：

    REQUIRED (默认)  | 有事务: 加入当前事务      | 无事务: 新建事务
    REQUIRES_NEW     | 有事务: 挂起当前，新建     | 无事务: 新建事务
    NESTED           | 有事务: 嵌套保存点         | 无事务: 新建事务
    SUPPORTS         | 有事务: 加入当前事务       | 无事务: 非事务执行
    NOT_SUPPORTED    | 有事务: 挂起当前事务       | 无事务: 非事务执行
    MANDATORY        | 有事务: 加入当前事务       | 无事务: 抛出异常
    NEVER            | 有事务: 抛出异常           | 无事务: 非事务执行

  本 Demo 通过真实数据验证了最关键的两种：REQUIRED 和 REQUIRES_NEW。

  关键差异（日志中也标注了）：
  - REQUIRED vs REQUIRES_NEW：共享事务 vs 独立事务
  - REQUIRES_NEW vs NESTED：独立事务（完全隔离） vs 嵌套保存点（外部回滚内部跟滚）

=== 重点观察 5：REQUIRES_NEW 的性能代价 ===

  日志面试速记中提到：
    "⚠ REQUIRES_NEW 需要独立连接，性能开销较大"

  源码机制：
  - REQUIRES_NEW 执行时，外部事务的连接 C1 被挂起（suspend）但未释放
  - 内部事务新建连接 C2
  - 此时同时占用 2 个数据库连接
  - 高并发场景下，如果大量方法使用 REQUIRES_NEW，可能耗尽连接池
  - 内部方法执行完毕后，C2 释放，恢复（resume）C1

=== 结论 ===

  REQUIRED 和 REQUIRES_NEW 是面试和实战中最重要的两种传播行为。
  本 Demo 用真实余额数据证明了：
  - REQUIRED（场景1）：Alice=1000.0, Bob=1000.0 -- 外部异常导致内外都回滚，"一损俱损"
  - REQUIRES_NEW（场景2）：Alice=1000.0, Bob=1010.0 -- 内部独立提交不受外部影响，"独善其身"
  记住口诀：REQUIRED 一损俱损，REQUIRES_NEW 独善其身。
  生产中大多数场景用 REQUIRED（默认）即可，只有审计日志等必须独立提交的场景才用 REQUIRES_NEW。
