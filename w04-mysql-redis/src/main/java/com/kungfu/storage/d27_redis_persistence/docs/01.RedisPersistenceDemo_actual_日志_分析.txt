========================================
  RedisPersistenceDemo 日志分析
========================================

整体结果：
  通过 INFO persistence 查看当前 Redis 持久化配置
  成功触发 BGSAVE，响应"Background saving started"
  对比了 RDB/AOF/混合三种持久化方案的优劣

逐步分析表：
┌──────────────────────────┬──────────────────────────────────────┐
│ INFO 字段                │ 实际值与含义                         │
├──────────────────────────┼──────────────────────────────────────┤
│ rdb_last_save_time       │ 1776647459（上次 RDB 保存的时间戳）  │
│ rdb_last_bgsave_status   │ ok（上次 BGSAVE 成功）               │
│ rdb_last_bgsave_time_sec │ -1（未执行过 BGSAVE 或耗时极短）    │
│ aof_enabled              │ 0（AOF 未开启）                      │
│ aof_last_rewrite_status  │ ok                                   │
│ aof_last_rewrite_time    │ -1（未执行过 AOF 重写）              │
└──────────────────────────┴──────────────────────────────────────┘

重点观察 1 — 当前配置分析
  aof_enabled=0 → 当前 Redis 只开启了 RDB，未开启 AOF
  ★ 这意味着如果 Redis 崩溃，会丢失上次 RDB 快照之后的所有写入
  ★ 生产环境应开启 AOF（appendonly yes）或混合持久化

重点观察 2 — BGSAVE 执行机制
  BGSAVE 响应: "Background saving started"
  执行流程：
    1. 主进程 fork() 子进程
    2. 子进程将内存数据写入临时 RDB 文件
    3. 写完后替换旧 RDB 文件
  ★ fork 使用 COW（Copy-On-Write）技术，不阻塞主线程
  ★ 但 fork 本身有短暂阻塞（内存越大阻塞越久，通常毫秒级）
  ★ rdb_last_bgsave_time_sec=-1 说明之前没有执行过或耗时极短

重点观察 3 — 三种持久化方案对比
  ┌──────────────┬──────────────┬──────────────┬──────────────┐
  │ 维度         │ RDB          │ AOF          │ 混合(4.0+)   │
  ├──────────────┼──────────────┼──────────────┼──────────────┤
  │ 数据安全     │ 可能丢分钟级 │ 最多丢1秒    │ 兼顾两者     │
  │ 恢复速度     │ ★ 快        │ 慢           │ 快           │
  │ 文件大小     │ ★ 小        │ 大           │ 中等         │
  │ 写入性能影响 │ fork 时短暂  │ everysec 低  │ 低           │
  └──────────────┴──────────────┴──────────────┴──────────────┘
  ★ 生产推荐混合持久化：aof-use-rdb-preamble yes

重点观察 4 — AOF 三种 fsync 策略
  always:   每条命令 fsync → 最安全但性能最差
  everysec: 每秒 fsync → ★ 推荐，最多丢 1 秒
  no:       OS 决定 → 性能最好但可能丢几十秒
  ★ 面试标准答案：生产用 everysec，平衡安全与性能

结论：
  当前 Redis 仅开启 RDB（aof_enabled=0），生产环境应开启混合持久化
  BGSAVE 通过 fork+COW 实现非阻塞快照
  面试四句话：RDB 快但可能丢数据，AOF 安全但恢复慢，
  混合持久化兼顾两者是生产首选，AOF 用 everysec 策略
