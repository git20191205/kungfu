========================================
  IndexOptimizationDemo 日志分析
========================================

整体结果：
  50000 行数据插入耗时 572ms，3 个案例分别验证了
  单列索引效果（1000x 提升）、联合索引最左前缀规则、覆盖索引消除回表

逐步分析表：
┌──────────┬──────────────┬──────────────┬──────────────┬──────────────┐
│ 案例     │ type 变化    │ rows 变化    │ key 变化     │ 核心收益     │
├──────────┼──────────────┼──────────────┼──────────────┼──────────────┤
│ 案例1    │ ALL → ref    │ 49993 → 50   │ NULL→idx     │ 扫描量降1000x│
│ 案例2 ✓ │ ref          │ 16 / 50      │ idx_composite│ 最左前缀命中 │
│ 案例2 ✗ │ ALL          │ 49993        │ NULL         │ 跳过首列失效 │
│ 案例3    │ ref → ref    │ 16 → 16      │ idx_composite│ Extra差异    │
└──────────┴──────────────┴──────────────┴──────────────┴──────────────┘

重点观察 1 — 案例1: 索引前后对比
  无索引: type=ALL, rows=49993, key=NULL → 全表扫描
  加索引: type=ref, rows=50,    key=idx_user_id → 索引查找
  ★ rows 从 49993 降到 50，约 1000 倍改善
  ★ key_len=8（BIGINT 占 8 字节），ref=const（与常量 42 比较）

重点观察 2 — 案例2: 联合索引最左前缀
  联合索引 (user_id, status, product_id)：
  ┌─────────────────────────────────┬──────┬───────┬──────────┐
  │ WHERE 条件                      │ 命中 │ type  │ key_len  │
  ├─────────────────────────────────┼──────┼───────┼──────────┤
  │ user_id=42 AND status=1         │ ✓    │ ref   │ 9        │
  │ user_id=42                      │ ✓    │ ref   │ 8        │
  │ status=1                        │ ✗    │ ALL   │ NULL     │
  │ product_id=100                  │ ✗    │ ALL   │ NULL     │
  └─────────────────────────────────┴──────┴───────┴──────────┘
  ★ key_len 从 8→9 说明多用了 status 列（1 字节 TINYINT）
  ★ 跳过 user_id 直接查 status/product_id → 索引完全失效

重点观察 3 — 案例3: 覆盖索引 vs 回表
  SELECT * (回表):     Extra = NULL（需要回聚簇索引取完整行）
  SELECT 索引列(覆盖): Extra = Using index（只读索引即可）
  ★ 两者 rows 相同（都是 16），但 IO 开销差异大
  ★ 回表 = 二级索引查到主键 → 再用主键查聚簇索引 → 两次 B+Tree 查找
  ★ 覆盖索引 = 一次索引扫描搞定，省掉回表的随机 IO

结论：
  索引优化三板斧：1) 给高频查询列加索引  2) 联合索引遵守最左前缀
  3) 高频查询用覆盖索引避免回表
  面试追问"索引不是越多越好"时回答：每个索引都有写入开销（INSERT/UPDATE
  需要维护索引 B+Tree）和存储开销（额外磁盘空间），需要权衡读写比
