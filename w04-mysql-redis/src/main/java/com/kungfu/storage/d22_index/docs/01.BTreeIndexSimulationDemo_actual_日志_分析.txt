========================================
  BTreeIndexSimulationDemo 日志分析
========================================

整体结果：
  B+Tree（ORDER=4）插入 16 个 key 后，高度=3，节点数=11
  等值查找 key=22 经过 3 层定位命中，key=99 经过 3 层确认不存在
  范围查找 [10,25] 返回 7 条结果，通过叶子链表顺序遍历完成

逐步分析表：
┌──────────────────┬──────────────────────────────────────────────────┐
│ 观测项           │ 实际数据                                          │
├──────────────────┼──────────────────────────────────────────────────┤
│ 树阶数           │ ORDER=4（每个节点最多 3 个 key，4 个子指针）      │
│ 插入 key 数      │ 16 个                                            │
│ 树高度           │ 3（根→内部→叶子）                                │
│ 节点总数         │ 11（1 根 + 3 内部 + 7 叶子）                     │
│ 叶子链表长度     │ 7 个叶子节点，双向链表串联                        │
│ 等值查找 IO 次数 │ 3 次（每层 1 次）                                │
│ 范围查找结果     │ 7 条（10,12,15,18,20,22,25）                     │
└──────────────────┴──────────────────────────────────────────────────┘

重点观察 1 — 等值查找路径
  查找 key=22 的路径：
    第1层 keys=[15,25] → 15<22<25 → 走子节点[1]
    第2层 keys=[20]    → 22>20    → 走子节点[1]
    第3层 keys=[20,22] → 命中！
  ★ 无论查哪个 key，最多 3 次比较即可定位
  ★ 查找 key=99（不存在）同样只需 3 次 IO 即可确认

重点观察 2 — 范围查找的高效性
  查找 [10,25]：先定位 key=10 所在叶子 → 沿链表向右遍历到 25
  不需要回到上层节点 → 范围查询的 IO = 定位IO + 顺序读叶子页
  这就是 B+Tree 比 B-Tree 范围查询快的根本原因

重点观察 3 — 高度与数据量的关系（生产推算）
  ┌──────────┬──────────────────────────────────────────┐
  │ 树高度   │ 可容纳行数（16KB页，key=8B，指针=6B）    │
  ├──────────┼──────────────────────────────────────────┤
  │ 2        │ 1170 × 16 ≈ 1.8 万行                    │
  │ 3        │ 1170 × 1170 × 16 ≈ 2190 万行            │
  │ 4        │ 约 256 亿行（几乎不会到这个高度）        │
  └──────────┴──────────────────────────────────────────┘
  ★ 2000 万行只需 3 次磁盘 IO — 这是面试最高频的数字

结论：
  B+Tree 的三大设计决策（非叶子不存数据、叶子链表、所有数据在叶子）
  共同保证了：树极矮（IO少）、范围查询快（链表遍历）、性能稳定（路径等长）
  面试回答"为什么用 B+Tree"时，对比 Hash（不支持范围）、红黑树（太高）、
  B-Tree（数据分散范围慢）三个反面即可
