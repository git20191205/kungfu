========================================
  MiniMVCCDemo 日志分析
========================================

整体结果：
  模拟 3 个事务（trx100/200/300）修改同一行，构建 2 节点版本链
  RC 隔离级别下两次读到不同值（1000→2000），RR 下两次读到相同值（1000→1000）
  核心差异仅在于 ReadView 的创建时机

逐步分析表：
┌──────────┬──────────────────────────────────────────────────────────┐
│ 组件     │ 实际数据                                                │
├──────────┼──────────────────────────────────────────────────────────┤
│ 版本链   │ trx200(value=2000) → trx100(value=1000) → NULL         │
│ 版本数   │ 2 个版本（T1 已提交，T2 未提交→后提交）                 │
│ 读取事务 │ T3(trx=300)，在 T2 提交前后各做一次 SELECT              │
│ RC 结果  │ 第一次=1000，第二次=2000（不可重复读）                  │
│ RR 结果  │ 第一次=1000，第二次=1000（可重复读）                    │
└──────────┴──────────────────────────────────────────────────────────┘

重点观察 1 — ReadView 的 4 个字段
  T3 第一次 SELECT 时（T2 未提交）：
    ReadView{creator=300, m_ids=[200,300], min=200, max=301}
  T3 第二次 SELECT 时（T2 已提交）：
    RC 新建: ReadView{creator=300, m_ids=[300], min=300, max=301}
    RR 复用: ReadView{creator=300, m_ids=[200,300], min=200, max=301}
  ★ m_ids 是关键：RC 第二次 m_ids 不含 200（已提交）→ trx200 可见
  ★ RR 复用旧 ReadView，m_ids 仍含 200 → trx200 不可见

重点观察 2 — 可见性判断的遍历过程
  RC 第二次读：
    版本 trx200(2000) → trx_id=200 < min=300 → 已提交 → 可见 ✓ → 返回 2000
  RR 第二次读：
    版本 trx200(2000) → trx_id=200 在 m_ids[200,300] 中 → 未提交 → 不可见 ✗
    版本 trx100(1000) → trx_id=100 < min=200 → 已提交 → 可见 ✓ → 返回 1000
  ★ 同一个 trx_id=200，在不同 ReadView 下判断结果不同

重点观察 3 — RC vs RR 的本质区别
  ┌─────────────┬──────────────────┬──────────────────┐
  │             │ RC               │ RR               │
  ├─────────────┼──────────────────┼──────────────────┤
  │ ReadView    │ 每次 SELECT 新建 │ 第一次创建后复用 │
  │ 第一次读    │ 1000             │ 1000             │
  │ 第二次读    │ 2000             │ 1000             │
  │ 问题        │ 不可重复读       │ 无               │
  └─────────────┴──────────────────┴──────────────────┘
  ★ 唯一区别就是 ReadView 创建时机，没有其他差异

重点观察 4 — MVCC 的边界
  MVCC 只对快照读（普通 SELECT）生效
  当前读（SELECT FOR UPDATE / UPDATE / DELETE）不走 MVCC，直接加锁
  ★ 面试追问"RR 能完全防幻读吗"时回答：
    快照读通过 MVCC 防幻读，当前读通过 Gap Lock 防幻读
    但混合使用快照读和当前读时仍可能出现"幻读假象"

结论：
  MVCC 三件套 = 隐藏列(DB_TRX_ID, DB_ROLL_PTR) + Undo Log 版本链 + ReadView
  RC 和 RR 的唯一区别是 ReadView 创建时机
  这是面试中"InnoDB 如何实现事务隔离"的标准答案
