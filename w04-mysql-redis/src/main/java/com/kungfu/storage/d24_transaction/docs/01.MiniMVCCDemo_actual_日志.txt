========================================
  手写 MVCC 模拟（版本链 + ReadView）
========================================

=== 一、MVCC 核心原理 ===

  MVCC = Multi-Version Concurrency Control（多版本并发控制）
  核心思想：读不加锁，读写不冲突，通过版本链实现一致性读

  三大组件：
  ┌──────────────────────────────────────────────────────────────┐
  │  1. 隐藏列                                                   │
  │     DB_TRX_ID  — 最后修改该行的事务 ID                       │
  │     DB_ROLL_PTR — 回滚指针，指向 Undo Log 中的旧版本          │
  │                                                              │
  │  2. Undo Log 版本链                                          │
  │     每次修改，旧数据写入 Undo Log，通过指针串成链表            │
  │     最新版本 → 次新版本 → ... → 最老版本                     │
  │                                                              │
  │  3. ReadView（一致性读视图）                                  │
  │     事务做快照读时创建，记录当时的活跃事务列表                  │
  │     用于判断版本链中哪个版本对自己可见                         │
  └──────────────────────────────────────────────────────────────┘

=== 二、模拟 3 个事务修改同一行 ===

  场景：t_account 表中 Alice 的 balance 字段

  时间线：
  ──────────────────────────────────────────────────────────────
  T1(trx=100)    T2(trx=200)    T3(trx=300)    时间 ↓
  ──────────────────────────────────────────────────────────────
  BEGIN
  UPDATE → 1000                                  t1
  COMMIT                                         t2
                  BEGIN                           t3
                  UPDATE → 2000                   t4
                                  BEGIN           t5
                                  SELECT(?)       t6 ← 这里读到什么？
                  COMMIT                          t7
                                  SELECT(?)       t8 ← 再读一次呢？
  ──────────────────────────────────────────────────────────────

  版本链（头部是最新版本）：
  ┌──────────────────┐     ┌──────────────────┐
  │ trx_id = 200     │     │ trx_id = 100     │
  │ value  = '2000'  │────>│ value  = '1000'  │───> NULL
  │ （T2 未提交）    │     │ （T1 已提交）    │
  └──────────────────┘     └──────────────────┘
        ↑ 最新版本                ↑ 旧版本

=== 三、ReadView 可见性判断算法 ===

  ┌──────────────────────────────────────────────────────────────┐
  │              ReadView 可见性判断流程                          │
  │                                                              │
  │  对版本链中每个版本的 trx_id 做判断：                         │
  │                                                              │
  │  trx_id == creator_trx_id ?  ──── YES ──→ 可见 ✓（自己改的） │
  │         │ NO                                                 │
  │         ↓                                                    │
  │  trx_id < min_trx_id ?       ──── YES ──→ 可见 ✓（已提交）  │
  │         │ NO                                                 │
  │         ↓                                                    │
  │  trx_id >= max_trx_id ?      ──── YES ──→ 不可见 ✗（未来的） │
  │         │ NO                                                 │
  │         ↓                                                    │
  │  trx_id 在 m_ids 中 ?        ──── YES ──→ 不可见 ✗（未提交） │
  │         │ NO                                                 │
  │         ↓                                                    │
  │  可见 ✓（已提交）                                            │
  └──────────────────────────────────────────────────────────────┘

=== 四、RC vs RR 隔离级别对比 ===

  核心区别：
  ┌────────────────────┬──────────────────────────────────────────┐
  │ 隔离级别           │ ReadView 创建时机                        │
  ├────────────────────┼──────────────────────────────────────────┤
  │ READ COMMITTED     │ 每次 SELECT 都创建新的 ReadView           │
  │ REPEATABLE READ    │ 事务第一次 SELECT 时创建，之后一直复用    │
  └────────────────────┴──────────────────────────────────────────┘

  ───────────────────────────────────────────────────────
  ★ READ COMMITTED（RC）：每次读都创建新 ReadView
  ───────────────────────────────────────────────────────

  【第一次 SELECT】T2 未提交时
    创建 ReadView{creator=300, m_ids=[200, 300], min=200, max=301}
    遍历版本链：
      版本 [trx_id=200, value='2000'] → 不可见 ✗
      版本 [trx_id=100, value='1000'] → 可见 ✓
    → T3 读到: balance = 1000

  【第二次 SELECT】T2 已提交后
    创建 ReadView{creator=300, m_ids=[300], min=300, max=301}
    遍历版本链：
      版本 [trx_id=200, value='2000'] → 可见 ✓
    → T3 读到: balance = 2000

    ★ RC 结果: 第一次读 1000，第二次读 2000
      两次读到不同值 → 不可重复读！

  ───────────────────────────────────────────────────────
  ★ REPEATABLE READ（RR）：始终用第一次的 ReadView
  ───────────────────────────────────────────────────────

  【第一次 SELECT】T2 未提交时
    创建 ReadView{creator=300, m_ids=[200, 300], min=200, max=301}
    遍历版本链：
      版本 [trx_id=200, value='2000'] → 不可见 ✗
      版本 [trx_id=100, value='1000'] → 可见 ✓
    → T3 读到: balance = 1000

  【第二次 SELECT】T2 已提交，但 RR 复用第一次 ReadView！
    复用 ReadView{creator=300, m_ids=[200, 300], min=200, max=301}
    遍历版本链：
      版本 [trx_id=200, value='2000'] → 不可见 ✗
      版本 [trx_id=100, value='1000'] → 可见 ✓
    → T3 读到: balance = 1000

    ★ RR 结果: 第一次读 1000，第二次读 1000
      两次读到相同值 → 可重复读！

  ┌─────────────┬──────────────┬──────────────┬────────────────────────┐
  │ 隔离级别    │ 第一次读     │ 第二次读     │ 原因                   │
  ├─────────────┼──────────────┼──────────────┼────────────────────────┤
  │ RC          │ 1000         │ 2000         │ 每次读新建 ReadView    │
  │ RR ★       │ 1000         │ 1000         │ 复用第一次 ReadView    │
  └─────────────┴──────────────┴──────────────┴────────────────────────┘

========================================
  面试速记
========================================

  1. MVCC = 隐藏列(DB_TRX_ID, DB_ROLL_PTR) + Undo Log 版本链 + ReadView
  2. ReadView 4 个字段: m_ids, min_trx_id, max_trx_id, creator_trx_id
  3. 可见性判断: 自己的→可见, 已提交→可见, 未来的→不可见, 活跃的→不可见
  4. RC vs RR 的唯一区别: ReadView 创建时机
     RC: 每次 SELECT 新建 ReadView → 能看到其他事务的最新提交 → 不可重复读
     RR: 第一次 SELECT 创建后复用 → 看不到后续提交 → 可重复读
  5. MVCC 只对 SELECT（快照读）生效，UPDATE/DELETE 是当前读（加锁）
