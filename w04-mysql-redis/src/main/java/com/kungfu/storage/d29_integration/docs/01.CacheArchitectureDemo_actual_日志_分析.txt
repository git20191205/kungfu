  完整缓存架构四层流水线验证：布隆过滤器→Redis→互斥锁→MySQL
  穿透拦截 4/4 个假 ID，20 并发请求收敛为 1 次 DB 查询
  Redis 缓存 vs MySQL 直查性能对比：3.5x 提升（28ms vs 98ms）

重点观察 1 — 完整查询流水线（id=42）
  ① 布隆过滤器: 可能存在 → 继续（不拦截有效 ID）
  ② Redis 缓存: MISS → 首次查询无缓存
  ③ 互斥锁: 获得 → 只有当前线程查 DB
  ④ MySQL 查询: Product_42:574.52 → 写入 Redis(TTL=3675s)
  ★ TTL=3675s = 3600 + random(0,300)，随机防雪崩
  ★ 第二次查询同一 ID 会直接命中 Redis，不走 ③④

重点观察 2 — 穿透拦截效果
  id=99999 → 拦截 ✓
  id=-1    → 拦截 ✓
  id=88888 → 拦截 ✓
  id=77777 → 拦截 ✓
  ★ 4 个不存在的 ID 全部在第一层被拦截，完全不触及 Redis 和 MySQL
  ★ 布隆过滤器的误判率取决于位数组大小和哈希函数个数
  ★ 生产中用 Redisson 的 RBloomFilter 或 Guava BloomFilter

重点观察 3 — 并发击穿防护
  20 个并发请求查询同一个未缓存的商品：
  DB 查询 = 1 次（只有获得锁的线程查 DB）
  缓存命中 = 19 次（其他线程等锁释放后读缓存）
  ★ 互斥锁将 20 次 DB 查询收敛为 1 次，保护率 95%
  ★ 锁实现：SETNX key value EX timeout（带超时防死锁）

重点观察 4 — 性能对比数据
  1000 次随机查询：
  ┌──────────────┬──────────┬──────────────────────────┐
  │ 方式         │ 耗时     │ 分析                     │
  ├──────────────┼──────────┼──────────────────────────┤
  │ Redis 缓存   │ 28ms     │ 内存读取，网络往返为主   │
  │ MySQL 直查   │ 98ms     │ 磁盘 IO + SQL 解析开销   │
  │ 提升倍数     │ 3.5x     │ 数据量小时差距不大       │
  └──────────────┴──────────┴──────────────────────────┘
  ★ 3.5x 是本地小数据量的结果，生产环境（大数据量+网络延迟）差距更大
  ★ 热点数据缓存命中率通常 > 95%，实际 QPS 提升远超 3.5x

重点观察 5 — 架构四层的职责分工
  布隆过滤器: 拦截不存在的请求（防穿透）— O(1) 判断
  Redis 缓存: 承接正常读请求（减少 DB 压力）— 毫秒级
  互斥锁:     收敛缓存重建的并发（防击穿）— SETNX
  MySQL:      数据源兜底（最终一致性保证）— 持久化

结论：
  四层架构覆盖了缓存三大问题（穿透/击穿/雪崩）的完整解决方案
  面试画架构图时按 布隆→Redis→Mutex→DB 四层画即可
  性能数据：20 并发→1 次 DB，Redis 比 MySQL 快 3.5x
