========================================
  缓存一致性方案对比
========================================

=== 三种方案对比 ===

  ┌──────────────────┬──────────────────────────────┬──────────────┐
  │ 方案             │ 流程                          │ 一致性       │
  ├──────────────────┼──────────────────────────────┼──────────────┤
  │ Cache-Aside ★   │ 更新 DB → 删缓存             │ 最终一致     │
  │ 延迟双删         │ 删缓存 → 更新 DB → sleep → 删│ 更强一致     │
  │ 先删缓存再更新DB │ 删缓存 → 更新 DB             │ ✗ 有并发问题│
  └──────────────────┴──────────────────────────────┴──────────────┘

=== 方案一：Cache-Aside（旁路缓存）===

  ★ 读流程：
    1. 查缓存: MISS
    2. 查 DB: iPhone:9999.00
    3. 写缓存: demo:d28:product:1001 → iPhone:9999.00 (TTL=3600s)

  ★ 写流程（价格改为 8888）：
    1. 更新 DB: price → 8888.00
    2. 删缓存: DEL demo:d28:product:1001
    → 下次读取时会重新从 DB 加载最新数据

=== 方案二：延迟双删 ===

  流程：
    1. 删缓存
    2. 更新 DB: price → 7777.00
    3. sleep 500ms（等待可能的并发读写入旧缓存）
    4. 再次删缓存
    → 即使 step 2-3 之间有并发读写入了旧缓存，step 4 也会清除它

  延迟时间怎么定？
    = 业务读取耗时 + 几百毫秒余量
    通常 500ms ~ 1s 即可

=== 为什么「先删缓存再更新 DB」有问题 ===

  并发场景时序：
  ┌──────────────────────────────────────────────────────────────┐
  │ 时间  │ 线程 A（写）              │ 线程 B（读）             │
  ├───────┼──────────────────────────┼─────────────────────────┤
  │ T1    │ 删缓存                   │                         │
  │ T2    │                          │ 查缓存 → MISS           │
  │ T3    │                          │ 查 DB → 旧值 9999       │
  │ T4    │                          │ 写缓存 → 9999（旧值！） │
  │ T5    │ 更新 DB → 8888           │                         │
  │       │                          │                         │
  │ 结果  │ DB=8888, 缓存=9999 → 不一致！                     │
  └──────────────────────────────────────────────────────────────┘

  ★ 面试速记：
    Q: 缓存和 DB 怎么保证一致性？
    A: 用 Cache-Aside（先更新 DB，再删缓存）
       更强一致性用延迟双删
       不要先删缓存再更新 DB（有并发问题）
