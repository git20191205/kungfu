========================================
  CacheConsistencyDemo 日志分析
========================================

整体结果：
  对比了 3 种缓存一致性方案：Cache-Aside（推荐）、延迟双删（更强一致）、
  先删缓存再更新 DB（有并发问题）
  通过实际操作验证了读写流程和并发竞态条件

逐步分析表：
┌──────────────────┬──────────────────────────────┬──────────────┐
│ 方案             │ 流程                          │ 一致性       │
├──────────────────┼──────────────────────────────┼──────────────┤
│ Cache-Aside ★   │ 更新 DB → 删缓存             │ 最终一致     │
│ 延迟双删         │ 删缓存→更新 DB→sleep→再删    │ 更强一致     │
│ 先删缓存再更新DB │ 删缓存→更新 DB               │ ✗ 有并发问题│
└──────────────────┴──────────────────────────────┴──────────────┘

重点观察 1 — Cache-Aside 读写流程
  读流程：
    查缓存 → MISS → 查 DB(iPhone:9999.00) → 写缓存(TTL=3600s)
  写流程（价格改为 8888）：
    更新 DB(price→8888) → 删缓存(DEL key)
  ★ 为什么是"删缓存"而不是"更新缓存"？
    1. 避免并发写导致缓存与 DB 不一致
    2. 懒加载思想：下次读时再从 DB 加载最新值
    3. 如果缓存更新逻辑复杂（涉及计算），删除更简单可靠

重点观察 2 — 延迟双删的时序
  Step 1: 删缓存
  Step 2: 更新 DB(price→7777)
  Step 3: sleep 500ms（等待可能的并发读写入旧缓存）
  Step 4: 再次删缓存
  ★ 延迟时间 = 业务读取耗时 + 几百毫秒余量（通常 500ms~1s）
  ★ 即使 Step 2~3 之间有并发读写入了旧缓存，Step 4 也会清除

重点观察 3 — "先删缓存再更新 DB"的竞态条件
  ┌───────┬──────────────────────────┬─────────────────────────┐
  │ 时间  │ 线程 A（写）             │ 线程 B（读）            │
  ├───────┼──────────────────────────┼─────────────────────────┤
  │ T1    │ 删缓存                   │                         │
  │ T2    │                          │ 查缓存 → MISS           │
  │ T3    │                          │ 查 DB → 旧值 9999       │
  │ T4    │                          │ 写缓存 → 9999（旧值！） │
  │ T5    │ 更新 DB → 8888           │                         │
  │ 结果  │ DB=8888, 缓存=9999 → 不一致！                     │
  └───────┴──────────────────────────┴─────────────────────────┘
  ★ 问题根因：删缓存和更新 DB 之间有时间窗口，并发读会写入旧值
  ★ Cache-Aside 为什么没这个问题？因为先更新 DB 再删缓存，
    即使删缓存失败，下次读到的也是旧缓存（最终会过期）

重点观察 4 — 面试进阶：Cache-Aside 也有极端竞态
  场景：缓存刚好过期 + 读写并发
  T1: 读线程查缓存 MISS → 查 DB 得旧值
  T2: 写线程更新 DB → 删缓存
  T3: 读线程将旧值写入缓存
  ★ 但这个窗口极小（需要 DB 读比 DB 写慢），实际概率很低
  ★ 兜底方案：给缓存设置合理 TTL，保证最终一致性

结论：
  生产首选 Cache-Aside（先更新 DB 再删缓存），简单可靠
  需要更强一致性时用延迟双删，绝不要用"先删缓存再更新 DB"
  面试回答：先更新 DB 再删缓存 + TTL 兜底 = 最终一致性
