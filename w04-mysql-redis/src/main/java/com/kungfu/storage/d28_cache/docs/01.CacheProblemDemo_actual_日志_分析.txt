========================================
  CacheProblemDemo 日志分析
========================================

整体结果：
  三大缓存问题逐一演示：穿透（布隆过滤器拦截 2/4 个假 ID）、
  击穿（10 并发线程只有 1 次 DB 查询）、雪崩（随机 TTL 分散过期）
  每种问题都给出了对应的解决方案和实际验证

逐步分析表：
┌──────────┬──────────────────────┬──────────────────────────────────┐
│ 问题     │ 解决方案             │ 实际验证结果                     │
├──────────┼──────────────────────┼──────────────────────────────────┤
│ 穿透     │ 空值缓存 + 布隆过滤器│ id=1500/99999 被拦截，42/500 放行│
│ 击穿     │ 互斥锁 SETNX         │ 10 线程→1 次 DB 查询            │
│ 雪崩     │ 随机 TTL             │ TTL 范围 3615s~4139s（差 524s） │
└──────────┴──────────────────────┴──────────────────────────────────┘

重点观察 1 — 穿透: 布隆过滤器的精确拦截
  加载 1000 个有效 product_id 到布隆过滤器
  id=42   → 放行（实际存在）✓
  id=500  → 放行（实际存在）✓
  id=1500 → 拦截（不存在，不查缓存不查 DB）✓
  id=99999→ 拦截（不存在）✓
  ★ 布隆过滤器特性：可能误判"存在"（假阳性），但绝不误判"不存在"
  ★ 空值缓存方案：缓存 key=-999 → "NULL"(TTL=60s)，防止同一 key 反复穿透

重点观察 2 — 击穿: 互斥锁收敛并发
  10 个并发线程同时查询过期的热点商品：
  [T0] 获得锁 → 查 DB → 重建缓存
  [T1~T9] 未获得锁 → 重试（等缓存重建完成）
  结果: DB 查询次数=1（预期 1）
  ★ 互斥锁用 SETNX 实现，只有一个线程能获得锁
  ★ 日志显示 T1~T9 "重试仍无缓存"，说明重建耗时 > 重试间隔
  ★ 生产中可加 sleep+重试 或用 Redisson 的 tryLock

重点观察 3 — 雪崩: 随机 TTL 分散过期时间
  10 个商品的 TTL 分布：
  最小: 3615s（product:7）
  最大: 4139s（product:9）
  差值: 524s（约 8.7 分钟）
  ★ 基础 TTL=3600s + random(0,600)s → 过期时间分散在 8.7 分钟内
  ★ 避免了所有 key 在同一秒过期导致的 DB 压力峰值

重点观察 4 — 三大问题的本质区别
  ┌──────────┬──────────────────────┬──────────────────────────┐
  │ 问题     │ 触发条件             │ 影响范围                 │
  ├──────────┼──────────────────────┼──────────────────────────┤
  │ 穿透     │ 查不存在的数据       │ 持续性，每次请求都打 DB  │
  │ 击穿     │ 热点 key 过期瞬间    │ 瞬时性，大量并发同时打DB │
  │ 雪崩     │ 大批 key 同时过期    │ 灾难性，DB 可能被打挂    │
  └──────────┴──────────────────────┴──────────────────────────┘
  ★ 严重程度：雪崩 > 击穿 > 穿透

结论：
  穿透用布隆过滤器+空值缓存，击穿用互斥锁，雪崩用随机 TTL+多级缓存
  面试速记口诀：穿透查不存在，击穿热点过期，雪崩批量过期
  三者的共同本质：缓存失效导致请求直接打到 DB
