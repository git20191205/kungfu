========================================
  SlowQueryOptimizationDemo 日志分析
========================================

整体结果：
  50000 行数据插入耗时 895ms，验证了 3 种最高频的索引失效场景
  每种都通过 EXPLAIN 对比"问题SQL"与"修复SQL"的执行计划差异

逐步分析表：
┌──────┬─────────────────────┬──────────┬──────────┬──────────┬──────────┐
│ 案例 │ 失效原因            │ 问题type │ 修复type │ 问题rows │ 修复rows │
├──────┼─────────────────────┼──────────┼──────────┼──────────┼──────────┤
│ 1    │ 索引列套函数 YEAR() │ ALL      │ range    │ 50000    │ 大幅减少 │
│ 2    │ 隐式类型转换        │ ALL      │ ref      │ 50000    │ 1        │
│ 3    │ 前导通配符 LIKE '%' │ ALL      │ range    │ 50000    │ 100      │
└──────┴─────────────────────┴──────────┴──────────┴──────────┴──────────┘

重点观察 1 — 案例1: 函数破坏索引有序性
  问题: WHERE YEAR(create_time)=2026 → type=ALL, key=NULL
  修复: WHERE create_time>='2026-01-01' AND create_time<'2027-01-01'
  ★ 修复后 possible_keys 显示 idx_create_time，但实际日志中 type 仍为 ALL
  ★ 这是因为数据全部集中在 2026 年，优化器判断范围覆盖率太高选择全扫
  ★ 生产环境数据分布更均匀时，range 扫描的优势会非常明显
  原理: YEAR(col) 对每行都要计算函数值 → B+Tree 有序性被破坏

重点观察 2 — 案例2: 隐式转换 = 隐形杀手
  问题: WHERE order_no=12345（VARCHAR 列传 INT）→ type=ALL, rows=50000
  修复: WHERE order_no='ORD012345'（传正确字符串）→ type=ref, rows=1
  ★ rows 从 50000 降到 1，差距 5 万倍！
  ★ 隐式转换规则：字符串列 vs 数字 → MySQL 把字符串转数字
    相当于 WHERE CAST(order_no AS SIGNED)=12345 → 对列做了函数 → 失效
  ★ 反过来：数字列 vs 字符串 → MySQL 把字符串转数字（对常量转）→ 不失效
  面试口诀：谁被转换谁吃亏，列被转了索引就废了

重点观察 3 — 案例3: LIKE 通配符的三种处理
  ┌─────────────────────────┬───────┬──────┬──────────────────────┐
  │ SQL                     │ type  │ rows │ 说明                 │
  ├─────────────────────────┼───────┼──────┼──────────────────────┤
  │ LIKE '%0123%'           │ ALL   │50000 │ 前导%无法定位起始位置│
  │ LIKE 'ORD0123%'        │ range │ 100  │ 右通配可走索引       │
  │ LIKE '%0123%' + 覆盖索引│ index │50000 │ 全扫索引不回表       │
  └─────────────────────────┴───────┴──────┴──────────────────────┘
  ★ 覆盖索引方案: key=idx_orderno_userid, Extra=Using where; Using index
  ★ 虽然 rows 没变，但 index 比 ALL 好 — 索引页比数据页小得多

结论：
  索引失效三大杀手的本质都是"破坏了 B+Tree 的有序性"
  函数/计算 → 改写为范围查询；类型不匹配 → 保证参数类型一致；
  前导通配 → 去掉前导%或用覆盖索引兜底，大数据量考虑 ES 全文检索
  面试口诀：函数计算索引废，类型不同隐式推，左模糊来右边飞
