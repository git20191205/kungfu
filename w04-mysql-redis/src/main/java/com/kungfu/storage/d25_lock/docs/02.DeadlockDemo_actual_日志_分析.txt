========================================
  DeadlockDemo 日志分析
========================================

整体结果：
  成功制造死锁：T1 锁 SKU001→等 SKU002，T2 锁 SKU002→等 SKU001
  InnoDB 检测到循环等待，自动回滚了 T2（代价较小的事务）
  T1 成功获取 SKU002 并完成事务

逐步分析表：
┌──────────┬──────────────────────────────────────────────────────┐
│ 时间线   │ 事件                                                │
├──────────┼──────────────────────────────────────────────────────┤
│ Step 1   │ [T1] 已锁 SKU001                                   │
│ Step 2   │ [T2] 已锁 SKU002                                   │
│ Step 3   │ [T1] 尝试锁 SKU002... → 被 T2 阻塞                │
│ Step 4   │ [T2] 尝试锁 SKU001... → 被 T1 阻塞 → 循环等待！   │
│ Step 5   │ InnoDB 检测到死锁 → 回滚 T2                        │
│ Step 6   │ [T1] 成功获取 SKU002 → 事务完成                    │
└──────────┴──────────────────────────────────────────────────────┘

重点观察 1 — 死锁四个必要条件全部满足
  互斥:       SKU001/SKU002 的行锁同一时刻只能被一个事务持有
  持有并等待: T1 持有 SKU001 的同时等待 SKU002
  不可剥夺:   锁不能被强行释放，只能等事务结束
  循环等待:   T1→T2→T1 形成等待环路
  ★ 破坏任一条件即可预防死锁

重点观察 2 — InnoDB 死锁检测机制
  检测算法: wait-for graph（等待图）
  检测到环路 → 选择回滚代价小的事务
  T2 被回滚，错误信息: "Deadlock found when trying to get lock; try restarting transaction"
  ★ InnoDB 的死锁检测是主动的（不是等超时），通常毫秒级发现
  ★ innodb_deadlock_detect=ON（默认开启）

重点观察 3 — 死锁排查 SOP
  Step 1: SHOW ENGINE INNODB STATUS → 搜索 LATEST DETECTED DEADLOCK
  Step 2: 查看锁等待（MySQL 8.0 用 performance_schema.data_lock_waits）
  Step 3: 查看当前事务（information_schema.INNODB_TRX）
  Step 4: 定位 SQL → 调整加锁顺序
  ★ 生产环境建议开启 innodb_print_all_deadlocks=ON 记录到错误日志

重点观察 4 — 6 种预防策略
  ┌───────────────────────┬──────────────────────────────────────┐
  │ 策略                  │ 原理                                 │
  ├───────────────────────┼──────────────────────────────────────┤
  │ 固定加锁顺序 ★       │ 破坏循环等待（本例按 SKU 升序即可）  │
  │ 缩短事务              │ 持锁时间短 → 死锁窗口小             │
  │ 降低隔离级别到 RC     │ 没有 Gap Lock → 死锁概率降低        │
  │ 合理使用索引          │ 避免全表扫描导致锁范围过大           │
  │ 乐观锁替代            │ version 字段 CAS 更新，无锁设计      │
  │ 设置锁等待超时        │ innodb_lock_wait_timeout=5s          │
  └───────────────────────┴──────────────────────────────────────┘
  ★ 最有效的是"固定加锁顺序"，本例中所有事务按 SKU 字母序加锁即可避免

结论：
  死锁是并发系统的常见问题，InnoDB 能自动检测并回滚
  面试标准回答：1) SHOW ENGINE INNODB STATUS 看死锁日志
  2) 找到两个事务的 SQL 和加锁顺序 3) 统一加锁顺序/缩短事务/降级到 RC
