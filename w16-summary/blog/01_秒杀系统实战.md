# 秒杀系统实战：从 QPS 3623 到 6711 的优化之路

> 基于 w09-seckill 项目的真实优化经验

## 一、项目背景

### 1.1 项目简介

秒杀系统是一个高并发场景的典型应用，需要在短时间内处理大量请求，同时保证数据一致性和系统稳定性。

**技术栈**：
- Spring Boot 2.7.x + Spring Cloud 2021.x
- MySQL 8.0 + Redis 7.0 + Kafka 3.3
- Nacos 2.2 + Sentinel 1.8 + Gateway 3.1
- Docker + Prometheus + Grafana

### 1.2 业务场景

- 10 万用户同时参与秒杀
- 库存 100 件
- 前 10 秒集中 80% 请求
- 瞬时 QPS 40000

## 三、优化历程

### 3.1 第一轮优化：Redis Lua 原子扣库存

**问题**：
- MySQL 扣��存性能差
- 存在超卖风险

**方案**：
```lua
-- Redis Lua 脚本
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock <= 0 then 
    return 0 
end
redis.call('DECR', KEYS[1])
return 1
```

**效果**：
- QPS：3623 → 4200（+16%）
- P99 延迟：45ms → 38ms

### 3.2 第二轮优化：Kafka 异步削峰

**问题**：
- 同步创建订单慢
- 数据库压力大

**方案**：
```java
// 秒杀成功后发送 Kafka 消息
kafkaTemplate.send("seckill-order", message);

// Kafka 消费者异步创建订单
@KafkaListener(topics = "seckill-order", concurrency = "10")
public void consume(SeckillMessage message) {
    orderService.create(message);
}
```

**效果**：
- QPS：4200 → 5100（+21%）
- P99 延迟：38ms → 32ms

### 3.3 第三轮优化：布隆过滤器

**问题**：
- 重复购买检查慢
- Redis 查询压力大

**方案**：
```java
// 布隆过滤器快速判断
BloomFilter<String> bloomFilter = BloomFilter.create(
    Funnels.stringFunnel(Charset.defaultCharset()),
    1_000_000,
    0.01
);

if (bloomFilter.mightContain(userId + ":" + activityId)) {
    return Result.fail("您已经购买过了");
}
```

**效果**：
- QPS：5100 → 5800（+14%）
- P99 延迟：32ms → 30ms

### 3.4 第四轮优化：连接池预热

**问题**：
- 冷启动慢
- 连接池未预热

**方案**：
```java
@Component

            pool.submit(() -> redisTemplate.opsForValue().get("warmup"));
        }
        
        // Kafka 连接池预热

            kafkaTemplate.send("warmup", "test");
        }
    }

```

**效果**：
- QPS：5800 → 6200（+7%）
- P99 ���迟：30ms → 29ms

### 3.5 第五轮优化：JVM 参数调优

**问题**：
- GC 停顿时间长
- 堆内存动态扩容

**方案**：
```bash
# JVM 参数
-Xms8g -Xmx8g                    # 固定堆大小
-XX:+UseG1GC                     # 使用 G1 GC
-XX:MaxGCPauseMillis=50          # 最大停顿 50ms
-XX:G1HeapRegionSize=16m         # Region 大小 16MB
-XX:+HeapDumpOnOutOfMemoryError  # OOM 时生成堆转储
```

**效果**：
- QPS：6200 → 6711（+8%）
- P99 延迟：29ms → 28ms
- GC 停顿：< 50ms

## 四、最终架构

### 4.1 四层过滤架构

```
用户（10 万请求）
  ↓
CDN + 前端（过滤 50%）→ 5 万请求
  ↓

  ↓

  ↓

```

```
第 2 道：MySQL 乐观锁
第 3 道：唯一索引（activity_id, user_id）
```

## 五、核心代码

### 5.1 秒杀接口

```java
@PostMapping("/seckill/{activityId}")
public Result<String> seckill(@PathVariable Long activityId, 
                               @RequestHeader("X-User-Id") Long userId) {
    // 1. 限流（Gateway 5000 QPS）
    // 2. 频率限制（Redis SETNX，1 req/s/用户）
    String lockKey = "freq:" + userId;
    Boolean locked = redisTemplate.opsForValue()
        .setIfAbsent(lockKey, "1", 1, TimeUnit.SECONDS);
    if (!locked) {
        return Result.fail("请求过于频繁");
    }
    
    // 3. 布隆过滤器（快速判断是否已购买）
    if (bloomFilter.mightContain(userId + ":" + activityId)) {
        return Result.fail("您已经购买过了");
    }
    
    // 4. Redis Lua 原子扣库存
    Long result = redisTemplate.execute(luaScript, 
        Collections.singletonList("stock:" + activityId));
    if (result == 0) {
        return Result.fail("库存不足");
    }
    
    // 5. 发送 Kafka 消息（异步创建订单）
    SeckillMessage message = new SeckillMessage(activityId, userId);
    kafkaTemplate.send("seckill-order", message);
    
    // 6. 添加到布隆过滤器
    bloomFilter.put(userId + ":" + activityId);
    
    return Result.success("下单成功");
}
```

### 5.2 Kafka 消费者

```java
@KafkaListener(topics = "seckill-order", concurrency = "10")
public void consume(SeckillMessage message) {
    try {
        // 创建订单
        Order order = new Order();
        order.setActivityId(message.getActivityId());
        order.setUserId(message.getUserId());
        order.setOrderNo("SK_" + System.currentTimeMillis());
        
        // MySQL 乐观锁扣库存
        int rows = activityMapper.deductStock(message.getActivityId());
        if (rows == 0) {
            log.warn("库存不足，订单创建失败");
            return;
        }
        
        // 保存订单
        orderMapper.insert(order);
        
        log.info("订单创建成功：{}", order.getOrderNo());
    } catch (Exception e) {
        log.error("订单创建失败", e);
        // Kafka 会自动重试
    }
}
```

## 七、经验总结

### 7.1 优化思路

1. **找瓶颈**：压测 → 分析 → 定位瓶颈
2. **定方案**：针对瓶颈设计优化方案
3. **验证效果**：压测验证 → 对比数据
4. **持续优化**：循环迭代

### 7.2 核心技术

- **Redis Lua**：原子操作，性能提升 16%
- **Kafka 异步**：削峰填谷，性能提升 21%
- **布隆过滤器**：快速判断，性能提升 14%
- **连接池预热**：避免冷启动，性能提升 7%
- **JVM 调优**：减少 GC 停顿，性能提升 8%

### 7.3 注意事项

1. **防超卖**：三道防线（Redis + MySQL + 唯一索引）
2. **限流保护**：多层限流（Gateway + 用户级）
3. **监控告警**：实时监控 + 异常告警
4. **压测验证**：每次优化都要压测验证
