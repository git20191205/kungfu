========================================
  消息重复消费 — 幂等性设计
========================================

=== 一、为什么消息会被重复消费 ===

  ┌──────────────────────────────────────────────────────────────┐
  │ 场景1: Consumer Rebalance                                    │
  │   Consumer 处理完消息但未提交 offset → Rebalance 发生        │
  │   → 新 Consumer 从旧 offset 开始消费 → 重复                 │
  │                                                              │
  │ 场景2: Consumer 宕机                                         │
  │   处理完消息 → 准备提交 offset → 宕机                       │
  │   → 重启后从未提交的 offset 开始 → 重复                     │
  │                                                              │
  │ 场景3: Producer 重试                                         │
  │   Producer 发送成功但未收到 ACK → 重试 → Broker 收到两条     │
  └──────────────────────────────────────────────────────────────┘

  ★ 结论：At-Least-Once 语义下，重复消费不可避免
    → 消费端必须做幂等处理

=== 二、方案1: 数据库唯一键去重 ===

  模拟同一消息消费 3 次: msgId=8c21f342-de24-43d2-94ac-c0064e9a1d36

  第 1 次消费: ✓ 首次消费，执行业务逻辑
  第 2 次消费: ✗ 已消费过，跳过（幂等）
  第 3 次消费: ✗ 已消费过，跳过（幂等）

=== 三、方案2: 业务状态机幂等 ===

  订单状态机: CREATED → PAID → SHIPPED → DELIVERED
  规则: 只能正向流转，不能回退

  模拟：收到 3 次「支付成功」消息

  第 1 次收到支付消息: ✓ 状态 CREATED→PAID，执行业务
  第 2 次收到支付消息: ✗ 状态已不是 CREATED，跳过（幂等）
  第 3 次收到支付消息: ✗ 状态已不是 CREATED，跳过（幂等）

  ★ 关键 SQL: UPDATE ... WHERE status='CREATED'
    第一次: affected=1（状态变更成功）
    后续:   affected=0（状态已变，不再处理）

=== 四、幂等策略对比 ===

  ┌──────────────────┬──────────────────────────┬──────────────────────────┐
  │ 方案             │ 实现方式                 │ 适用场景                  │
  ├──────────────────┼──────────────────────────┼──────────────────────────┤
  │ 唯一键去重 ★    │ msg_id 做 DB 主键/唯一索引│ 通用，最常用             │
  │ 状态机           │ WHERE status=前置状态    │ 有状态流转的业务         │
  │ Redis SETNX      │ SETNX msg_id + TTL      │ 高并发，允许短暂窗口     │
  │ 乐观锁           │ WHERE version=N          │ 更新操作                 │
  │ Token 机制       │ 请求前获取 token，用后删除│ 前端防重复提交           │
  └──────────────────┴──────────────────────────┴──────────────────────────┘

  ★ 面试必背：
    Q: 如何保证消息不被重复消费？
    A: 消费端做幂等处理：
       1) 用 msg_id 做数据库唯一键，消费前查重
       2) 业务层面用状态机（只能正向流转）
       3) 高并发场景用 Redis SETNX 做分布式去重
