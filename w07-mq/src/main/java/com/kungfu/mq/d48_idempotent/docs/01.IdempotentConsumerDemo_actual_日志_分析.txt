IdempotentConsumerDemo 日志分析

运行环境：JDK 1.8.0_441, Windows 11, MySQL localhost:3306

整体结果：两种幂等方案全部验证成功 — 同一消息消费 3 次只处理 1 次。

重点观察 1 — 唯一键去重效果：

  日志原文：
    第 1 次消费: ✓ 首次消费，执行业务逻辑
    第 2 次消费: ✗ 已消费过，跳过（幂等）
    第 3 次消费: ✗ 已消费过，跳过（幂等）

  分析：
    - 第 1 次：t_consume_log 中无记录 → 处理业务 + INSERT 消费记录
    - 第 2/3 次：SELECT 发现 msg_id 已存在 → 直接跳过
    - 核心：msg_id 做 PRIMARY KEY，天然去重
    - 业务处理 + 消费记录在同一个事务中 → 要么都成功，要么都回滚

重点观察 2 — 状态机幂等效果：

  日志原文：
    第 1 次收到支付消息: ✓ 状态 CREATED→PAID，执行业务
    第 2 次收到支付消息: ✗ 状态已不是 CREATED，跳过（幂等）
    第 3 次收到支付消息: ✗ 状态已不是 CREATED，跳过（幂等）

  分析：
    - SQL: UPDATE ... WHERE status='CREATED'
    - 第 1 次：affected=1（状态从 CREATED 变为 PAID）
    - 第 2/3 次：affected=0（状态已是 PAID，WHERE 不匹配）
    - 无需额外去重表，利用业务状态本身做幂等
    - 适用于有明确状态流转的业务（订单、支付、物流）

重点观察 3 — 两种方案的选择：

  唯一键去重：通用方案，任何场景都能用，但需要额外的去重表
  状态机幂等：更优雅，不需要额外表，但只适用于有状态流转的业务

  生产建议：两者结合使用
  - 外层用 Redis SETNX 快速去重（挡住 99% 的重复）
  - 内层用数据库唯一键兜底（防止 Redis 故障时的极端情况）

结论：
  1. At-Least-Once 语义下重复消费不可避免，必须做幂等
  2. 唯一键去重是最通用的方案（msg_id 做主键）
  3. 状态机幂等更优雅（WHERE status=前置状态）
  4. 生产建议：Redis SETNX 快速去重 + DB 唯一键兜底
