========================================
  手写 Mini 熔断器 — 熔断降级原理
========================================

=== 一、熔断器原理 ===

  ┌──────────────────────────────────────────────────────────────┐
  │                    熔��器状态机                                │
  │                                                              │
  │     ┌──────────┐   失败达到阈值   ┌──────────┐              │
  │     │ CLOSED   │ ──────────────→ │  OPEN    │              │
  │     │ (正常)   │                  │ (熔断)   │              │
  │     └────┬─────┘ ←────────────── └────┬─────┘              │
  │          ↑       探测成功恢复          │                     │
  │          │                        超时到期                    │
  │          │       ┌──────────┐         │                     │
  │          └────── │HALF_OPEN │ ←───────┘                     │
  │     探测成功     │ (半开)   │ ──→ 探测失败回到 OPEN          │
  │                  └──────────┘                                │
  └──────────────────────────────────────────────────────────────┘

  配置参数：
    ├─ failureThreshold = 5  (连续失败 5 次触发熔断)
    ├─ timeout = 5s          (熔断 5 秒后进入半开)
    └─ halfOpenProbes = 1    (半开状态允许 1 个探测请求)

=== 二、模拟熔断全流程 ===

  ── 阶段1: 正常请求 (CLOSED) ──
    请求 #1 → OK: req-1776657772441
    请求 #2 → OK: req-1776657772441
    请求 #3 → OK: req-1776657772441

  ── 阶段2: 服务故障，失败累积 ──
      [order-service] [✓] CLOSED — 失败 1/5
    请求 #1 → FALLBACK: 降级响应
      [order-service] [✓] CLOSED — 失败 2/5
    请求 #2 → FALLBACK: 降级响应
      [order-service] [✓] CLOSED — 失败 3/5
    请求 #3 → FALLBACK: 降级响应
      [order-service] [✓] CLOSED — 失败 4/5
    请求 #4 → FALLBACK: 降级响应

    请求 #5 → FALLBACK: 降级响应
      [order-service] [✗] OPEN — OPEN 状态，直接降级 (剩余 3000ms)
    请求 #6 → FALLBACK: 降级响应
    状态: OPEN

  ── 阶段3: OPEN 状态，请求直接降级 ──
      [order-service] [✗] OPEN — OPEN 状态，直接降级 (剩余 2999ms)
    请求 #1 → FALLBACK: 降级响应
      [order-service] [✗] OPEN — OPEN 状态，直接降级 (剩余 2999ms)
    请求 #2 → FALLBACK: 降级响应
      [order-service] [✗] OPEN — OPEN 状态，直接降级 (剩余 2999ms)
    请求 #3 → FALLBACK: 降级响应

  ── 阶段4: 等待 3 秒超时... ──
  ── 阶段5: 服务恢复，半开探测 ──
      [order-service] [?] HALF_OPEN — 超时到期，切换到 HALF_OPEN，允许探测

    探测请求 → OK: probe-request
    状态: CLOSED

  ── 阶段6: 恢复正常 (CLOSED) ──
    请求 #1 → OK: req-recovered
    请求 #2 → OK: req-recovered
    请求 #3 → OK: req-recovered
    状态: CLOSED

=== 三、滑动窗口 ===

  1. 计数滑动窗口（Sentinel 默认）
  ┌───┬───┬───┬───┬───┬───┬───┬───┬───┬───┐
  │ ✓ │ ✓ │ ✗ │ ✓ │ ✗ │ ✗ │ ✓ │ ✗ │ ✗ │ ✗ │ ← 最近 10 次请求
  └───┴───┴───┴───┴───┴───┴───┴───┴───┴───┘
    失败率 = 6/10 = 60% > 阈值 50% → 触发熔断

  2. 时间滑动窗口（按时间段统计）
  ┌──────┬──────┬──────┬──────┬──────┐
  │ 0-1s │ 1-2s │ 2-3s │ 3-4s │ 4-5s │ ← 最近 5 秒
  │ 2/10 │ 1/8  │ 5/12 │ 8/10 │ 9/10 │
  └──────┴──────┴──────┴──────┴──────┘
    最近 5s 失败率 = 25/50 = 50% → 触发熔断

  ★ 区别：
    计数窗口: 统计最近 N 次请求的失败率
    时间窗口: 统计最近 N 秒内的失败率

=== 四、熔断框架对比 ===

  ┌──────────────┬──────────────────┬──────────────────┬──────────────────┐
  │              │ Sentinel ★      │ Hystrix          │ Resilience4j     │
  ├──────────────┼──────────────────┼──────────────────┼──────────────────┤
  │ 维护状态     │ ★ 活跃(阿里)    │ ✗ 停止维护      │ ✓ 活跃          │
  │ 隔离策略     │ 信号量           │ 线程池/信号量    │ 信号量           │
  │ 熔断策略     │ 慢调用/异常比例  │ 异常比例         │ 异常比例/慢调用  │
  │ 流控         │ ✓ QPS/线程数    │ ✗               │ ✓ 速率限制      │
  │ 控制台       │ ✓ Dashboard     │ ✓ Dashboard     │ ✗               │
  │ 动态规则     │ ✓ Nacos/Apollo  │ ✗               │ ✗               │
  │ 适用场景     │ Spring Cloud     │ 老项目           │ 轻量级           │
  └──────────────┴──────────────────┴──────────────────┴──────────────────┘

=== 五、Sentinel 规则体系 ===

  ┌──────────────────────────────────────────────────────────────┐
  │                    Sentinel 规则                               │
  │                                                              │
  │  1. 流控规则 (FlowRule)                                      │
  │     ├─ QPS 模式: 每秒请求数超过阈值 → 拒绝/排队/预热         │
  │     └─ 线程数模式: 并发线程超过阈值 → 拒绝                    │
  │                                                              │
  │  2. 降级规则 (DegradeRule)                                   │
  │     ├─ 慢调用比例: RT > 阈值的比例超限 → 熔断                 │
  │     ├─ 异常比例: 异常请求比例超限 → 熔断                      │
  │     └─ 异常数: 异常请求数超限 → 熔断                          │
  │                                                              │
  │  3. 系统保护规则 (SystemRule)                                 │
  │     ├─ LOAD: 系统负载超限                                     │
  │     ├─ CPU: CPU 使用率超限                                    │
  │     └─ 总 QPS / 线程数 / RT                                  │
  └──────────────────────────────────────────────────────────────┘

  ★ 面试必背：
    Q: Sentinel 和 Hystrix 的区别？
    A: 1) Sentinel 支持流控+熔断+系统保护，Hystrix 只有熔断
       2) Sentinel 用信号量隔离（轻量），Hystrix 用线程池（重）
       3) Sentinel 有 Dashboard 实时监控 + 动态规则推送
       4) Hystrix 已停止维护，Sentinel 是阿里主推方案
