【重点观察 1】状态机转换精确性
  ★ CLOSED → OPEN: 连续失败 5 次（failureThreshold=5）
  ├─ 日志清晰记录: 失败 1/5 → 2/5 → ... → 5/5 触发
  ├─ 第 5 次失败时立即切换，第 6 次已是 OPEN 直接降级
  └─ 计数器在成功时重置（P1 正常请求不累积失败数）

【重点观察 2】OPEN 状态的快速失败
  ★ P3 阶段 3 次请求耗时几乎为 0（剩余 2999ms ≈ 3000ms）
  ├─ OPEN 状态不发起真实调用 → 保护下游服务
  ├─ 直接返回 FALLBACK 降级响应 → 用户体验可控
  └─ 这就是"熔断"的核心价值: 防止级联故障（雪崩效应）

【重点观察 3】滑动窗口 — 生产级熔断策略
  ★ 本 Demo 用连续失败计数（简化版）
  ├─ Sentinel: 计数滑动窗口（最近 N 次请求的失败率）
  ├─ 日志示例: 6/10=60% > 50% 阈值 → 触发熔断
  ├─ 时间滑动窗口: 最近 5s 内 25/50=50% → 触发
  └─ 面试区分: 计数窗口看请求数，时间窗口看时间段

【结论】
  ✅ 熔断器三态: CLOSED(放行) → OPEN(拒绝) → HALF_OPEN(探测)
  ✅ 核心参数: failureThreshold / timeout / halfOpenProbes
  ★ 面试关键词: 状态机 / 快速失败 / 滑动窗口 / 降级兜底
  ★ Sentinel vs Hystrix: Sentinel 活跃维护 + 流控 + Dashboard
