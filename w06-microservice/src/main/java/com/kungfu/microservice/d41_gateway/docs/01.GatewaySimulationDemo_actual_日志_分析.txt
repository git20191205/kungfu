【重点观察 1】过滤器链执行顺序与短路机制
  ★ Pre-Filters 执行顺序: Log → RateLimit → Auth
  ├─ 任一 Filter 返回 false → 短路，直接返回错误码
  ├─ S2: Auth 失败 → 短路返回 401（不执行 Route）
  ├─ S4: RateLimit 失败 → 短路返回 429（不执行 Auth）
  └─ 类似 Servlet FilterChain（D16 Spring MVC）

【重点观察 2】令牌桶限流算法
  ★ 初始令牌数 = 3（场景 S1/S2/S3 各消耗 1 个）
  ├─ S1: 剩余 2.0 → S2: 剩余 1.0 → S3: 剩余 0.0
  ├─ S4 起: 令牌耗尽，连续 429
  ├─ 令牌桶 vs 漏桶: 令牌桶允许突发流量，漏桶匀速
  └─ Gateway 生产配置: redis-rate-limiter + replenishRate

【重点观察 3】路由匹配规则
  ★ Path 谓词匹配:
  ├─ /api/order/** → order-service:8081
  ├─ /api/stock/** → stock-service:8082
  ├─ /api/auth/**  → auth-service:8083
  └─ 无匹配 → 被限流拦截（未到路由阶段）

【结论】
  ✅ 网关三大核心: 路由转发 + 过滤器链 + 限流保护
  ✅ 过滤器短路机制避免无效请求穿透到后端服务
  ★ 面试关键词: 过滤器链 / 令牌桶限流 / 白名单 / 谓词匹配
  ★ Gateway vs Zuul: Gateway 基于 Netty+WebFlux 异步非阻塞，性能更优
