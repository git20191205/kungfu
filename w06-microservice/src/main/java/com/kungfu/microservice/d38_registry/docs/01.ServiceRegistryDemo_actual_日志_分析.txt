  ServiceRegistryDemo 日志分析
========================================

【整体结果】✅ 手写 Mini 注册中心完整演示成功
  注册 → 发现 → 心跳续约 → 超时剔除，四阶段全部验证通过

【逐步分析】

┌──────┬──────────────────────────────┬──────────────────────────────────────┐
│ 步骤 │ 操作                         │ 实际数据 & 分析                       │
├──────┼──────────────────────────────┼──────────────────────────────────────┤
│ S1   │ 注册 3 个 order-service 实例 │ 192.168.1.1/1.2/1.3:8080 全部成功    │
│      │                              │ ✓ 注册表 key=服务名, value=实例列表  │
├──────┼──────────────────────────────┼──────────────────────────────────────┤
│ S2   │ 服务发现                     │ 返回 3 个实例，实例数=3              │

└──────┴──────────────────────────────┴──────────────────────────────────────┘

【重点观察 1】注册表数据结构
  ★ ConcurrentHashMap<String, List<Instance>>
  ├─ key: 服务名 "order-service"
  ├─ value: 实例列表 [ip:port + 元数据]
  └─ 与 Nacos 的 ServiceInfo 结构一致

【重点观察 2】心跳超时剔除机制
  ★ 心跳间隔 5s，超时阈值 > 5s（日志等待 6s）
  ├─ order3 未续约 → 最后心跳时间过期 → 被剔除
  ├─ Nacos 临时实例: 15s 未心跳标记不健康，30s 剔除
  └─ Eureka: 90s 未心跳剔除（自我保护模式下不剔除）

【重点观察 3】CAP 理论在注册中心的体现
  ★ Nacos AP 模式: 临时实例，心跳检测，最终一致性
  ★ Nacos CP 模式: 持久实例，Raft 协议，强一致性
  ├─ 面试高频: Nacos 如何同时支持 AP 和 CP？
  └─ 答: 临时实例走 Distro(AP)，持久实例走 Raft(CP)

【结论】
  ✅ 注册中心核心三件事: 注册、发现、健康检查
  ✅ 心跳超时剔除是保证服务列表准确性的关键
  ★ 面试关键词: ConcurrentHashMap 注册表 / 心跳续约 / 超时剔除
  ★ 与 Nacos 对比: 本Demo简化了推送通知机制（Nacos 用长轮询+推送）
