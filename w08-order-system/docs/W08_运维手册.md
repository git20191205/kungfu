### 2.5 Seata 1.7.1

#### 2.5.1 建库

在 MySQL 中执行（只需一次）：

```sql
CREATE DATABASE IF NOT EXISTS seata_server DEFAULT CHARACTER SET utf8mb4;
USE seata_server;

CREATE TABLE IF NOT EXISTS global_table (
    xid                       VARCHAR(128)  NOT NULL,
    transaction_id            BIGINT,
    status                    TINYINT       NOT NULL,
    application_id            VARCHAR(32),
    transaction_service_group VARCHAR(32),
    transaction_name          VARCHAR(128),
    timeout                   INT,
    begin_time                BIGINT,
    application_data          VARCHAR(2000),
    gmt_create                DATETIME,
    gmt_modified              DATETIME,
    PRIMARY KEY (xid),
    KEY idx_status_gmt_modified (status, gmt_modified),
    KEY idx_transaction_id (transaction_id)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4;

CREATE TABLE IF NOT EXISTS branch_table (
    branch_id         BIGINT        NOT NULL,
    xid               VARCHAR(128)  NOT NULL,
    transaction_id    BIGINT,
    resource_group_id VARCHAR(32),
    resource_id       VARCHAR(256),
    branch_type       VARCHAR(8),
    status            TINYINT,
    client_id         VARCHAR(64),
    application_data  VARCHAR(2000),
    gmt_create        DATETIME(6),
    gmt_modified      DATETIME(6),
    PRIMARY KEY (branch_id),
    KEY idx_xid (xid)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4;

CREATE TABLE IF NOT EXISTS lock_table (
    row_key        VARCHAR(128) NOT NULL,
    xid            VARCHAR(128),
    transaction_id BIGINT,
    branch_id      BIGINT       NOT NULL,
    resource_id    VARCHAR(256),
    table_name     VARCHAR(32),
    pk             VARCHAR(36),
    status         TINYINT      NOT NULL DEFAULT 0,
    gmt_create     DATETIME,
    gmt_modified   DATETIME,
    PRIMARY KEY (row_key),
    KEY idx_branch_id (branch_id),
    KEY idx_xid (xid)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4;

CREATE TABLE IF NOT EXISTS distributed_lock (
    lock_key       CHAR(20)    NOT NULL,
    lock_value     VARCHAR(20) NOT NULL,
    expire         BIGINT,
    PRIMARY KEY (lock_key)
) ENGINE = InnoDB DEFAULT CHARSET = utf8mb4;
```

## 四、启动顺序

严格按以下顺序启动，每个服务开一个独立的终端窗口：

### 第一层：中间件

```
1. MySQL        → 已运行
2. Redis        → 已运行
3. ZooKeeper    → 已运行（Kafka 依赖）
4. Kafka        → 已运行
5. Nacos        → startup.cmd -m standalone
6. Seata        → seata-server.bat -p 8091
```

等待 Nacos 和 Seata 完全启动（各约 10-20 秒）。

### 第二层：业务服务（无依赖关系的先启动）

```bash
# 终端 1: 库存服务（无依赖其他业务服务）
cd D:\workspace\kungfu\w08-order-system\w08-stock-service
mvn spring-boot:run

# 终端 2: 支付服务（无依赖其他业务服务）
cd D:\workspace\kungfu\w08-order-system\w08-payment-service
mvn spring-boot:run
```

等待两个服务启动完成（看到 `Started XxxApplication in x.xx seconds`）。

### 第三层：订单服务（依赖 stock + payment）

```bash
# 终端 3: 订单服务
cd D:\workspace\kungfu\w08-order-system\w08-order-service
mvn spring-boot:run
```

### 第四层：网关（依赖所有业务服务）

```bash
# 终端 4: API 网关
cd D:\workspace\kungfu\w08-order-system\w08-gateway

```

## 七、常见问题排查

### 7.1 服务启动失败

| 错误信息 | 原因 | 解决 |
|---------|------|------|
| `Connection refused: localhost:8848` | Nacos 未启动 | 先启动 Nacos |
| `service.grouplist.default configuration item is required` | Seata 配置缺少 TC 地址 | application.yml 中加 `seata.service.grouplist.default: localhost:8091` |
| `Access denied for user 'root'@'localhost'` | MySQL 密码错误 | 检查 application.yml 中的 password |
| `Unknown database 'kungfu_order'` | 数据库未初始化 | 运行 DatabaseInitializer |
| `Port already in use: 8081` | 端口被占用 | 杀掉占用进程或改端口 |
| `No qualifying bean of type 'xxxMapper'` | MyBatis 扫描路径错误 | 检查 @MapperScan 注解 |

### 7.2 下单失败

| 错误信息 | 原因 | 解决 |
|---------|------|------|
| `No available service` | Feign 找不到目标服务 | 1) 检查目标服务是否启动 2) 检查 Nacos 服务列表 3) 等 10 秒重试 |
| `库存扣减失败` | 库存不足或 Redis 锁超时 | 检查 t_stock 表的 quantity |
| `支付单创建失败` | payment-service 异常 | 查看 payment-service 终端日志 |
| `Read timed out` | Feign 调用超时 | 目标服务响应慢，检查 DB 连接 |

### 7.3 订单状态一直是 PENDING

原因：Kafka 支付结果消息未被消费

排查步骤：

```bash
# 1. 检查 payment-service 是否发送了 Kafka 消息
# 查看 payment-service 终端日志，搜索 "Kafka" 或 "payment-result"

# 2. 检查 Kafka topic 是否有消息
kafka-console-consumer.bat --bootstrap-server localhost:9092 ^
  --topic payment-result --from-beginning --max-messages 1

# 3. 检查 order-service 的 Kafka consumer 是否正常
# 查看 order-service 终端日志，搜索 "收到支付结果" 或 "PaymentResultConsumer"

# 4. 检查 consumer group offset
kafka-consumer-groups.bat --bootstrap-server localhost:9092 ^
  --describe --group order-service-group
```

### 7.4 Seata 分布式事务不生效

排查步骤：

```bash
# 1. 检查 Seata Server 是否运行
powershell -Command "(Test-NetConnection localhost -Port 8091).TcpTestSucceeded"

# 2. 检查 order-service 日志中是否有 Seata 注册信息
# 搜索 "register TM" 或 "register RM"

# 3. 检查 undo_log 表
# 正常提交后 undo_log 会被清理
# 如果回滚，undo_log 中会有记录然后被清理
USE kungfu_order;
SELECT * FROM undo_log;

# 4. 检查 Seata Server 控制台
# http://localhost:7091 → 全局事务列表
```

### 7.5 Gateway 路由不通

```bash
# 1. 直接访问目标服务（绕过 Gateway）
curl http://localhost:8082/api/stock/1
# 如果直接访问正常，说明是 Gateway 问题

# 2. 检查 Gateway 日志
# 搜索 "RouteDefinition" 或 "No route found"

# 3. 检查 Nacos 中服务名是否匹配
# Gateway 的 uri: lb://stock-service
# Nacos 中的服务名必须是 stock-service（区分大小写）
```
