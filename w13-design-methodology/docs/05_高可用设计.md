# 高可用设计

> 系统设计第五步：保证系统 99.9% 以上的可用性

## 一、什么是高可用？

**高可用（High Availability, HA）**：系统能够持续提供服务的能力

**可用性计算**：

```
可用性 = 正常运行时间 / 总时间 × 100%

例如：
- 99%：年停机 3.65 天
- 99.9%：年停机 8.76 小时
- 99.99%：年停机 52.56 分钟
- 99.999%：年停机 5.26 分钟
```

**为什么需要高可用？**
- 避免单点故障
- 减少停机时间
- 提升用户体验
- 降低业务损失

## 二、冗余设计

### 2.1 服务冗余

**单机 → 多机**

```
单机（可用性 99%）：
┌──────────┐
│ Service  │ ← 单点故障
└──────────┘

多机（可用性 99.99%）：
┌──────────┐  ┌──────────┐  ┌──────────┐
│ Service1 │  │ Service2 │  │ Service3 │
└──────────┘  └──────────┘  └──────────┘
       ↑             ↑             ↑
       └─────────────┴─────────────┘
              负载均衡
```

**计算**：
- 单机可用性：99%
- 3 台机器同时故障概率：(1 - 0.99)³ = 0.0001%
- 3 台机器可用性：1 - 0.0001% = 99.9999%

### 2.2 数据库冗余

**主从复制**

```
┌──────────┐
│  Master  │ ← 写
└────┬─────┘
     │ 同步
     ├─────────┬─────────┐
     ↓         ↓         ↓
┌────────┐ ┌────────┐ ┌────────┐
│ Slave1 │ │ Slave2 │ │ Slave3 │ ← 读
└────────┘ └────────┘ └────────┘
```

**优点**：
- 读写分离，提升性能
- Master 故障，Slave 可以提升为 Master

**缺点**：
- 主从延迟（通常 < 1 秒）
- 需要手动或自动切换

### 2.3 缓存冗余

**Redis Cluster**

```
┌─────────────────────────────────────┐
│         Redis Cluster               │
├─────────────────────────────────────┤
│  Master1  ←→  Slave1                │
│  Master2  ←→  Slave2                │
│  Master3  ←→  Slave3                │
└─────────────────────────────────────┘
```

**优点**：
- 自动故障转移
- 数据分片，支持水平扩展

### 2.4 多机房冗余

**同城双活**

```
机房 A（主）          机房 B（备）
┌──────────┐        ┌──────────┐
│ Service  │ ←─────→│ Service  │
│ MySQL    │        │ MySQL    │
│ Redis    │        │ Redis    │
└──────────┘        └──────────┘
     ↑                    ↑
     └────────┬───────────┘
              │
         负载均衡（50% / 50%）
```

**异地多活**

```
北京机房          上海机房          深圳机房
┌──────────┐    ┌──────────┐    ┌──────────┐
│ Service  │    │ Service  │    │ Service  │
│ MySQL    │    │ MySQL    │    │ MySQL    │
│ Redis    │    │ Redis    │    │ Redis    │
└──────────┘    └──────────┘    └──────────┘
     ↑               ↑               ↑
     └───────────────┴───────────────┘
              全局负载均衡
```

## 三、故障检测与自动切换

### 3.1 健康检查

**HTTP 健康检查**

```java
@GetMapping("/actuator/health")
public Map<String, Object> health() {
    Map<String, Object> health = new HashMap<>();
    
    // 检查数据库
    boolean dbOk = checkDatabase();
    // 检查 Redis
    boolean redisOk = checkRedis();
    // 检查 Kafka
    boolean kafkaOk = checkKafka();
    
    if (dbOk && redisOk && kafkaOk) {
        health.put("status", "UP");
    } else {
        health.put("status", "DOWN");
    }
    
    return health;
}
```

**TCP 健康检查**

```bash
# Nginx 配置
upstream backend {
    server 192.168.1.10:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.11:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.12:8080 max_fails=3 fail_timeout=30s;
}

# 健康检查：连续 3 次失败，标记为不可用，30 秒后重试
```

### 3.2 自动故障转移

**MySQL 主从切换（MHA）**

```
1. 检测 Master 故障（心跳超时）
2. 选举新 Master（数据最新的 Slave）
3. 提升 Slave 为 Master
4. 修改应用配置，指向新 Master
5. 通知其他 Slave 同步新 Master
```

**Redis 哨兵（Sentinel）**

```
┌──────────┐
│  Master  │ ← 写
└────┬─────┘
     │
     ├─────────┬─────────┐
     ↓         ↓         ↓
┌────────┐ ┌────────┐ ┌────────┐
│ Slave1 │ │ Slave2 │ │ Slave3 │ ← 读
└────────┘ └────────┘ └────────┘
     ↑         ↑         ↑
     └─────────┴─────────┘
          哨兵监控

故障转移流程：
1. 哨兵检测 Master 故障
2. 哨兵投票选举新 Master
3. 提升 Slave 为 Master
4. 通知客户端新 Master 地址
```

## 四、限流 / 熔断 / 降级

### 4.1 限流（Rate Limiting）

**目的**：防止系统过载

**算法 1：固定窗口**

```java
// 每秒最多 100 个请求
AtomicInteger counter = new AtomicInteger(0);
long windowStart = System.currentTimeMillis() / 1000;

public boolean tryAcquire() {
    long now = System.currentTimeMillis() / 1000;
    if (now > windowStart) {
        // 新窗口，重置计数器
        counter.set(0);
        windowStart = now;
    }
    
    return counter.incrementAndGet() <= 100;
}
```

**算法 2：滑动窗口**

```java
// Redis 实现
String key = "rate_limit:" + userId;
long now = System.currentTimeMillis();
long windowStart = now - 1000; // 1 秒窗口

// 1. 删除过期数据
redisTemplate.opsForZSet().removeRangeByScore(key, 0, windowStart);

// 2. 统计当前窗口请求数
Long count = redisTemplate.opsForZSet().zCard(key);

// 3. 判断是否超限
if (count < 100) {
    // 4. 添加当前请求
    redisTemplate.opsForZSet().add(key, UUID.randomUUID().toString(), now);
    return true;
}

return false;
```

**算法 3：令牌桶**

```java
// Guava RateLimiter
RateLimiter rateLimiter = RateLimiter.create(100); // 每秒 100 个令牌

public boolean tryAcquire() {
    return rateLimiter.tryAcquire(100, TimeUnit.MILLISECONDS);
}
```

**秒杀系统限流策略**：

```
Nginx 限流：100 req/s/IP
  ↓
Gateway 限流：5000 QPS（全局）
  ↓
Service 限流：1 req/s/用户
  ↓
Redis 限流：用户级别（布隆过滤器）
```

### 4.2 熔断（Circuit Breaker）

**目的**：防止故障扩散

**状态机**：

```
关闭（Closed）
  ↓ 失败率 > 50%
打开（Open）← 拒绝所有请求
  ↓ 等待 30 秒
半开（Half-Open）← 允许少量请求
  ↓ 成功率 > 80%
关闭（Closed）
```

**实现（Resilience4j）**：

```java
@CircuitBreaker(name = "orderService", fallbackMethod = "fallback")
public Order createOrder(Long activityId, Long userId) {
    // 调用订单服务
    return orderService.create(activityId, userId);
}

public Order fallback(Long activityId, Long userId, Exception e) {
    // 熔断后的降级逻辑
    log.error("订单服务熔断", e);
    throw new ServiceUnavailableException("订单服务暂时不可用");
}
```

**配置**：

```yaml
resilience4j:
  circuitbreaker:
    instances:
      orderService:
        failure-rate-threshold: 50        # 失败率阈值 50%
        wait-duration-in-open-state: 30s  # 打开状态等待 30 秒
        sliding-window-size: 10           # 滑动窗口大小 10 个请求
```

### 4.3 降级（Degradation）

**目的**：保证核心功能可用

**降级策略**：

| 功能 | 优先级 | 降级方案 |
|------|--------|---------|
| 秒杀下单 | P0 | 不降级 |
| 订单查询 | P1 | 返回缓存数据 |
| 活动详情 | P1 | 返回缓存数据 |
| 推荐商品 | P2 | 关闭功能 |
| 评论列表 | P2 | 关闭功能 |

**实现**：

```java
@GetMapping("/activities/{id}")
public Result<Activity> getActivity(@PathVariable Long id) {
    // 1. 检查是否降级
    if (degradationSwitch.isOn("activity_detail")) {
        // 2. 返回缓存数据
        Activity activity = redisTemplate.opsForValue().get("activity:" + id);
        if (activity != null) {
            return Result.success(activity);
        }
        // 3. 缓存也没有，返回默认数据
        return Result.success(Activity.defaultActivity());
    }
    
    // 4. 正常查询数据库
    Activity activity = activityService.getById(id);
    return Result.success(activity);
}
```

## 五、超时与重试策略

### 5.1 超时设置

**原则**：每个外部调用都要设置超时

**超时时间设置**：

| 调用类型 | 超时时间 | 说明 |
|---------|---------|------|
| HTTP 调用 | 3-5 秒 | 根据 P99 延迟设置 |
| 数据库查询 | 1-3 秒 | 简单查询 1 秒，复杂查询 3 秒 |
| Redis 操作 | 100-500 ms | 纯内存操作，应该很快 |
| Kafka 发送 | 1-3 秒 | 批量发送 |

**实现**：

```java
// HTTP 调用超时
RestTemplate restTemplate = new RestTemplate();
HttpComponentsClientHttpRequestFactory factory = new HttpComponentsClientHttpRequestFactory();
factory.setConnectTimeout(3000);  // 连接超时 3 秒
factory.setReadTimeout(5000);     // 读取超时 5 秒
restTemplate.setRequestFactory(factory);

// 数据库查询超时
@Select("SELECT * FROM seckill_order WHERE id = #{id}")
@Options(timeout = 3)  // 3 秒超时
Order selectById(Long id);

// Redis 操作超时
RedisTemplate<String, Object> redisTemplate = new RedisTemplate<>();
redisTemplate.setConnectionFactory(connectionFactory);
redisTemplate.expire(key, 5, TimeUnit.SECONDS);  // 5 秒超时
```

### 5.2 重试策略

**什么时候重试？**
- 网络抖动（超时）
- 服务暂时不可用（503）
- 幂等操作（GET / PUT / DELETE）

**什么时候不重试？**
- 参数错误（400）
- 资源不存在（404）
- 非幂等操作（POST）

**重试策略**：

```java
// Spring Retry
@Retryable(
    value = {TimeoutException.class, ServiceUnavailableException.class},
    maxAttempts = 3,
    backoff = @Backoff(delay = 1000, multiplier = 2)
)
public Order createOrder(Long activityId, Long userId) {
    // 调用订单服务
    return orderService.create(activityId, userId);
}

// 重试 3 次，延迟 1 秒、2 秒、4 秒（指数退避）
```

**注意事项**：
- 只重试幂等操作
- 设置最大重试次数（通常 3 次）
- 使用指数退避（避免雪崩）
- 记录重试日志

## 六、数据一致性（CAP / BASE）

### 6.1 CAP 理论

**CAP**：
- **C（Consistency）**：一致性，所有节点看到相同的数据
- **A（Availability）**：可用性，系统一直可以响应请求
- **P（Partition Tolerance）**：分区容错性，网络分区时系统仍能工作

**定理**：分布式系统只能同时满足 CAP 中的两个

**选择**：
- **CP**：牺牲可用性，保证一致性（如 ZooKeeper）
- **AP**：牺牲一致性，保证可用性（如 Cassandra）
- **CA**：不存在（网络分区无法避免）

**秒杀系统选择**：
- 库存扣减：CP（强一致性，不能超卖）
- 订单查询：AP（最终一致性，可以延迟）

### 6.2 BASE 理论

**BASE**：
- **BA（Basically Available）**：基本可用
- **S（Soft State）**：软状态，允许中间状态
- **E（Eventually Consistent）**：最终一致性

**秒杀系统案例**：

```
1. 用户秒杀成功（Redis 扣库存）
   ↓ 基本可用
2. 发送 Kafka 消息（异步）
   ↓ 软状态（订单创建中）
3. 消费 Kafka 消息，创建订单
   ↓ 最终一致性（订单创建完成）
```

**最终一致性实现**：

```java
// 1. Redis 扣库存（同步）
Long result = redisTemplate.execute(luaScript, 
    Collections.singletonList("stock:" + activityId));

if (result == 0) {
    return Result.fail("库存不足");
}

// 2. 发送 Kafka 消息（异步）
kafkaTemplate.send("seckill-order", message);

// 3. 消费 Kafka 消息，创建订单
@KafkaListener(topics = "seckill-order")
public void consume(SeckillMessage message) {
    // 创建订单
    orderService.create(message);
    
    // 如果失败，Kafka 会自动重试
}

// 4. 定时任务对账（兜底）
@Scheduled(cron = "0 */10 * * * ?")  // 每 10 分钟
public void reconcile() {
    // 对比 Redis 和 MySQL 的库存
    // 如果不一致，以 MySQL 为准
}
```

### 7.2 可用性计算

```
组件可用性：
- Nginx：99.9%（主备）
- Gateway：99.99%（3 台）
- Seckill：99.99%（4 台）
- Redis：99.99%（Cluster）
- MySQL：99.9%（主从）
- Kafka：99.99%（Cluster）

系统可用性 = 99.9% × 99.99% × 99.99% × 99.99% × 99.9% × 99.99%
           ≈ 99.7%

年停机时间 = 365 × 24 × (1 - 0.997) = 26 小时
```

### 7.3 故障场景与应对

| 故障场景 | 影响 | 应对方案 |
|---------|------|---------|

| 机房断电 | 部分不可用 | 切换到备用机房 |

## 九、高可用设计速查表

### 9.1 冗余 Checklist

```
□ 服务冗余：多机部署（≥ 3 台）
□ 数据库冗余：主从复制（1 主 2 从）
□ 缓存冗余：Redis Cluster（3 主 3 从）
□ 消息队列冗余：Kafka Cluster（3 节点）
□ 机房冗余：同城双活 / 异地多活
```

### 9.2 保护措施 Checklist

```
□ 限流：IP / 用户 / 接口
□ 熔断：失败率 > 50% 打开
□ 降级：关闭非核心功能
□ 超时：所有外部调用设置超时
□ 重试：幂等操作重试 3 次
```

### 9.3 监控告警 Checklist

```
□ 健康检查：HTTP / TCP
□ 心跳监控：每 10 秒一次
□ 故障告警：钉钉 / 企业微信
□ 自动切换：哨兵 / MHA
□ 日志收集：ELK / Loki
```

## 十、总结

高可用设计是系统稳定运行的保障，需要从多个维度考虑。

**核心要点**：
1. **冗余**：服务多机、数据库主从、Redis Cluster
2. **故障检测**：健康检查、心跳监控
3. **自动切换**：负载均衡、哨兵、MHA
4. **保护措施**：限流、熔断、降级、超时、重试
5. **数据一致性**：强一致性（CP）vs 最终一致性（BASE）

**面试话术**：
- "我先识别一下单点..."
- "服务需要多机部署，至少 3 台..."
- "数据库需要主从复制，自动故障转移..."
- "需要限流、熔断、降级保护系统..."
