# 扩展性设计

> 系统设计第六步：支持系统从 1 台机器扩展到 1000 台

## 一、什么是扩展性？

**扩展性（Scalability）**：系统通过增加资源来提升性能的能力

**为什么需要扩展性？**
- 业务增长（用户从 10 万到 1000 万）
- 流量波动（秒杀、大促）
- 成本优化（按需扩容）

**扩展性 vs 性能**：
- 性能：单机能处理多少请求
- 扩展性：增加机器能提升多少性能

## 二、水平扩展 vs 垂直扩展

### 2.1 垂直扩展（Scale Up）

**定义**：增加单机资源（CPU、内存、磁盘）

```
2C4G → 4C8G → 8C16G → 16C32G
```

**优点**：
- 简单，不需要修改代码
- 无需考虑分布式问题

**缺点**：
- 有上限（单机最多 128C 512G）
- 成本高（性能提升不是线性的）
- 停机升级

**适用场景**：
- 数据库（MySQL 单机性能更好）
- 缓存（Redis 单机性能更好）
- 初期业务（用户 < 10 万）

### 2.2 水平扩展（Scale Out）

**定义**：增加机器数量

```
1 台 → 2 台 → 4 台 → 8 台 → ...
```

**优点**：
- 无上限（理论上可以无限扩展）
- 成本低（使用普通机器）
- 无需停机

**缺点**：
- 复杂，需要考虑分布式问题
- 数据一致性
- 负载均衡

**适用场景**：
- 应用服务器（无状态）
- 分布式存储（HDFS、Cassandra）
- 大规模业务（用户 > 100 万）

### 2.3 对比表

| 维度 | 垂直扩展 | 水平扩展 |
|------|---------|---------|
| 扩展方式 | 增加单机资源 | 增加机器数量 |
| 上限 | 有上限 | 无上限 |
| 成本 | 高 | 低 |
| 复杂度 | 低 | 高 |
| 停机 | 需要 | 不需要 |
| 适用场景 | 数据库、缓存 | 应用服务器 |

## 三、无状态设计

### 3.1 什么是无状态？

**有状态**：服务器保存用户会话信息

```
用户 A → 服务器 1（保存 Session）
用户 A → 服务器 2（找不到 Session）✗
```

**无状态**：服务器不保存用户会话信息

```
用户 A → 服务器 1（从 Redis 读取 Session）✓
用户 A → 服务器 2（从 Redis 读取 Session）✓
```

### 3.2 无状态设计原则

**原则 1：Session 外部化**

```java
// 有状态（不推荐）
HttpSession session = request.getSession();
session.setAttribute("user", user);

// 无状态（推荐）
String token = JWT.create(user);
redisTemplate.opsForValue().set("token:" + token, user, 30, TimeUnit.MINUTES);
```

**原则 2：避免本地缓存**

```java
// 有状态（不推荐）
private static Map<String, Activity> cache = new ConcurrentHashMap<>();

// 无状态（推荐）
Activity activity = redisTemplate.opsForValue().get("activity:" + id);
```

**原则 3：避免本地文件**

```java
// 有状态（不推荐）
File file = new File("/tmp/upload/" + filename);

// 无状态（推荐）
String url = ossClient.upload(file);  // 上传到 OSS
```

## 四、分片策略

### 4.1 数据库分片

**垂直分片（按业务）**

```
原始（单库）：
seckill_db
  ├── user
  ├── activity
  ├── order
  └── payment

分片后（多库）：
user_db          activity_db      order_db         payment_db
  └── user         └── activity     └── order        └── payment
```

**水平分片（按数据）**

```
原始（单表）：
order（1 亿条）

分片后（分库分表）：
db_0                db_1                db_2                db_3
  ├── order_0         ├── order_0         ├── order_0         ├── order_0
  ├── order_1         ├── order_1         ├── order_1         ├── order_1
  ├── order_2         ├── order_2         ├── order_2         ├── order_2
  └── order_3         └── order_3         └── order_3         └── order_3

总计：4 个库 × 4 张表 = 16 张表
每张表：1 亿 / 16 = 625 万条
```

**分片键选择**：

| 分片键 | 优点 | 缺点 | 适用场景 |
|--------|------|------|---------|
| 用户 ID | 数据均匀、查询简单 | 跨用户查询困难 | 用户维度查询 |
| 时间 | 冷热分离、易于归档 | 热点数据集中 | 时序数据 |
| Hash | 数据均匀 | 扩容困难 | 通用场景 |

**分片路由算法**：

```java
// 按用户 ID Hash 分片
public String route(Long userId) {
    int dbIndex = (int) (userId % 4);      // 4 个库
    int tableIndex = (int) (userId % 4);   // 4 张表
    return "db_" + dbIndex + ".order_" + tableIndex;
}

// 例如：userId = 10001
// dbIndex = 10001 % 4 = 1 → db_1
// tableIndex = 10001 % 4 = 1 → order_1
// 最终路由到：db_1.order_1
```

### 4.2 缓存分片

**Redis Cluster 分片**

```
┌─────────────────────────────────────┐
│         Redis Cluster               │
├─────────────────────────────────────┤
│  Master1 (slot 0-5460)              │
│  Master2 (slot 5461-10922)          │
│  Master3 (slot 10923-16383)         │
└─────────────────────────────────────┘

分片算法：CRC16(key) % 16384
```

**一致性 Hash**

```
                    Hash 环
                 ┌─────────┐
          Node1  │         │  Node2
                 │         │
          Node4  │         │  Node3
                 └─────────┘

优点：增删节点影响小（只影响相邻节点）
缺点：数据分布不均（需要虚拟节点）
```

### 4.3 消息队列分片

**Kafka Partition**

```
Topic: seckill-order (10 个 Partition)

Partition 0  ← Consumer 1
Partition 1  ← Consumer 1
Partition 2  ← Consumer 2
Partition 3  ← Consumer 2
...
Partition 9  ← Consumer 5

分片算法：hash(key) % partition_count
```

**优点**：
- 并行消费，提升吞吐量
- 水平扩展，增加 Partition 即可

## 五、异步化 / 事件驱动

### 5.1 同步 vs 异步

**同步（阻塞）**

```
用户请求 → 扣库存 → 创建订单 → 发送短信 → 返回结果
           (10ms)    (50ms)      (200ms)     (260ms)
```

**异步（非阻塞）**

```
用户请求 → 扣库存 → 发送 MQ → 返回结果
           (10ms)    (5ms)      (15ms)

MQ → 创建订单 → 发送短信（异步）
     (50ms)      (200ms)
```

**对比**：
- 同步：260ms
- 异步：15ms（提升 17 倍）

### 5.2 异步化场景

| 场景 | 同步方案 | 异步方案 | 提升 |
|------|---------|---------|------|
| 秒杀下单 | 扣库存 + 创建订单 | 扣库存 + MQ | 5 倍 |
| 用户注册 | 注册 + 发邮件 + 发短信 | 注册 + MQ | 10 倍 |
| 订单支付 | 支付 + 更新订单 + 发货 | 支付 + MQ | 3 倍 |

### 5.3 事件驱动架构

**定义**：通过事件触发业务流程

```
秒杀成功事件
  ↓
  ├→ 创建订单
  ├→ 扣减库存
  ├→ 发送短信
  └→ 推送通知
```

**实现（Kafka）**：

```java
// 1. 发布事件
@PostMapping("/seckill/{activityId}")
public Result<String> seckill(@PathVariable Long activityId, 
                               @RequestHeader("X-User-Id") Long userId) {
    // 扣库存

    // 发布事件
    SeckillSuccessEvent event = new SeckillSuccessEvent(activityId, userId);
    kafkaTemplate.send("seckill-success", event);
    
    return Result.success("下单成功");
}

// 2. 订阅事件（创建订单）
@KafkaListener(topics = "seckill-success")

    orderService.create(event.getActivityId(), event.getUserId());
}

// 3. 订阅事件（发送短信）
@KafkaListener(topics = "seckill-success")
public void onSeckillSuccess(SeckillSuccessEvent event) {
    smsService.send(event.getUserId(), "秒杀成功");
}
```

## 六、微服务拆分原则

### 6.1 为什么要拆分微服务？

**单体应用问题**：
- 代码耦合，难以维护
- 部署困难，牵一发动全身
- 扩展困难，无法独立扩展
- 技术栈单一

**微服务优势**：
- 独立开发、部署、扩展
- 技术栈多样化
- 故障隔离

### 6.2 拆分原则

**原则 1：按业务领域拆分（DDD）**

```
单体应用：
seckill-app
  ├── user
  ├── activity
  ├── order
  └── payment

微服务：
user-service      activity-service  order-service     payment-service
  └── user          └── activity      └── order         └── payment
```

**原则 2：高内聚、低耦合**

```
✓ 订单服务 → 支付服务（低耦合）
✗ 订单服务 ↔ 用户服务 ↔ 商品服务（高耦合）
```

**原则 3：单一职责**

```
✓ 订单服务：只负责订单相关业务
✗ 订单服务：订单 + 支付 + 物流（职责过多）
```

**原则 4：数据独立**

```
✓ 每个服务有独立的数据库
✗ 多个服务共享一个数据库
```

### 6.3 秒杀系统微服务拆分

```
┌──────────────┐
│   Gateway    │ ← 统一入口
└──────┬───────┘
       │
       ├─────────────┬─────────────┬─────────────┐
       ↓             ↓             ↓             ↓
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ Seckill  │  │  Order   │  │ Payment  │  │   User   │
│ Service  │  │ Service  │  │ Service  │  │ Service  │
└────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
     │             │             │             │
     ↓             ↓             ↓             ↓
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│Seckill  │   │ Order   │   │Payment  │   │  User   │
│   DB    │   │   DB    │   │   DB    │   │   DB    │
└─────────┘   └─────────┘   └─────────┘   └─────────┘
```

**服务职责**：

| 服务 | 职责 | 数据库 |
|------|------|--------|
| Seckill Service | 秒杀活动管理、库存扣减 | seckill_db |
| Order Service | 订单创建、查询 | order_db |
| Payment Service | 支付处理、回调 | payment_db |
| User Service | 用户管理、鉴权 | user_db |

## 七、实战案例：秒杀系统扩展性设计

### 7.1 扩展路线图

```
阶段 1：单机（QPS 6711）
  1 台应用服务器 + 1 台 MySQL + 1 台 Redis

阶段 2：水平扩展（QPS 15000）
  3 台应用服务器 + 1 主 2 从 MySQL + Redis Cluster

阶段 3：分库分表（QPS 30000）
  10 台应用服务器 + 4 个 MySQL 分片 + Redis Cluster

阶段 4：微服务（QPS 100000+）
  微服务集群 + 分库分表 + Redis Cluster + Kafka Cluster
```

### 7.2 扩展方案

**应用服务器扩展**

```java
// 无状态设计，支持水平扩展
@RestController
public class SeckillController {
    // 所有数据从 Redis / MySQL 读取，不保存在本地
}

// 负载均衡
upstream seckill {
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
    server 192.168.1.12:8080;
}
```

**数据库扩展**

```sql
-- 分库分表（按用户 ID Hash）
db_0.order_0  ← userId % 4 = 0
db_1.order_1  ← userId % 4 = 1
db_2.order_2  ← userId % 4 = 2
db_3.order_3  ← userId % 4 = 3
```

**缓存扩展**

```
Redis Cluster（3 主 3 从）
  Master1 (slot 0-5460)     ←→  Slave1
  Master2 (slot 5461-10922) ←→  Slave2
  Master3 (slot 10923-16383)←→  Slave3
```

**消息队列扩展**

```
Kafka Cluster（3 节点，10 个 Partition）
  Partition 0-3 ← Consumer Group 1
  Partition 4-6 ← Consumer Group 2
  Partition 7-9 ← Consumer Group 3
```

## 九、扩展性设计速查表

### 9.1 扩展方式选择

```
应用服务器：水平扩展（无状态）
数据库：垂直扩展 → 主从复制 → 分库分表
缓存：垂直扩展 → Redis Cluster
消息队列：Kafka Cluster（增加 Partition）
```

### 9.2 无状态设计 Checklist

```
□ Session 外部化（Redis / JWT）
□ 避免本地缓存（使用 Redis）
□ 避免本地文件（使用 OSS）
□ 避免单例模式（使用 Spring Bean）
```

### 9.3 分片策略 Checklist

```
□ 选择分片键（用户 ID / 时间 / Hash）
□ 确定分片数量（库数 × 表数）
□ 设计路由算法（Hash / Range）
□ 考虑扩容方案（翻倍扩容）
□ 处理跨片查询（避免或异步）
```

## 十、总结

扩展性设计是系统长期发展的基础，需要提前规划。

**核心要点**：
1. **水平扩展 vs 垂直扩展**：应用服务器水平扩展，数据库垂直扩展 + 分库分表
2. **无状态设计**：Session 外部化、避免本地缓存、避免本地文件
3. **分片策略**：数据库分片、缓存分片、消息队列分片
4. **异步化**：同步改异步，提升性能 5-10 倍
5. **微服务拆分**：按业务领域拆分，高内聚低耦合
