# 需求分析方法论

> 系统设计第一步：把模糊需求变成清晰的技术方案

## 一、为什么需求分析是系统设计的第一步？

在系统设计面试中，面试官通常会给出一个模糊的需求：

- "设计一个短链系统"
- "设计一个类似 Twitter 的 Feed 流"
- "设计一个秒杀系统"

如果直接开始画架构图，90% 会失败。因为：
1. **需求不明确**：不知道要支持多少用户、多少 QPS
2. **边界不清晰**：不知道哪些功能是必须的，哪些是可选的
3. **优先级不明**：不知道先做什么，后做什么

**正确的做法**：花 5-10 分钟做需求澄清，把模糊需求变成清晰的技术指标。

## 二、功能需求 vs 非功能需求

### 2.1 功能需求（Functional Requirements）

**定义**：系统必须提供的功能，用户能直接感知到的。

**秒杀系统案例**：

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 用户浏览活动 | 查看秒杀商品详情、库存、价格 | P0（必须） |
| 用户下单 | 点击秒杀按钮，生成订单 | P0（必须） |
| 库存扣减 | 扣减库存，防止超卖 | P0（必须） |
| 订单支付 | 支付订单，完成交易 | P0（必须） |
| 订单查询 | 查询订单状态 | P1（重要） |
| 活动预热 | 提前加载活动数据到缓存 | P1（重要） |
| 用户限��� | 每个用户只能买一件 | P1（重要） |
| 活动倒计时 | 显示距离开始还有多久 | P2（可选） |
| 秒杀提醒 | 活动开始前推送通知 | P2（可选） |

**短链系统案例**：

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 生成短链 | 输入长链接，返回短链接 | P0（必须） |
| 短链跳转 | 访问短链，重定向到长链 | P0（必须） |
| 短链统计 | 记录访问次数、来源 | P1（重要） |
| 自定义短链 | 用户指定短链后缀 | P2（可选） |
| 短链过期 | 设置短链有效期 | P2（可选） |

### 2.2 非功能需求（Non-Functional Requirements）

**定义**：系统的性能、可用性、扩展性等技术指标，用户不直接感知，但影响体验。

**常见非功能需求**：

| 类型 | 指标 | 秒杀系统案例 | 短链系统案例 |
|------|------|-------------|-------------|
| **性能** | QPS / TPS | 峰值 QPS 5000+ | 峰值 QPS 10000+ |
| **延迟** | P99 延迟 | < 200ms | < 50ms |
| **可用性** | SLA | 99.9%（年停机 8.76 小时） | 99.99%（年停机 52 分钟） |
| **一致性** | 数据一致性 | 强一致（库存不能超卖） | 最终一致（统计数据可延迟） |
| **扩展性** | 水平扩展 | 支持动态扩容 | 支持动态扩容 |
| **安全性** | 防刷 | IP 限流、验证码 | 防止恶意生成短链 |

**秒杀系统非功能需求详细拆解**：

```
性能需求：
- 峰值 QPS：5000+（100 并发实测 6711）
- P99 延迟：< 200ms（实测 28ms）
- 库存准确率：100%（不能超卖）

可用性需求：
- SLA：99.9%（年停机 < 9 小时）
- 故障恢复时间：< 5 分钟
- 数据备份：每天备份，保留 30 天

扩展性需求：
- 支持水平扩展（增加机器提升 QPS）
- 支持多活动并发（不同商品独立秒杀）
- 支持动态调整库存

安全性需求：
- 防刷：IP 限流 100 req/s
- 防重复购买：每个用户只能买一件
- 防机器人：验证码 / 滑块验证
```

## 三、需求拆解模板（SMART 原则）

SMART 原则：Specific（具体）、Measurable（可衡量）、Achievable（可实现）、Relevant（相关）、Time-bound（有时限）

### 3.1 秒杀系统需求拆解

**原始需求**："设计一个秒杀系统"

**SMART 拆解**：

```
S（Specific）：
- 支持 iPhone 15 秒杀活动
- 库存 100 件，价格 3999 元
- 活动时间 2026-04-27 20:00:00

M（Measurable）：
- 峰值 QPS：5000+
- P99 延迟：< 200ms
- 库存准确率：100%
- 成功率：100%（无崩溃）

A（Achievable）：
- 单机 QPS 6711（已实测）
- 分机部署可达 15000+
- 集群部署可达 30000+

R（Relevant）：
- 解决高并发下的库存超卖问题
- 保护数据库不被打垮
- 提升用户体验（快速响应）

T（Time-bound）：
- 开发周期：4 周（W9-W12）
- 上线时间：2026-04-27
- 压测时间：上线前 1 周
```

### 3.2 短链系统需求拆解

**原始需求**："设计一个短链系统"

**SMART 拆解**：

```
S（Specific）：
- 输入长链接（如 https://example.com/product/12345?utm_source=ad）
- 返回短链接（如 https://short.ly/abc123）
- 访问短链，重定向到长链

M（Measurable）：
- 峰值 QPS：10000+（读多写��）
- P99 延迟：< 50ms
- 短链长度：6-8 个字符
- 存储容量：支持 10 亿条短链

A（Achievable）：
- Redis 缓存热点短链（QPS 10 万+）
- MySQL 存储全量短链
- 分库分表支持 10 亿+

R（Relevant）：
- 缩短链接长度，方便分享
- 统计点击数据，分析用户行为
- 防止恶意链接，提升安全性

T（Time-bound）：
- MVP 版本：1 周（生成 + 跳转）
- 完整版本：2 周（统计 + 自定义）
```

## 四、需求优先级矩阵

使用 **MoSCoW 方法** 对需求进行优先级划分：

| 优先级 | 含义 | 秒杀系统案例 | 短链系统案例 |
|--------|------|-------------|-------------|
| **Must Have（P0）** | 必须有，否则系统无法运行 | 下单、扣库存、支付 | 生成短链、跳转 |
| **Should Have（P1）** | 应该有，重要但不紧急 | 订单查询、活动预热 | 短链统计 |
| **Could Have（P2）** | 可以有，锦上添花 | 倒计时、提醒 | 自定义短链、过期 |
| **Won't Have（P3）** | 不会有，明确不做 | 秒杀直播、弹幕 | 短链编辑、删除 |

**优先级矩阵（重要性 vs 紧急性）**：

```
重要性
  ↑
  │  P1: 活动预热      │  P0: 下单、扣库存
  │  订单查询          │  支付、防超卖
  │─────────────────────┼─────────────────────→ 紧急性
  │  P3: 秒杀直播      │  P2: 倒计时
  │  弹幕              │  提醒
  │
```

**面试中如何使用**：

```
面试官："设计一个秒杀系统"

你："好的，我先确认一下需求优先级：
1. P0（必须）：下单、扣库存、支付、防超卖
2. P1（重要）：订单查询、活动预热、用户限购
3. P2（可选）：倒计时、提醒

我们先聚焦 P0 功能，确保核心流程跑通，
P1 和 P2 功能可以后续迭代。您觉得这样可以吗？"
```

## 五、面试中的需求澄清技巧（5W1H）

### 5.1 5W1H 方法

| 问题 | 英文 | 秒杀系统案例 | 短链系统案例 |
|------|------|-------------|-------------|
| **What** | 做什么 | 秒杀 iPhone 15 | 生成短链 |
| **Who** | 谁用 | 普通用户 | 营销人员、开发者 |
| **When** | 什么时候 | 2026-04-27 20:00 | 随时 |
| **Where** | 在哪里 | 移动端 + PC 端 | 主要移动端 |
| **Why** | 为什么 | 促销、清库存 | 缩短链接、统计数据 |
| **How** | 怎么做 | 点击按钮下单 | 输入长链，返回短链 |

### 5.2 面试澄清话术模板

**模板 1：确认功能范围**

```
面试官："设计一个秒杀系统"

你："好的，我先确认一下功能范围：
1. 核心功能：用户浏览活动、下单、支付、订单查询，对吗？
2. 是否需要支持多个活动并发？
3. 是否需要支持用户限购（每人只能买一���）？
4. 是否需要支持活动预热（提前加载数据到缓存）？"
```

**模板 2：确认非功能需求**

```
你："接下来确认一下非功能需求：
1. 预计有多少用户参与？（如 10 万用户）
2. 峰值 QPS 大概多少？（如 5000 QPS）
3. 库存有多少件？（如 100 件）
4. 对延迟有什么要求？（如 P99 < 200ms）
5. 对可用性有什么要求？（如 99.9% SLA）"
```

**模板 3：确认技术约束**

```
你："最后确认一下技术约束：
1. 可以使用哪些技术栈？（如 Java + Spring Boot + Redis + MySQL）
2. 是否有预算限制？（如单机 vs 集群）
3. 是否有时间限制？（如 MVP 1 周 vs 完整版 4 周）
4. 是否有安全要求？（如防刷、防机器人）"
```

### 5.3 常见��问及应对

| 追问 | 应对 |
|------|------|
| "如果 QPS 从 5000 增长到 50000 怎么办？" | "可以通过水平扩展（增加机器）+ 分库分表 + CDN 加速来支持" |
| "如果 Redis 宕机怎么办？" | "Redis Cluster 多副本 + 主从切换，或者降级到 MySQL" |
| "如果库存超卖怎么办？" | "Redis Lua 原子扣减 + MySQL 乐观锁兜底 + 唯一索引防重复" |
| "如果用户恶意刷单怎么办？" | "IP 限流 + 用户限购 + 验证码 + 风控系统" |

## 七、实战案例：秒杀系统需求分析

### 7.1 原始需求

"设计一个秒杀系统，支持 iPhone 15 秒杀活动"

### 7.2 需求澄清（5W1H）

```
What: 秒杀 iPhone 15，库存 100 件，价格 3999 元
Who: 普通用户（预计 10 万用户参与）
When: 2026-04-27 20:00:00 开始
Where: 移动端 + PC 端
Why: 促销、清库存、提升品牌知名度
How: 用户点击秒杀按钮，生成订单，支付完成交易
```

### 7.3 功能需求拆解

| 功能 | 描述 | 优先级 | 技术方案 |
|------|------|--------|---------|
| 用户浏览活动 | 查看商品详情、库存、价格 | P0 | Redis 缓存 + MySQL |
| 用户下单 | 点击秒杀按钮，生成订单 | P0 | Redis Lua 扣库存 + Kafka 异步下单 |
| 库存扣减 | 扣减库存，防止超卖 | P0 | Redis Lua 原子操作 + MySQL 乐观锁 |
| 订单支付 | 支付订单，完成交易 | P0 | 第三方支付接口 |
| 订单查询 | 查询订单状态 | P1 | MySQL + Redis 缓存 |
| 活动预热 | 提前加载活动数据到缓存 | P1 | 定时任务 + Redis |
| 用户限购 | 每个用户只能买一件 | P1 | 布隆过滤器 + 唯一索引 |

### 7.4 非功能需求拆解

```
性能需求：
- 峰值 QPS：5000+（实测 6711）
- P99 延迟：< 200ms（实测 28ms）
- 库存准确率：100%

可用性需求：
- SLA：99.9%
- 故障恢复时间：< 5 分钟

扩展性需求：
- 支持水平扩展（单机 → 集群）
- 支持多活动并发

安全性需求：
- IP 限流：100 req/s
- 用户限购：每人一件
- 验证码：防机器人
```

### 7.5 技术选型

| 组件 | 技术 | 原因 |
|------|------|------|
| 网关 | Spring Cloud Gateway | 限流、路由 |
| 缓存 | Redis | 高性能、原子操作 |
| 消息队列 | Kafka | 异步削峰 |
| 数据库 | MySQL | 数据持久化 |
| 注册中心 | Nacos | 服务发现 |
| 监控 | Prometheus + Grafana | 实时监控 |

## 八、总结

需求分析是系统设计的第一步，也是最重要的一步。

**核心要点**：
1. **功能需求 vs 非功能需求**：明确系统要做什么，以及性能、可用性等指标
2. **SMART 原则**：把模糊需求变成具体、可衡量的技术指标
3. **优先级矩阵**：P0（必须）→ P1（重要）→ P2（可选）→ P3（不做）
4. **5W1H 方法**：What、Who、When、Where、Why、How
5. **面试话术**：先澄清需求，再画架构图

**面试中的时间分配**（45 分钟）：
- 需求澄清：5-10 分钟
- 容量估算：5 分钟
- 高层架构：10 分钟
- 深入设计：20 分钟
- 总结与追问：5 分钟

**下一步**：掌握容量估算方法，把需求转化为具体的 QPS、存储、带宽等技术指标。
