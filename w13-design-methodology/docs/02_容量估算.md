# 容量估算

> 系统设计第二步：把需求转化为具体的 QPS、存储、带宽等技术指标

## 一、为什么需要容量估算？

在系统设计面试中，容量估算是连接需求和架构的桥梁：

**需求** → **容量估算** → **架构设计**

例如：
- "支持 100 万用户" → QPS 多少？存储多少？带宽多少？ → 需要几台服务器？
- "设计一个短链系统" → 每天生成多少短链？存储多久？ → 数据库如何设计？

**容量估算的价值**：
1. **量化需求**：把模糊的"高并发"变成具体的"QPS 5000"
2. **指导架构**：知道需要几台服务器、多大的数据库
3. **成本预估**：知道需要多少预算
4. **展示思维**：面试官看重的是思考过程，不是精确答案

## 二、容量估算的核心指标

### 2.1 QPS / TPS

**QPS（Queries Per Second）**：每秒查询数
**TPS（Transactions Per Second）**：每秒事务数

**计算公式**：

```
QPS = DAU × 平均每用户请求数 / 86400 × 峰值系数

其中：
- DAU：日活跃用户数
- 86400：一天的秒数
- 峰值系数：通常取 2-5（高峰期是平均值的 2-5 倍）
```

**秒杀系统案例**：

```
已知条件：
- 预计 10 万用户参与秒杀
- 活动持续 1 小时
- 每个用���平均点击 5 次（刷新页面、点击按钮）

计算过程：
平均 QPS = 100,000 × 5 / 3600 = 139 QPS
峰值 QPS = 139 × 5 = 695 QPS（活动开始瞬间）

但实际上，秒杀开始的前 10 秒会有 80% 的请求：
瞬时 QPS = 100,000 × 5 × 0.8 / 10 = 40,000 QPS

结论：需要支持 4 万 QPS 的瞬时峰值
```

**短链系统案例**：

```
已知条件：
- 日活 1000 万用户
- 每个用户平均生成 0.1 个短链（10% 用户生成）
- 每个用户平均点击 2 个短链

计算过程：
写 QPS（生成短链）：
- 平均 QPS = 10,000,000 × 0.1 / 86400 = 12 QPS
- 峰值 QPS = 12 × 3 = 36 QPS

读 QPS（点击短链）：
- 平均 QPS = 10,000,000 × 2 / 86400 = 231 QPS
- 峰值 QPS = 231 × 3 = 693 QPS

读写比 = 693 / 36 ≈ 20:1（读多写少）

结论：需要支持 700 QPS，且读多写少，适合用缓存优化
```

### 2.2 存储容量

**计算公式**：

```
存储容量 = 单条记录大小 × 记录总数 × 副本数 × (1 + 增长率)

其中：
- 单条记录大小：根据数据库表结构估算
- 记录总数：根据业务量估算
- 副本数：通常 2-3 个副本（主从复制）
- 增长率：预留 20-30% 增长空间
```

**秒杀系统案例**：

```
订单表（seckill_order）：
- 字段：id(8B) + user_id(8B) + activity_id(8B) + order_no(20B) 
        + status(1B) + create_time(8B) + update_time(8B)
- 单条记录：约 60 字节
- 索引：约 40 字节（主键 + 唯一索引）
- 总计：100 字节/条

假设：
- 每天 10 场秒杀活动
- 每场 100 件商品
- 保存 1 年数据

计算：
记录总数 = 10 × 100 × 365 = 365,000 条
存储容量 = 100B × 365,000 × 2（副本）× 1.3（增长）
         = 95 MB

结论：订单表 1 年数据约 100 MB，非常小，单机 MySQL 足够
```

**短链系统案例**：

```
短链表（short_url）：
- 字段：id(8B) + short_code(8B) + long_url(200B) + create_time(8B)
        + expire_time(8B) + click_count(4B)
- 单条记录：约 240 字节
- 索引：约 60 字节
- 总计：300 字节/条

假设：
- 每天生成 100 万个短链
- 保存 5 年数据

计算：
记录总数 = 1,000,000 × 365 × 5 = 1,825,000,000（18 亿）
存储容量 = 300B × 1,825,000,000 × 2（副本）× 1.3（增长）
         = 1.4 TB

结论：需要 1.4 TB 存储，需要分库分表
```

### 2.3 带宽

**计算公式**：

```
带宽 = QPS × 平均响应大小 × 8（转换为 bit）

其中：
- QPS：每秒请求数
- 平均响应大小：根据接口返回数据估算
- 8：1 字节 = 8 bit
```

**秒杀系统案例**：

```
秒杀接口：
- 请求大小：200 字节（HTTP 头 + Body）
- 响应大小：500 字节（JSON 数据）
- 峰值 QPS：5000

计算：
入带宽 = 5000 × 200B × 8 = 8 Mbps
出带宽 = 5000 × 500B × 8 = 20 Mbps

结论：需要 20 Mbps 出带宽（通常出带宽是瓶颈）
```

**短链系统案例**：

```
跳转接口：
- 请求大小：150 字节
- 响应大小：300 字节（302 重定向）
- 峰值 QPS：700

计算：
入带宽 = 700 × 150B × 8 = 0.84 Mbps
出带宽 = 700 × 300B × 8 = 1.68 Mbps

结论：需要 2 Mbps 出带宽，非常小
```

### 2.4 内存

**计算公式**：

```
内存 = 热点数据大小 + JVM 堆内存 + 操作系统预留

其中：
- 热点数据：Redis 缓存的数据
- JVM 堆内存：通常 2-8 GB
- 操作系统预留：通常 2-4 GB
```

**秒杀系统案例**：

```
Redis 缓存：
- 活动信息：10 场 × 1 KB = 10 KB
- 库存信息：10 场 × 100 字节 = 1 KB
- 用户购买记录（布隆过滤器）：100 万用户 × 1.2 MB = 1.2 MB
- 总计：约 2 MB

JVM 堆内存：4 GB（-Xms4g -Xmx4g）
操作系统：2 GB

总内存 = 2 MB + 4 GB + 2 GB ≈ 6 GB

结论：单机 8 GB 内存足够
```

## 三、粗算方法（Envelope Calculation）

面试中不需要精确计算，使用**粗算**即可：

### 3.1 常用数字速查表

| 指标 | 数值 | 说明 |
|------|------|------|
| **1 台服务器** | 5000-10000 QPS | 单机极限（优化后） |
| **1 台 MySQL** | 1000-5000 QPS | 读多写少场景 |
| **1 台 Redis** | 100,000 QPS | 纯内存操作 |
| **1 台 Kafka** | 50,000 QPS | 批量写入 |
| **1 GB 内存** | 100 万条记录 | 假设每条 1 KB |
| **1 TB 硬盘** | 10 亿条记录 | 假设每条 1 KB |
| **1 Mbps 带宽** | 125 KB/s | 1 Mbps = 1024 Kbps / 8 |
| **1 天** | 86400 秒 | 约 10 万秒（粗算） |
| **1 年** | 365 天 | 约 400 天（粗算） |

### 3.2 粗算技巧

**技巧 1：2 的幂次方**

```
1 KB = 2^10 字节 ≈ 1000 字节
1 MB = 2^20 字节 ≈ 100 万字节
1 GB = 2^30 字节 ≈ 10 亿字节
1 TB = 2^40 字节 ≈ 1 万亿字节
```

**技巧 2：时间换算**

```
1 天 = 86400 秒 ≈ 10^5 秒（粗算）
1 年 = 365 天 ≈ 400 天（粗算）
```

**技巧 3：百分比估算**

```
80/20 法则：80% 的请求集中在 20% 的时间
峰值系数：通常取 2-5 倍
增长率：通常预留 20-30%
```

### 3.3 粗算案例：Twitter Feed 流

**需求**：设计一个类似 Twitter 的 Feed 流系统

**粗算过程**：

```
1. 用户规模：
   - 日活用户：1 亿（10^8）
   - 每个用户关注 200 人
   - 每个用户每天发 1 条推文

2. QPS 估算：
   - 发推文 QPS = 10^8 × 1 / 10^5 = 1000 QPS
   - 读 Feed QPS = 10^8 × 10（每天刷 10 次）/ 10^5 = 10,000 QPS
   - 读写比 = 10:1

3. 存储估算：
   - 单条推文：200 字节（文本 + 元数据）
   - 每天新增：10^8 × 1 × 200B = 20 GB
   - 保存 5 年：20 GB × 400 × 5 = 40 TB

4. 带宽估算：
   - 读 Feed：10,000 QPS × 10 KB（10 条推文）× 8 = 800 Mbps
   - 需要 1 Gbps 带宽

5. 服务器估算：
   - 应用服务器：10,000 QPS / 5000 = 2 台（考虑冗余，实际 4 台）
   - 数据库：40 TB / 1 TB = 40 台（分库分表）
   - 缓存：热点数据 10% = 4 TB，需要 Redis Cluster
```

## 四、实战案例：秒杀系统容量估算

### 4.1 需求回顾

```
- 预计 10 万用户参与
- 库存 100 件
- 活动持续 1 小时
- 每个用户平均点击 5 次
```

### 4.2 QPS 估算

```
场景 1：均匀分布（理想情况）
平均 QPS = 100,000 × 5 / 3600 = 139 QPS

场景 2：前 10 秒集中（实际情况）
瞬时 QPS = 100,000 × 5 × 0.8 / 10 = 40,000 QPS

场景 3：考虑失败重试
实际 QPS = 40,000 × 1.5（重试系数）= 60,000 QPS

结论：需要支持 6 万 QPS 的瞬时峰值
```

### 4.3 存储估算

```
订单表：
- 单条记录：100 字节
- 成功订单：100 条（库存）
- 失败订单：0（不存储）
- 总存储：100 × 100B = 10 KB

活动表：
- 单条记录：500 字节
- 活动数量：10 场/天 × 365 天 = 3650 场
- 总存储：3650 × 500B = 1.8 MB

结论：存储需求极小，单机 MySQL 足够
```

### 4.4 带宽估算

```
秒杀接口：
- 请求大小：200 字节
- 响应大小：500 字节
- 峰值 QPS：60,000

入带宽 = 60,000 × 200B × 8 = 96 Mbps
出带宽 = 60,000 × 500B × 8 = 240 Mbps

结论：需要 250 Mbps 出带宽（约 30 MB/s）
```

### 4.5 内存估算

```
Redis 缓存：
- 活动信息：10 场 × 1 KB = 10 KB
- 库存信息：10 场 × 100 字节 = 1 KB
- 布隆过滤器：100 万用户 × 1.2 MB = 1.2 MB
- 总计：约 2 MB

JVM 堆内存：8 GB（-Xms8g -Xmx8g）
操作系统：2 GB

总内存 = 2 MB + 8 GB + 2 GB ≈ 10 GB

结论：单机 16 GB 内存足够
```

### 4.6 服务器估算

```
应用服务器：
- 单机 QPS：6711（实测）
- 需要台数：60,000 / 6711 = 9 台
- 考虑冗余：9 × 1.5 = 14 台

数据库：
- 存储需求：10 KB（极小）
- 单机 MySQL 足够

缓存：
- 存储需求：2 MB（极小）
- 单机 Redis 足够

消息队列：
- 峰值 TPS：100 条/秒（只有成功的订单进 Kafka）
- 单机 Kafka 足够

结论：需要 14 台应用服务器 + 1 台 MySQL + 1 台 Redis + 1 台 Kafka
```

## 五、实战案例：短链系统容量估算

### 5.1 需求回顾

```
- 日活 1000 万用户
- 每个用户平均生成 0.1 个短链
- 每个用户平均点击 2 个短链
- 保存 5 年数据
```

### 5.2 QPS 估算

```
写 QPS（生成短链）：
- 平均 QPS = 10,000,000 × 0.1 / 86400 = 12 QPS
- 峰值 QPS = 12 × 3 = 36 QPS

读 QPS（点击短链）：
- 平均 QPS = 10,000,000 × 2 / 86400 = 231 QPS
- 峰值 QPS = 231 × 3 = 693 QPS

读写比 = 693 / 36 ≈ 20:1

结论：读多写少，适合用缓存优化
```

### 5.3 存储估算

```
短链表：
- 单条记录：300 字节
- 每天新增：1,000,000 条
- 保存 5 年：1,000,000 × 365 × 5 = 1,825,000,000（18 亿）
- 存储容量：300B × 1,825,000,000 × 2（副本）× 1.3（增长）= 1.4 TB

分库分表策略：
- 单表最大：1000 万条（MySQL 推荐）
- 需要分表：1,825,000,000 / 10,000,000 = 183 张表
- 分库：183 / 16 = 12 个库（每库 16 张表）

结论：需要 12 个 MySQL 实例，每个 120 GB
```

### 5.4 带宽估算

```
跳转接口：
- 请求大小：150 字节
- 响应大小：300 字节（302 重定向）
- 峰值 QPS：700

入带宽 = 700 × 150B × 8 = 0.84 Mbps
出带宽 = 700 × 300B × 8 = 1.68 Mbps

结论：需要 2 Mbps 出带宽，非常小
```

### 5.5 缓存估算

```
热点短链（20% 的短链占 80% 的访问）：
- 热点数量：1,825,000,000 × 0.2 = 365,000,000（3.6 亿）
- 单条缓存：300 字节
- 总缓存：365,000,000 × 300B = 110 GB

Redis Cluster：
- 单机 Redis：16 GB
- 需要节点：110 GB / 16 GB = 7 个节点
- 考虑副本：7 × 2 = 14 个节点

结论：需要 Redis Cluster（7 主 7 从）
```

### 5.6 服务器估算

```
应用服务器：
- 单机 QPS：5000（假设）
- 需要台数：700 / 5000 = 1 台
- 考虑冗余：1 × 2 = 2 台

数据库：
- 需要 12 个 MySQL 实例

缓存：
- 需要 14 个 Redis 节点（7 主 7 从）

结论：2 台应用服务器 + 12 台 MySQL + 14 台 Redis = 28 台服务器
```

## 六、容量估算面试话术

### 6.1 标准流程

```
1. 澄清需求（5W1H）
   "我先确认一下需求：日活多少？每个用户多少请求？"

2. 估算 QPS
   "假设日活 1000 万，每人 10 次请求，平均 QPS = 1000 万 × 10 / 10 万秒 = 1000 QPS"
   "考虑峰值系数 3 倍，峰值 QPS = 3000"

3. 估算存储
   "假设单条记录 1 KB，每天 100 万条，保存 1 年 = 1 KB × 100 万 × 365 = 365 GB"

4. 估算带宽
   "假设响应 10 KB，QPS 3000，带宽 = 3000 × 10 KB × 8 = 240 Mbps"

5. 估算服务器
   "假设单机 5000 QPS，需要 3000 / 5000 = 1 台，考虑冗余 2 台"
```

### 6.2 常见追问

| 追问 | 应对 |
|------|------|
| "如果 QPS 增长 10 倍怎么办？" | "可以水平扩展，增加服务器数量" |
| "如果存储增长到 10 TB 怎么办？" | "可以分库分表，或者使用 NoSQL" |
| "如果带宽不够怎么办？" | "可以用 CDN 加速，或者压缩响应数据" |
| "你的估算准确吗？" | "这是粗算，实际需要压测验证，但数量级是对的" |

## 七、容量估算速查表

### 7.1 QPS 速查表

| 场景 | QPS | 说明 |
|------|-----|------|
| 小型网站 | < 100 | 单机足够 |
| 中型网站 | 100-1000 | 2-3 台服务器 |
| 大型网站 | 1000-10000 | 10+ 台服务器 |
| 超大型网站 | > 10000 | 集群 + 分布式 |

### 7.2 存储速查表

| 数据量 | 存储 | 方案 |
|--------|------|------|
| < 1 GB | 单表 | 单机 MySQL |
| 1 GB - 100 GB | 分表 | 单机 MySQL + 分表 |
| 100 GB - 1 TB | 分库分表 | 多台 MySQL |
| > 1 TB | NoSQL | MongoDB / Cassandra |

### 7.3 带宽速查表

| 带宽 | 速度 | 场景 |
|------|------|------|
| 1 Mbps | 125 KB/s | 小型网站 |
| 10 Mbps | 1.25 MB/s | 中型网站 |
| 100 Mbps | 12.5 MB/s | 大型网站 |
| 1 Gbps | 125 MB/s | 超大型网站 |

## 八、总结

容量估算是系统设计的核心技能，面试中不需要精确计算，但要展示清晰的思维过程。

**核心要点**：
1. **QPS 估算**：DAU × 请求数 / 86400 × 峰值系数
2. **存储估算**：单条记录 × 记录数 × 副本数 × 增长率
3. **带宽估算**：QPS × 响应大小 × 8
4. **粗算技巧**：使用 2 的幂次方、时间换算、百分比估算
5. **常用数字**：1 台服务器 5000 QPS、1 GB 存 100 万条记录

**面试话术**：
- "我先粗算一下 QPS..."
- "假设单条记录 1 KB..."
- "考虑峰值系数 3 倍..."
- "这是粗算，实际需要压测验证"

**下一步**：掌握数据模型设计，把容量估算转化为具体的数据库表结构。
