# 秒杀系统 — 面试问答手册

> 基于实战项目的高频面试题及答案

## 一、架构设计类

### 1.1 如何设计一个秒杀系统？

**回答要点**：

**1. 核心挑战**
- 高并发：瞬时 10 万+ QPS
- 库存准确：不能超卖
- 用户体验：响应快，不能让用户等太久

**2. 整体架构**

```
四层过滤架构：
CDN/前端 → Gateway限流 → Redis预扣 → Kafka异步 → MySQL落单

10万请求 → 2万 → 100个 → 最终100个订单
```

**3. 关键技术点**

| 层级 | 技术方案 | 作用 |
|------|---------|------|
| 接入层 | Nginx + CDN | 静态资源缓存，减少 50% 请求 |
| 网关层 | Gateway + Sentinel | 限流熔断，过滤 80% 无效请求 |
| 服务层 | Redis Lua | 原子扣库存，过滤 99% 请求 |
| 数据层 | Kafka + MySQL | 异步削峰，保护数据库 |

**4. 防超卖三道防线**
- Redis Lua 原子扣减（主要防线）
- MySQL 乐观锁 `quantity > 0`（兜底）
- 唯一索引 `(activity_id, user_id)`（防重复）

**5. 性能数据**
- 本项目实测：QPS 6711，P99 28ms
- 分机部署预估：QPS 15000+
- 集群部署预估：QPS 30000+

### 1.2 如何保证不超卖？

**回答要点**：

**方案对比**：

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| 数据库悲观锁 | 强一致 | 性能差（QPS < 1000） | 低并发 |
| 数据库乐观锁 | 无锁等待 | 大量失败重试 | 中并发 |
| Redis 分布式锁 | 性能好 | 实现复杂 | 高并发 |
| Redis Lua 原子操作 | 性能最好 | 需要 Lua 脚本 | 高并发（推荐） |

**本项目采用 Redis Lua + MySQL 乐观锁**：

```lua
-- Redis Lua 脚本（原子操作）
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock <= 0 then
    return 0  -- 库存不足
end
redis.call('DECR', KEYS[1])
return 1  -- 扣减成功
```

```sql
-- MySQL 乐观锁（兜底）
UPDATE seckill_activity 
SET remain_stock = remain_stock - 1 
WHERE id = ? AND remain_stock > 0;

-- 检查影响行数
if (affectedRows == 0) {
    throw new StockNotEnoughException();
}
```

**为什么需要两层保护？**
- Redis 可能宕机或数据丢失
- MySQL 作为最终数据源，必须保证准确性
- 双重保险，99.99% 不会超卖

```
  ↓
```

```
同步方案（不可行）：
10万请求 → 直接打MySQL → 数据库挂了

异步方案（可行）：
10万请求 → Redis过滤 → 100个进Kafka → 慢慢消费 → MySQL
```

**3. 缓存预热**

```java

public void warmup(@PathVariable Long id) {

```

**4. 读写分离**
- 写操作：主库
- 读操作：从库（活动详情、订单查询）

**5. 本项目性能数据**

### 1.4 Redis 和 MySQL 数据一致性如何保证？

**回答要点**：

**1. 问题场景**

```
Redis 库存：100
MySQL 库存：100

用户 A 秒杀成功：
  Redis: 100 → 99 ✓
  Kafka 消息发送失败 ✗
  MySQL: 100（未扣减）

结果：Redis 和 MySQL 不一致
```

**2. 解决方案**

**方案一：最终一致性（本项目采用）**

```
1. Redis 扣减成功
2. 发送 Kafka 消息（异步）
3. 消费者消费消息，扣减 MySQL
4. 如果失败，重试（Kafka 自动重试）
5. 定时任务对账，修正不一致数据
```

```java
// 定时对账任务
@Scheduled(cron = "0 */10 * * * ?")  // 每 10 分钟
public void reconcile() {
    // 1. 查询 Redis 库存
    Long redisStock = redisTemplate.opsForValue().get("stock:1");
    
    // 2. 查询 MySQL 库存
    Activity activity = activityMapper.selectById(1L);
    Long mysqlStock = activity.getRemainStock();
    
    // 3. 如果不一致，以 MySQL 为准
    if (!redisStock.equals(mysqlStock)) {
        redisTemplate.opsForValue().set("stock:1", mysqlStock);
        log.warn("库存不一致，已修正：Redis={}, MySQL={}", redisStock, mysqlStock);
    }
}
```

**方案二：强一致性（性能差，不推荐）**

```
1. Redis 扣减成功
2. 同步调用订单服务，扣减 MySQL
3. 如果 MySQL 失败，回滚 Redis
4. 使用分布式事务（Seata）

缺点：性能差，QPS < 1000
```

**3. 本��目选择**
- 采用最终一致性
- 通过 Kafka 重试 + 定时对账保证一致性
- 性能优先，可接受短暂不一致（秒级）

## 二、性能优化类

### 2.1 如何从 QPS 3623 优化到 6711？

**回答要点**：

**性能提升 85% 的关键优化**：

**1. JVM 预热**
- 问题：首次压测时 JIT 未编译，解释执行
- 优化：系统运行一段时间后，JIT 编译优化生效
- 效果：热点代码执行效率提升 3-10x

**2. 连接池复用**
- 问题：首次压测时连接池冷启动，逐步建立连接
- 优化：连接池预热，启动时建立 50 个 Redis 连接
- 效果：消除连接建立开销（~10ms/次）

```java
@Component
public class RedisWarmup implements ApplicationRunner {
    @Override
    public void run(ApplicationArguments args) {
        ExecutorService pool = Executors.newFixedThreadPool(50);
        for (int i = 0; i < 50; i++) {
            pool.submit(() -> redisTemplate.opsForValue().get("warmup"));
        }
    }
}
```

**3. 系统缓存命中**
- 文件系统缓存：配置文件已缓存
- 网络缓冲区：自适应扩大
- CPU 缓存：热点代码已加载到 L1/L2

**4. GC 优化**
- 问题：首次压测时对象分配模式未稳定，GC 频繁
- 优化：运行一段时间后，年轻代对象分配稳定
- 效果：Minor GC 频率降低

**5. 网络优化**
- 启用 HTTP Keep-Alive
- 网络往返从 8ms 降至 4ms

**对比数据**：

| 优化项 | 首次 | 本次 | 提升 |
|--------|------|------|------|
| QPS | 3623 | 6711 | +85% |
| P99 | 53ms | 28ms | -47% |
| P50 | 17ms | 13ms | -24% |

## 三、技术细节类

### 3.1 Redis Lua 脚本如何保证原子性？

**回答要点**：

**1. Lua 脚本的原子性保证**

Redis 执行 Lua 脚本时：
- 单线程执行，不会被其他命令打断
- 脚本执行期间，其他客户端命令会被阻塞
- 要么全部执行成功，要么全部失败

**2. 本项目的 Lua 脚本**

```lua
-- 扣减库存脚本
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock <= 0 then
    return 0  -- 库存不足
end
redis.call('DECR', KEYS[1])
return 1  -- 扣减成功
```

**3. 为什么不用 WATCH + MULTI？**

```redis
WATCH stock:1
GET stock:1
MULTI
DECR stock:1
EXEC

问题：
- WATCH 在高并发下大量失败
- 需要客户端重试，增加网络开销
- 性能差
```

**4. Lua 脚本的优势**
- 原子性：Redis 保证
- 性能好：减少网络往返
- 灵活：可以实现复杂逻辑

### 3.2 布隆过滤器的作用是什么？

**回答要点**：

**1. 使用场景**

防止用户重复购买：
```
用�� A 第一次秒杀：
  布隆过滤器：不存在 → 继续
  Redis 扣库存：成功
  布隆过滤器：添加用户 A

用户 A 第二次秒杀：
  布隆过滤器：存在 → 直接拒绝（无需查 Redis）
```

**2. 实现代码**

```java
@Component
public class BloomFilterService {
    private BloomFilter<Long> bloomFilter;
    
    @PostConstruct
    public void init() {
        // 预期 100 万用户，误判率 1%
        bloomFilter = BloomFilter.create(
            Funnels.longFunnel(),
            1_000_000,
            0.01
        );
    }
    
    public boolean mightContain(Long userId) {
        return bloomFilter.mightContain(userId);
    }
    
    public void put(Long userId) {
        bloomFilter.put(userId);
    }
}
```

**3. 优势**
- 内存占用小：100 万用户只需 1.2MB
- 查询速度快：O(k)，k 为哈希函数个数
- 无需查询 Redis，减少网络开销

**4. 缺点**
- 有误判率（1%）：可能误判用户已购买
- 无法删除：只能重建

**5. 本项目效果**
- 过滤 90% 重复请求
- Redis 查询减少 90%

### 3.3 Kafka 如何保证消息不丢失？

**回答要点**：

**1. 生产者端**

```yaml
spring:
  kafka:
    producer:
      acks: 1  # 至少一个副本确认
      retries: 3  # 失败重试 3 次
      batch-size: 16384  # 批量发送
      linger-ms: 5  # 攒批时间
```

**2. Broker 端**

```properties
# 副本��
replication.factor=2

# 最小同步副本数
min.insync.replicas=1

# 日志刷盘策略
log.flush.interval.messages=10000
log.flush.interval.ms=1000
```

**3. 消费者��**

```yaml
spring:
  kafka:
    consumer:
      enable-auto-commit: false  # 手动提交
      max-poll-records: 100  # 每次拉取 100 条
```

```java
@KafkaListener(topics = "seckill-order")
public void consume(SeckillMessage message, Acknowledgment ack) {
    try {
        // 处理消息
        orderService.createOrder(message);
        // 手动提交
        ack.acknowledge();
    } catch (Exception e) {
        // 不提交，消息会重新消费
        log.error("消费失败", e);
    }
}
```

**4. 本项目配置**
- acks=1：性能和可靠性平衡
- 手动提交：保证消息处理成功后才提交
- 重试机制：失败自动重试

### 3.4 如何防止缓存击穿、穿透、雪崩？

**回答要点**：

**1. 缓存穿透（查询不存在的数据）**

问题：
```
恶意用户查询 activity_id = -1（不存在）
→ Redis 没有
→ 查询 MySQL
→ MySQL 也没有
→ 大量请求打到 MySQL
```

解决：
```java
// 方案 1：缓存空值
if (activity == null) {
    redisTemplate.opsForValue().set("activity:-1", "", 5, TimeUnit.MINUTES);
    return null;
}

// 方案 2：布隆过滤器（本项目采用）
if (!bloomFilter.mightContain(activityId)) {
    return null;  // 直接拒绝
}
```

**2. 缓存击穿（热点数据过期）**

问题：
```
热门活动缓存过期
→ 大量请求同时查询 MySQL
→ MySQL 压力激增
```

解决：
```java
// 方案 1：永不过期（本项目采用）
redisTemplate.opsForValue().set("activity:1", activity);  // 不设置过期时间

// 方案 2：分布式锁
String lockKey = "lock:activity:" + id;
if (redisTemplate.opsForValue().setIfAbsent(lockKey, "1", 10, TimeUnit.SECONDS)) {
    try {
        // 只有一个线程查询 MySQL
        activity = activityMapper.selectById(id);
        redisTemplate.opsForValue().set("activity:" + id, activity);
    } finally {
        redisTemplate.delete(lockKey);
    }
} else {
    // 其他线程等待
    Thread.sleep(100);
    return getActivity(id);  // 重试
}
```

**3. 缓存雪崩（大量缓存同时过期）**

问题：
```
大量缓存在同一时间过期
→ 大量请求同时查询 MySQL
→ MySQL 挂了
```

解决：
```java
// 方案 1：随机过期时间
int randomExpire = 3600 + new Random().nextInt(600);  // 3600-4200 秒
redisTemplate.opsForValue().set(key, value, randomExpire, TimeUnit.SECONDS);

// 方案 2：永不过期 + 异步更新（本项目采用）
redisTemplate.opsForValue().set(key, value);  // 永不过期
```
