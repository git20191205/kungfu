## 二、短期优化（1 周内）

### 2.1 JVM 参数调优

**目标**：减少 GC 停顿，提升 QPS 至 8000

#### 当前配置问题

```bash
# 当前使用默认 JVM 参数
java -jar app.jar

问题：
- 堆内存大小不固定，频繁扩容
- 使用 Parallel GC，停顿时间长
- 未启用 GC 日志，无法分析
```

#### 优化配置

**方案 1：G1GC（推荐）**

```bash
java -Xms512m -Xmx512m \
  -XX:+UseG1GC \
  -XX:MaxGCPauseMillis=50 \
  -XX:G1HeapRegionSize=4m \
  -XX:InitiatingHeapOccupancyPercent=45 \
  -XX:+ParallelRefProcEnabled \
  -XX:+AlwaysPreTouch \
  -Xlog:gc*:file=logs/gc.log:time,uptime:filecount=5,filesize=10M \
  -jar app.jar
```

**参数说明：**
- `-Xms512m -Xmx512m`：固定堆大小，避免扩容
- `-XX:+UseG1GC`：使用 G1 垃圾回收器
- `-XX:MaxGCPauseMillis=50`：目标停顿时间 50ms
- `-XX:+AlwaysPreTouch`：启动时预分配内存
- `-Xlog:gc*`：启用 GC 日志

**方案 2：ZGC（Java 11+）**

```bash
java -Xms512m -Xmx512m \
  -XX:+UseZGC \
  -XX:ZCollectionInterval=5 \
  -XX:+AlwaysPreTouch \
  -Xlog:gc*:file=logs/gc.log:time,uptime:filecount=5,filesize=10M \
  -jar app.jar
```

**预期效果：**
- GC 停顿从 100ms 降至 10ms
- QPS 提升 10-20%
- P99 延迟降低 20%

### 2.2 连接池预热

**目标**：消除冷启动延迟

#### Redis 连接池预热

```java
@Component
public class RedisWarmup implements ApplicationRunner {
    
    @Autowired
    private StringRedisTemplate redisTemplate;
    
    @Override
    public void run(ApplicationArguments args) {
        log.info("开始预热 Redis 连接池...");
        
        // 预热 50 个连接
        ExecutorService pool = Executors.newFixedThreadPool(50);
        CountDownLatch latch = new CountDownLatch(50);
        
        for (int i = 0; i < 50; i++) {
            pool.submit(() -> {
                try {
                    redisTemplate.opsForValue().get("warmup");
                } finally {
                    latch.countDown();
                }
            });
        }
        
        try {
            latch.await(5, TimeUnit.SECONDS);
            log.info("Redis 连接池预热完成");
        } catch (InterruptedException e) {
            log.error("Redis 连接池预热超时", e);
        } finally {
            pool.shutdown();
        }
    }
}
```

#### Kafka 生产者预热

```java
@Component
public class KafkaWarmup implements ApplicationRunner {
    
    @Autowired
    private KafkaTemplate<String, String> kafkaTemplate;
    
    @Override
    public void run(ApplicationArguments args) {
        log.info("开始预热 Kafka 连接...");
        
        // 发送一条预热消息
        kafkaTemplate.send("warmup-topic", "warmup")
            .addCallback(
                success -> log.info("Kafka 连接预热完成"),
                failure -> log.error("Kafka 连接预热失败", failure)
            );
    }
}
```

**预期效果：**
- 首次请求延迟从 100ms 降至 20ms
- 50 并发 QPS 提升 50%

        ┌────────────┼────────────┬────────────┐

   └────┬───┘   └───┬────┘  └───┬────┘  └───┬────┘
        │           │           │           │

#### 机器配置建议

| 角色 | CPU | 内存 | 磁盘 | 数量 |
|------|-----|------|------|------|
| Gateway | 2C | 4G | 50G | 1 |
| Seckill | 4C | 8G | 50G | 1 |
| Order | 2C | 4G | 50G | 1 |
| Payment | 2C | 4G | 50G | 1 |
| Redis | 4C | 8G | 100G | 1 |
| MySQL | 4C | 16G | 200G | 1 |
| Kafka | 4C | 8G | 200G | 1 |
| Nacos | 2C | 4G | 50G | 1 |

**总计**：24C + 56G 内存

#### Nginx 配置

```nginx
# /etc/nginx/conf.d/seckill.conf
upstream gateway {
    server 192.168.1.11:8090 weight=1 max_fails=3 fail_timeout=30s;
}

server {
    listen 80;
    server_name seckill.example.com;
    
    # 限流配置
    limit_req_zone $binary_remote_addr zone=seckill:10m rate=100r/s;
    limit_req zone=seckill burst=200 nodelay;
    
    location /api/seckill {
        proxy_pass http://gateway;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        
        # 超时配置
        proxy_connect_timeout 5s;
        proxy_send_timeout 10s;
        proxy_read_timeout 10s;
    }
}
```

**预期效果：**
- QPS 提升至 15000+
- P99 延迟降至 20ms
- 资源利用率提升至 80%

## 四、长期优化（3 个月内）

### 4.1 容器化集群部署

**目标**：弹性伸缩，QPS 30000+

#### Kubernetes 部署

```yaml
# seckill-service-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: seckill-service
spec:
  replicas: 3  # 3 个副本
  selector:
    matchLabels:
      app: seckill-service
  template:
    metadata:
      labels:
        app: seckill-service
    spec:
      containers:
      - name: seckill-service
        image: seckill-service:latest
        ports:
        - containerPort: 8084
        resources:
          requests:
            cpu: "2"
            memory: "4Gi"
          limits:
            cpu: "4"
            memory: "8Gi"
        env:
        - name: JAVA_OPTS
          value: "-Xms4g -Xmx4g -XX:+UseG1GC"
---
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: seckill-service-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: seckill-service
  minReplicas: 3
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

**预期效果：**
- 自动扩缩容，应对流量波动
- 3 节点 QPS 30000+
- 高可用，单节点故障无影响

### 4.2 数据库读写分离

**目标**：提升数据库吞吐量

#### 主从架构

```
MySQL Master (写)
    ↓ 同步
MySQL Slave 1 (读)
MySQL Slave 2 (读)
```

#### ShardingSphere 配置

```yaml
# shardingsphere.yml
dataSources:
  master:
    url: jdbc:mysql://192.168.1.60:3306/seckill
    username: root
    password: password
  slave1:
    url: jdbc:mysql://192.168.1.61:3306/seckill
    username: root
    password: password
  slave2:
    url: jdbc:mysql://192.168.1.62:3306/seckill
    username: root
    password: password

rules:
  - !READWRITE_SPLITTING
    dataSources:
      seckill_ds:
        writeDataSourceName: master
        readDataSourceNames:
          - slave1
          - slave2
        loadBalancerName: round_robin
```

**预期效果：**
- 读 QPS 提升 3 倍
- 写性能不受影响
