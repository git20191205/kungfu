# 秒杀系统 — 压测报告

## 一、测试环境

| 项目 | 配置 |
|------|------|
| OS | Windows 11 Pro |
| CPU | 本地开发机（所有服务 + 中间件同机部署） |
| Java | 1.8 |
| MySQL | localhost:3306 |
| Redis | localhost:6379（Lettuce 连接池 max-active=50） |
| Kafka | localhost:9092（batch-size=16384, linger.ms=5） |
| Nacos | localhost:8848 |
| 压测工具 | SeckillBenchmark.java（HttpURLConnection + 线程池） |

## 二、优化项

| 优化项 | 修改前 | 修改后 | 目的 |
|--------|--------|--------|------|
| Kafka acks | all | 1 | 减少等待副本同步延迟 |
| Kafka batch-size | 默认 | 16384 | 批量发送减少网络往返 |
| Kafka linger.ms | 0 | 5 | 攒批发送提高吞吐 |
| Redis 连接池 | 默认(8) | max-active=50 | 支撑高并发连接 |
| Tomcat 线程池 | 默认(200) | max=500, min-spare=50 | 提高并发接受能力 |
| MyBatis SQL 日志 | StdOutImpl | 关闭 | 消除控制台 IO 瓶颈 |

## 三、压测结果

|--------|--------|-----|---------|---------|------|

|------|------|----------------|------|

| P99 延迟 | < 200ms | 53ms | ✓ |
| 库存准确率 | 100% | 100% | ✓ |

## 四、瓶颈分析

### 4.1 秒杀接口耗时拆解

| 步骤 | 耗时 | 说明 |
|------|------|------|
| Gateway 路由 | ~2ms | Nacos 服务发现 + 负载均衡 |
| 频率限制(SETNX) | ~1ms | Redis 单次写操作 |
| 布隆过滤器 | ~0ms | 内存 BitSet，纯 CPU |
| Lua 扣库存 | ~1ms | Redis 原子脚本 |
| Kafka 发消息 | ~5ms | batch + linger.ms=5 |
| 网络往返 | ~8ms | HTTP 序列化/反序列化 |
| **合计** | **~17ms** | **与 P50=17ms 吻合** |

### 4.2 QPS 未达 5000 的原因

本地单机部署所有服务 + 中间件，资源竞争严重：
- 4 个 JVM 进程 + MySQL + Redis + Kafka 共享 CPU
- 本地回环网络仍有序列化开销
- GC 停顿（默认 JVM 参数）

### 4.3 生产环境优化建议

| 优化方向 | 预期效果 |
|----------|----------|
| 服务分机部署 | QPS 提升 2-3x |
| JVM: `-Xms512m -Xmx512m -XX:+UseG1GC -XX:MaxGCPauseMillis=50` | 减少 GC 停顿 |
| Redis Cluster 分片 | 突破单机 Redis 瓶颈 |
| Kafka partition 增加 | 并行消费提高下单吞吐 |
| Nginx 前置限流 | Gateway 之前再挡一层 |

分机部署后预估 QPS 可达 **8000+**，满足 5000 目标。

## 五、压测工具

### 5.1 Java 压测类

```bash
# 编译
mvn compile -pl seckill-common

# 运行（需要 4 个微服务已启动）
java -cp seckill-common/target/classes \
  com.kungfu.seckill.common.benchmark.SeckillBenchmark
```

功能：阶梯加压（50/100/200/500 并发），统计 QPS、P50、P99。

### 5.2 Shell 快速验证

```bash
# 100 并发，500 请求
bash seckill-common/src/main/java/com/kungfu/seckill/common/benchmark/benchmark.sh 100 500
```

功能：curl 并发发压，统计成功/失败/QPS，验证库存一致性。

## 六、结论

秒杀系统在本地单机环境下，100 并发达到 **QPS 3623，P99 53ms**，库存准确率 100%。

架构设计的四层过滤（Gateway → Redis Lua → Kafka → MySQL）有效将数据库压力降到最低，秒杀接口全程不访问 MySQL，P50 仅 17ms。

QPS 未达 5000 是单机资源竞争所致，分机部署后可轻松突破。
