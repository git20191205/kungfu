# 电商秒杀系统 — 架构设计文档

## 一、项目概述

### 1.1 业务背景

电商平台限时秒杀活动：少量商品（如 100 件 iPhone）在指定时间开放抢购，
瞬间涌入大量请求（预估 10 万+ 用户同时抢购），需要保证：

- 不超卖（库存准确率 100%）
- 不少卖（有库存就能卖出去）
- 高性能（QPS ≥ 5000，P99 < 200ms）
- 高可用（核心链路不能挂）

### 1.2 非功能需求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| QPS | ≥ 5000 | 下单接口峰值 |
| P99 延迟 | < 200ms | 99% 请求在 200ms 内返回 |
| 库存准确率 | 100% | 绝对不超卖 |
| 可用性 | 99.9% | 年停机 < 8.76 小时 |
| 并发用户 | 10 万+ | 同时在线抢购 |

### 1.3 核心挑战

```
挑战1: 瞬时高并发 → 10万请求在1秒内涌入
挑战2: 库存一致性 → 100件商品不能卖出101件
挑战3: 恶意请求   → 机器人刷单、重复提交
挑战4: 系统雪崩   → 一个服务挂了不能拖垮全部
```

### 2.2 分层设计

| 层级 | 组件 | 职责 | 过滤比例 |
|------|------|------|---------|
| 第1层 | CDN + 前端 | 静态资源缓存，倒计时在前端，减少无效请求 | 过滤 50% |
| 第2层 | Gateway + Sentinel | 限流（令牌桶）+ 鉴权 + 防刷 | 过滤 80% |
| 第3层 | Seckill Service + Redis | 库存预扣（Lua 原子操作），快速判断有没有库存 | 过滤 99% |
| 第4层 | Kafka + Order Service | 异步下单，削峰填谷 | 最终落库 |

核心思想：**层层过滤，把请求挡在数据库之前**

```
10万请求 → Gateway限流 → 2万请求 → Redis预扣 → 100个成功 → Kafka → MySQL落单
```

## 三、核心设计

### 3.1 秒杀库存方案

#### 方案选型

| 方案 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| MySQL 直接扣减 | 简单 | QPS < 1000，扛不住 | ✗ |
| Redis DECR | 快 | 非原子（判断+扣减分两步） | ✗ |
| Redis Lua 脚本 | 原子操作，QPS 10万+ | 需要预热库存到 Redis | ★ 选这个 |
| Redis + MySQL 双写 | 强一致 | 复杂，性能下降 | ✗ |

#### Lua 脚本（核心）

```lua
-- 秒杀扣库存 Lua 脚本
-- KEYS[1] = 库存 key (seckill:stock:{activityId})
-- ARGV[1] = 扣减数量 (通常为 1)

local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil then
    return -1  -- 活动不存在
end
if stock < tonumber(ARGV[1]) then
    return 0   -- 库存不足
end
redis.call('DECRBY', KEYS[1], ARGV[1])
return 1       -- 扣减成功
```

为什么用 Lua 而不是 DECR？
- DECR 可能减到负数（先 GET 判断再 DECR 不是原子的）
- Lua 脚本在 Redis 中原子执行，判断+扣减一步完成

#### 库存预热

```
秒杀开始前 5 分钟：
  1. 从 MySQL 加载库存到 Redis: SET seckill:stock:{activityId} 100
  2. 加载商品信息到 Redis 缓存
  3. 初始化布隆过滤器（已购用户去重）
```

### 3.2 防超卖设计

```
三道防线：

第1道: Redis Lua 原子扣减（挡住 99.9% 的并发）
  → 库存从 100 扣到 0，第 101 个请求直接返回"已售罄"

第2道: MySQL 乐观锁兜底
  UPDATE t_seckill_stock SET quantity = quantity - 1
  WHERE activity_id = ? AND quantity > 0
  → 即使 Redis 出问题，MySQL 也不会超卖

第3道: 唯一索引防重复购买
  t_seckill_order 表: UNIQUE(activity_id, user_id)
  → 同一用户同一活动只能买一次
```

### 3.3 异步下单（削峰填谷）

```
为什么不同步下单？
  同步: 10万请求 → 全部打到 MySQL → MySQL 挂了
  异步: 10万请求 → Redis 过滤 → 100个进 Kafka → 慢慢消费写 MySQL

流程:
  1. Seckill Service: Redis Lua 扣库存成功
  2. Seckill Service: 发送 Kafka 消息 {userId, activityId, productId}
  3. 返回用户: "下单中，请稍候查询"
  4. Order Service: 消费 Kafka → 创建订单 → MySQL 扣库存
  5. 用户轮询订单状态（或 WebSocket 推送）
```

### 3.4 限流方案

```
三级限流:

第1级: Nginx 限流（连接数限制 + 请求速率限制）
  limit_req_zone $binary_remote_addr zone=seckill:10m rate=10r/s;

第2级: Gateway + Sentinel（接口级限流）
  秒杀接口: QPS 上限 5000
  普通接口: QPS 上限 1000
  降级策略: 超限直接返回 "系统繁忙，请稍后重试"

第3级: 业务层防刷
  同一用户 5 秒内只能请求 1 次（Redis SETNX + TTL）
  布隆过滤器: 已购买用户直接拦截
```

### 3.5 防刷设计

| 策略 | 实现 | 说明 |
|------|------|------|
| 用户频率限制 | Redis SETNX user:{userId}:seckill TTL=5s | 5 秒内只能抢一次 |
| 重复购买拦截 | 布隆过滤器 + DB 唯一索引 | 同一活动只能买一次 |
| 验证码 | 秒杀前弹出验证码 | 防机器人，分散请求 |
| 接口隐藏 | 秒杀开始前不暴露真实 URL | 防提前刷接口 |
| 黑名单 | Redis Set 存储黑名单用户 | 异常用户直接拦截 |

```
t_seckill_activity (秒杀活动)
  ├── id, activity_name, product_id, seckill_price

  ├── id, activity_id (UNIQUE)
  └── quantity

  ├── id, order_no, activity_id, user_id
  ├── product_id, seckill_price, quantity
  ├── status (PENDING/PAID/CANCELLED/TIMEOUT)
  └── UNIQUE(activity_id, user_id) ← 防重复购买

t_payment (支付表 — 复用 W08)
  ├── id, payment_no, order_no
  ├── amount, status
  └── create_time
```

## 七、性能优化策略

### 7.1 读优化

| 优化点 | 方案 | 效果 |
|--------|------|------|
| 活动详情 | Redis 缓存 + 本地缓存（Caffeine） | 0 次 DB 查询 |
| 库存数量 | 直接读 Redis（GET） | 毫秒级响应 |
| 秒杀结果 | Redis 缓存订单状态 | 减少 DB 压力 |

### 7.2 写优化

| 优化点 | 方案 | 效果 |
|--------|------|------|
| 库存扣减 | Redis Lua 原子操作 | QPS 10万+ |
| 下单 | Kafka 异步 | MySQL 不被打爆 |
| 支付回调 | Kafka 异步 | 解耦 |

### 7.3 预期性能

```
压测目标（单机）:
  秒杀接口 QPS: ≥ 5000
  P99 延迟: < 200ms
  P50 延迟: < 50ms

瓶颈分析:
  Redis Lua: 单机 10万+ QPS → 不是瓶颈
  Kafka 发送: 单机 5万+ QPS → 不是瓶颈
  Gateway 限流: 配置 5000 QPS → 控制入口
  MySQL 写入: 通过 Kafka 削峰，实际 QPS < 1000 → 可承受
```

## 八、高可用设计

### 8.1 服务高可用

| 组件 | 方案 |
|------|------|
| Seckill Service | 多实例部署（Nacos 负载均衡） |
| Redis | 主从 + Sentinel（或 Cluster） |
| MySQL | 主从复制 + 读写分离 |
| Kafka | 3 Broker + replication-factor=3 |
| Nacos | 集群模式（3 节点） |

### 8.2 降级方案

```
降级策略（按严重程度递增）:

Level 1: 限流降级
  → Sentinel 限流，超限返回 "系统繁忙"

Level 2: 熔断降级
  → 下游服务不可用时，直接返回降级结果

Level 3: 静态化降级
  → 秒杀页面切换为静态页（CDN 直接返回 "已售罄"）

Level 4: 关闭秒杀
  → 紧急情况下关闭秒杀入口，保护核心交易链路
```

## 十二、面试话术

```
面试官: 说说你做过的秒杀系统？

回答框架:

1. 业务背景（10秒）
   "我们做了一个电商秒杀系统，100件商品10万人抢，
    核心目标是不超卖、高性能、高可用。"

2. 架构设计（30秒）
   "采用分层过滤架构：
    第1层 Gateway + Sentinel 限流，挡住 80% 请求；
    第2层 Redis Lua 原子扣库存，挡住 99%；
```
