# 秒杀系统 — 故障排查手册

> 生产环境常见问题诊断与解决方案

## 一、性能问题

### 1.1 QPS 突然下降

**现象**：QPS 从 6000+ 突然降至 2000

**排查步骤**：

```bash
# 1. 检查服务状态
curl http://localhost:8084/actuator/health

# 2. 查看 CPU 使用率
top -p $(pgrep -f seckill-service)

# 3. 查看 GC 情况
jstat -gcutil $(pgrep -f seckill-service) 1000

# 4. 查看线程状态
jstack $(pgrep -f seckill-service) | grep -A 10 "BLOCKED\|WAITING"

# 5. 查看网络连接
netstat -an | grep 8084 | wc -l
```

**常见原因及解决方案**：

| 原因 | 症状 | 解决方案 |
|------|------|---------|
| Full GC 频繁 | CPU 100%，响应慢 | 增加堆内存，优化 GC 参数 |
| 线程池耗尽 | 大量 WAITING 线程 | 增加线程池大小 |
| Redis 连接池耗尽 | 大量 timeout 异常 | 增加连接池大小 |
| 数据库慢查询 | 响应时间长 | 优化 SQL，添加索引 |
| 网络带宽打满 | 网络延迟高 | 升级带宽，优化数据传输 |

**示例：Full GC 频繁**

```bash
# 1. 查看 GC 日志
tail -f logs/gc.log

# 2. 生成堆转储
jmap -dump:live,format=b,file=heap.hprof $(pgrep -f seckill-service)

# 3. 分析堆转储（使用 MAT 或 jhat）
jhat -J-Xmx4g heap.hprof

# 4. 临时解决：重启服务
docker restart seckill-service

# 5. 长期解决：调整 JVM 参数
JAVA_OPTS="-Xms8g -Xmx8g -XX:+UseG1GC -XX:MaxGCPauseMillis=50"
```

### 1.2 P99 延迟突增

**现象**：P99 延迟从 28ms 突增至 500ms

**排查步骤**：

```bash
# 1. 查看 Prometheus 指标
curl http://localhost:9090/api/v1/query?query=histogram_quantile(0.99,http_server_requests_seconds_bucket)

# 2. 查看慢请求日志
grep "took more than" logs/application.log | tail -20

# 3. 查看 Redis 延迟
redis-cli --latency -h 192.168.1.70

# 4. 查看 Kafka 延迟
kafka-consumer-perf-test.sh \
  --bootstrap-server localhost:9092 \
  --topic seckill-order \
  --messages 1000

# 5. 查看数据库慢查询
mysql -e "SELECT * FROM mysql.slow_log ORDER BY query_time DESC LIMIT 10;"
```

**常见原因及解决方案**：

| 原因 | 症状 | 解决方案 |
|------|------|---------|
| Redis 慢查询 | Redis 延迟 > 10ms | 优化 Lua 脚本，避免大 key |
| Kafka 消费堆积 | lag 持续增长 | 增加消费者实例，增加 partition |
| 数据库锁等待 | 大量 Waiting for lock | 优化事务，减少锁范围 |
| 网络抖动 | 偶发性延迟 | 检查网络设备，优化路由 |
| JVM GC 停顿 | 周期性延迟峰值 | 优化 GC 参数，减少停顿时间 |

**示例：Redis 慢查询**

```bash
# 1. 查看慢查询日志
redis-cli slowlog get 10

# 2. 分析慢查询
redis-cli --bigkeys

# 3. 优化 Lua 脚本
# 避免在 Lua 中遍历大量 key
# 使用 pipeline 批量操作

# 4. 监控 Redis 性能
redis-cli --stat
```

### 1.3 内存泄漏

**现象**：内存持续增长，最终 OOM

**排查步骤**：

```bash
# 1. 查看内存使用
docker stats seckill-service

# 2. 查看堆内存
jmap -heap $(pgrep -f seckill-service)

# 3. 查看对象统计
jmap -histo:live $(pgrep -f seckill-service) | head -20

# 4. 生成堆转储
jmap -dump:live,format=b,file=heap_$(date +%Y%m%d_%H%M%S).hprof $(pgrep -f seckill-service)

# 5. 分析堆转储
# 使用 Eclipse MAT 或 VisualVM
```

**常见原因及解决方案**：

| 原因 | 症状 | 解决方案 |
|------|------|---------|
| 缓存未清理 | 大量缓存对象 | 设置缓存过期时间，使用 LRU |
| 连接未关闭 | 大量 Connection 对象 | 使用 try-with-resources |
| 线程未释放 | 大量 Thread 对象 | 使用线程池，避免手动创建线程 |
| 监听器未移除 | 大量 Listener 对象 | 及时移除监听器 |
| 静态集合持有引用 | 静态 Map/List 持续增长 | 避免静态集合，使用弱引用 |

## 二、功能问题

### 2.1 库存超卖

**现象**：库存为 100，但卖出了 105 件

**排查步骤**：

```bash
# 1. 查询数据库实际库存
mysql -e "SELECT * FROM seckill_activity WHERE id=1;"

# 2. 查询 Redis 库存
redis-cli -c -h 192.168.1.70 GET "seckill:stock:1"

# 3. 查询订单数量
mysql -e "SELECT COUNT(*) FROM seckill_order WHERE activity_id=1 AND status='SUCCESS';"

# 4. 检查 Lua 脚本
redis-cli -c -h 192.168.1.70 SCRIPT LOAD "$(cat lua/deduct_stock.lua)"

# 5. 查看日志
grep "超卖" logs/application.log
```

**可能原因**：

1. **Redis Lua 脚本问题**
   ```lua
   -- 错误示例：未检查库存
   local stock = redis.call('GET', KEYS[1])
   redis.call('DECR', KEYS[1])  -- 直接扣减，可能为负
   
   -- 正确示例：先检查再扣减
   local stock = tonumber(redis.call('GET', KEYS[1]))
   if stock <= 0 then
       return 0
   end
   redis.call('DECR', KEYS[1])
   return 1
   ```

2. **MySQL 乐观锁失效**
   ```sql
   -- 错误示例：未检查更新结果
   UPDATE seckill_activity 
   SET remain_stock = remain_stock - 1 
   WHERE id = 1;
   
   -- 正确示例：检查库存
   UPDATE seckill_activity 
   SET remain_stock = remain_stock - 1 
   WHERE id = 1 AND remain_stock > 0;
   
   -- 检查影响行数
   if (affectedRows == 0) {
       throw new StockNotEnoughException();
   }
   ```

3. **唯一索引失效**
   ```sql
   -- 检查唯一索引
   SHOW INDEX FROM seckill_order WHERE Key_name = 'uk_activity_user';
   
   -- 如果不存在，创建唯一索引
   ALTER TABLE seckill_order 
   ADD UNIQUE INDEX uk_activity_user (activity_id, user_id);
   ```

**解决方案**：

```bash
# 1. 停止秒杀活动
redis-cli -c -h 192.168.1.70 SET "seckill:status:1" "CLOSED"

# 2. 核对数据
# 统计订单数量
ORDER_COUNT=$(mysql -N -e "SELECT COUNT(*) FROM seckill_order WHERE activity_id=1 AND status='SUCCESS';")
# 查询初始库存
INIT_STOCK=$(mysql -N -e "SELECT stock FROM seckill_activity WHERE id=1;")
# 计算剩余库存
REMAIN_STOCK=$((INIT_STOCK - ORDER_COUNT))

# 3. 修正库存
mysql -e "UPDATE seckill_activity SET remain_stock=$REMAIN_STOCK WHERE id=1;"
redis-cli -c -h 192.168.1.70 SET "seckill:stock:1" $REMAIN_STOCK

# 4. 取消超卖订单（如果有）
mysql -e "DELETE FROM seckill_order WHERE activity_id=1 AND id > (SELECT id FROM (SELECT id FROM seckill_order WHERE activity_id=1 ORDER BY id LIMIT $INIT_STOCK) t ORDER BY id DESC LIMIT 1);"

# 5. 重新开启活动
redis-cli -c -h 192.168.1.70 SET "seckill:status:1" "RUNNING"
```

### 2.2 订单重复

**现象**：同一用户生成了多个订单

**排查步骤**：

```bash
# 1. 查询重复订单
mysql -e "SELECT user_id, activity_id, COUNT(*) as cnt FROM seckill_order GROUP BY user_id, activity_id HAVING cnt > 1;"

# 2. 检查唯一索引
mysql -e "SHOW INDEX FROM seckill_order WHERE Key_name = 'uk_activity_user';"

# 3. 查看布隆过滤器状态
curl http://localhost:8084/actuator/metrics/bloom.filter.contains

# 4. 查看日志
grep "重复下单" logs/application.log
```

**可能原因**：

1. **唯一索引未生效**
2. **布隆过滤器误判**
3. **并发请求导致**

**解决方案**：

```bash
# 1. 创建唯一索引（如果不存在）
mysql -e "ALTER TABLE seckill_order ADD UNIQUE INDEX uk_activity_user (activity_id, user_id);"

# 2. 删除重复订单（保留最早的）
mysql -e "
DELETE o1 FROM seckill_order o1
INNER JOIN seckill_order o2 
WHERE o1.activity_id = o2.activity_id 
  AND o1.user_id = o2.user_id 
  AND o1.id > o2.id;
"

# 3. 重建布隆过滤器
curl -X POST http://localhost:8084/api/seckill/activity/1/warmup
```

### 2.3 消息堆积

**现象**：Kafka 消息堆积，订单延迟创建

**排查步骤**：

```bash
# 1. 查看消费者 lag
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe --group seckill-order-group

# 2. 查看 topic 详情
kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --describe --topic seckill-order

# 3. 查看消费者日志
docker logs -f seckill-order | grep "Consumed"

# 4. 查看消费速率
curl http://localhost:8085/actuator/metrics/kafka.consumer.records.consumed.rate
```

**可能原因**：

| 原因 | 症状 | 解决方案 |
|------|------|---------|
| 消费者处理慢 | lag 持续增长 | 优化消费逻辑，增加消费者实例 |
| partition 数量少 | 单个 partition 堆积 | 增加 partition 数量 |
| 数据库写入慢 | 消费者阻塞 | 优化 SQL，批量插入 |
| 消费者宕机 | 无消费进度 | 重启消费者，检查日志 |

**解决方案**：

```bash
# 1. 增加消费者实例
docker-compose scale seckill-order=3

# 2. 增加 partition（需要停止消费者）
kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --alter --topic seckill-order \
  --partitions 10

# 3. 优化消费逻辑（批量插入）
# 修改代码，使用批量插入
@KafkaListener(topics = "seckill-order", concurrency = "10")
public void consume(List<SeckillMessage> messages) {
    orderService.batchInsert(messages);
}

# 4. 临时方案：手动消费堆积消息
kafka-console-consumer.sh \
  --bootstrap-server localhost:9092 \
  --topic seckill-order \
  --from-beginning \
  --max-messages 10000
```

## 三、中间件问题

### 3.1 Redis 连接超时

**现象**：大量 `RedisConnectionException: Could not get a resource from the pool`

**排查步骤**：

```bash
# 1. 查看 Redis 连接数
redis-cli -c -h 192.168.1.70 INFO clients

# 2. 查看连接池配置
curl http://localhost:8084/actuator/configprops | jq '.spring.redis.lettuce.pool'

# 3. 查看应用连接数
netstat -an | grep 6379 | wc -l

# 4. 查看 Redis 慢查询
redis-cli -c -h 192.168.1.70 SLOWLOG GET 10
```

**解决方案**：

```yaml
# 增加连接池大小
spring:
  redis:
    lettuce:
      pool:
        max-active: 100  # 从 50 增加到 100
        max-idle: 50
        min-idle: 10
        max-wait: 5000
```

### 3.2 MySQL 连接池耗尽

**现象**：`HikariPool - Connection is not available`

**排查步骤**：

```bash
# 1. 查看数据库连接数
mysql -e "SHOW PROCESSLIST;"

# 2. 查看连接池配置
curl http://localhost:8084/actuator/configprops | jq '.spring.datasource.hikari'

# 3. 查看慢查询
mysql -e "SELECT * FROM mysql.slow_log ORDER BY query_time DESC LIMIT 10;"
```

**解决方案**：

```yaml
# 增加连接池大小
spring:
  datasource:
    hikari:
      maximum-pool-size: 100  # 从 50 增加到 100
      minimum-idle: 20
      connection-timeout: 30000
      idle-timeout: 600000
      max-lifetime: 1800000
```

### 3.3 Nacos 服务下线

**现象**：服务在 Nacos 中显示为下线状态

**排查步骤**：

```bash
# 1. 查看 Nacos 服务列表
curl http://192.168.1.60:8848/nacos/v1/ns/instance/list?serviceName=seckill-service

# 2. 查看服务心跳

# 3. 查看网络连接
telnet 192.168.1.60 8848

# 4. 查看日志
grep "nacos" logs/application.log | tail -20
```

**解决方案**：

```bash

```

## 四、监控告警

### 4.1 配置告警规则

```yaml
# prometheus/alert-rules.yml
groups:
- name: seckill-critical
  rules:
  # 服务宕机
  - alert: ServiceDown
    expr: up{job="seckill-service"} == 0
    for: 1m
    labels:
      severity: critical
    annotations:
      summary: "服务 {{ $labels.instance }} 宕机"
      description: "服务已宕机超过 1 分钟"
  
  # 高延迟
  - alert: HighLatency
    expr: histogram_quantile(0.99, http_server_requests_seconds_bucket) > 0.2
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "P99 延迟超过 200ms"
      description: "当前 P99 延迟: {{ $value }}s"
  
  # 高错误率
  - alert: HighErrorRate
    expr: rate(http_server_requests_seconds_count{status=~"5.."}[5m]) / rate(http_server_requests_seconds_count[5m]) > 0.01
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "错误率超过 1%"
      description: "当前错误率: {{ $value | humanizePercentage }}"
  
  # 内存使用过高
  - alert: HighMemoryUsage
    expr: jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} > 0.9
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "堆内存使用超过 90%"
      description: "当前使用率: {{ $value | humanizePercentage }}"
  
  # Kafka 消息堆积
  - alert: KafkaLag
    expr: kafka_consumer_lag > 10000
    for: 5m
    labels:
      severity: warning
    annotations:
      summary: "Kafka 消息堆积超过 10000"
      description: "当前 lag: {{ $value }}"
```

## 五、应急响应流程

### 5.1 故障响应流程

```
1. 接收告警
   ↓
2. 确认故障（查看监控、日志）
   ↓
3. 评估影响（影响范围、用户数）
   ↓
4. 快速止损（降级、限流、切流量）
   ↓
5. 定位问题（日志分析、性能分析）
   ↓
6. 修复问题（代码修复、配置调整）
   ↓
7. 验证恢复（压测验证、监控观察）
   ↓
8. 复盘总结（根因分析、改进措施）
```

### 5.2 快速止损手册

| 故障类型 | 止损措施 | 命令 |
|---------|---------|------|
| 服务宕机 | 重启服务 | `docker restart seckill-service` |
| QPS 过高 | 限流 | 修改 Nginx 限流配置 |
| 数据库慢 | 降级 | 关闭非核心功能 |
| Redis 故障 | 切换 | 切换到备用 Redis |
| 消息堆积 | 扩容 | 增加消费者实例 |

## 七、总结

本手册涵盖了秒杀系统生产环境中最常见的故障场景和解决方案。关键要点：

1. **快速定位**：通过监控、日志快速定位问题
2. **快速止损**：优先恢复服务，再分析根因
3. **预防为主**：完善监控告警，提前发现问题
4. **持续改进**：每次故障后复盘，避免重复发生

建议定期演练故障场景，提高应急响应能力。
