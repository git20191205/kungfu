# 数据模型设计

> 系统设计第三步：把业务需求转化为数据库表结构

## 一、为什么数据模型设计很重要？

数据模型是系统的基石，设计不好会导致：
1. **性能问题**：查询慢、索引失效
2. **扩展困难**：无法分库分表
3. **数据不一致**：并发更新冲突
4. **维护成本高**：表结构混乱

**好的数据模型**：
- 满足业务需求
- 性能高效
- 易于扩展
- 数据一致

## 二、关系型 vs NoSQL 选型

### 2.1 选型决策树

```
开始
  ↓
需要事务支持？
  ├─ 是 → 关系型数据库（MySQL / PostgreSQL）
  └─ 否 → 继续
           ↓
       数据结构复杂？
         ├─ 是 → 关系型数据库
         └─ 否 → 继续
                  ↓
              需要 JOIN 查询？
                ├─ 是 → 关系型数据库
                └─ 否 → 继续
                         ↓
                     数据量 > 1 TB？
                       ├─ 是 → NoSQL（MongoDB / Cassandra）
                       └�� 否 → 关系型数据库
```

### 2.2 关系型数据库（MySQL）

**适用场景**：
- 需要事务支持（ACID）
- 数据结构复杂，需要 JOIN
- 数据量 < 1 TB
- 强一致性要求

**典型案例**：
- 订单系统（需要事务）
- 用户系统（需要 JOIN）
- 秒杀系��（需要强一致性）

**优点**：
- 事务支持
- 数据一致性强
- 生态成熟

**缺点**：
- 扩展困难（需要分库分表）
- 性能有限（单机 QPS < 5000）

### 2.3 NoSQL 数据库

#### MongoDB（文档型）

**适用场景**：
- 数据结构灵活（Schema-less）
- 不需要 JOIN
- 读多写少

**典型案例**：
- 日志存储
- 用户画像
- 商品详情

**优点**：
- Schema 灵活
- 水平扩展容易
- 查询性能好

**缺点**：
- 不支持事务（4.0 后支持）
- 不支持 JOIN

#### Redis（KV 型）

**适用场景**：
- 缓存
- 计数器
- 排行榜

**典型案例**：
- 秒杀库存（Redis Lua）
- 短链缓存
- 用户 Session

**优点**：
- 性能极高（QPS 10 万+）
- 数据结构丰富（String / Hash / List / Set / ZSet）

**缺点**：
- 内存限制
- 持久化有限

#### Cassandra（列式）

**适用场景**：
- 海量数据（PB 级）
- 写多读少
- 时序数据

**典型案例**：
- 监控数据
- 日志分析
- IoT 数据

**优点**：
- 水平扩展容易
- 写性能极高

**缺点**：
- 查询灵活性差
- 学习成本高

### 2.4 选型对比表

| 数据库 | 事务 | JOIN | 扩展性 | 性能 | 适用场景 |
|--------|------|------|--------|------|---------|
| MySQL | ✓ | ✓ | 中 | 中 | 订单、用户 |
| MongoDB | △ | ✗ | 高 | 高 | 日志、画像 |
| Redis | ✗ | ✗ | 高 | 极高 | 缓存、计数 |
| Cassandra | ✗ | ✗ | 极高 | 极高 | 监控、日志 |

## 三、ER 图设计规范

### 3.1 实体（Entity）

**定义**：业务中的对象，对应数据库中的表

**秒杀系统实体**：
- 用户（User）
- 活动（Activity）
- 订单（Order）
- 支付（Payment）

**ER 图表示**：

```
┌─────────────┐
│   Activity  │  ← 实体（矩形）
├─────────────┤
│ id          │  ← 主键（下划线）
│ name        │
│ stock       │
│ price       │
└─────────────┘
```

### 3.2 属性（Attribute）

**定义**：实体的特征，对应表中的字段

**属性类型**：
- 主键（Primary Key）：唯一标识
- 外键（Foreign Key）：关联其他表
- 普通属性：业务字段
- 时间戳：create_time / update_time

**命名规范**：
- 使用小写 + 下划线（snake_case）
- 避免使用保留字（order / user / group）
- 时间字段统一用 _time 后缀

### 3.3 关系（Relationship）

**一对一（1:1）**：
- 用户 ↔ 用户详情
- 订单 ↔ 支付记录

**一对多（1:N）**：
- 活动 ↔ 订单（一个活动有多个订单）
- 用户 ↔ 订单（一个用户有多个订单）

**多对多（M:N）**：
- 用户 ↔ 角色（一个用户有多个角色，一个角色有多个用户）
- 需要中间表

**ER 图表示**：

```
┌──────────┐       1       N  ┌──────────┐
│ Activity │───────────────────│  Order   │
└──────────┘                   └──────────┘
    一个活动有多个订单（1:N）
```

### 3.4 秒杀系统 ER 图


## 四、分库分表策略

### 4.1 什么时候需要分库分表？

**垂直拆分**：按业务拆分
- 单表数据量 > 1000 万
- 单库 QPS > 5000
- 不同业务混在一起

**水平拆分**：按数据拆分
- 单表数据量 > 1000 万
- 查询变慢
- 需要扩展

### 4.2 垂直拆分

**定义**：按业务模块拆分成多个库

**秒杀系统案例**：

```
原始（单库）：
seckill_db
  ├── user
  ├── activity
  ├── order
  └── payment

拆分后（多库）：
user_db          activity_db      order_db         payment_db
  └── user         └── activity     └── order        └── payment
```

**优点**：
- 业务隔离，互不影响
- 扩展容易

**缺点**：
- 无法 JOIN
- 分布式事务

### 4.3 水平拆分

**定义**：按数据范围拆分成多个表

**短链系统案例**：

```
原始（单表）：
short_url（18 亿条）

拆分后（分库分表）：
db_0                db_1                db_2
  ├── short_url_0     ├── short_url_0     ├── short_url_0
  ├── short_url_1     ├── short_url_1     ├── short_url_1
  ├── ...             ├── ...             ├── ...
  └── short_url_15    └── short_url_15    └── short_url_15

总计：3 个库 × 16 张表 = 48 张表
每张表：18 亿 / 48 = 3750 万条
```

**分片键选择**：
- 用户 ID：按用户分片（适合用户维度查询）
- 时间：按时间分片（适合时序数据）
- Hash：按 Hash 分片（数据均匀分布）

**短链系统分片策略**：

```java
// 按 short_code 的 Hash 分片
int dbIndex = hash(short_code) % 3;      // 3 个库
int tableIndex = hash(short_code) % 16;  // 16 张表

// 例如：short_code = "abc123"
// hash("abc123") = 12345678
// dbIndex = 12345678 % 3 = 0 → db_0
// tableIndex = 12345678 % 16 = 14 → short_url_14
// 最终路由到：db_0.short_url_14
```

### 4.4 分库分表中间件

| 中间件 | 类型 | 优点 | 缺点 |
|--------|------|------|------|
| ShardingSphere | 客户端 | 性能好、功能强 | 侵入性强 |
| MyCat | 代理 | 透明、易用 | 性能损耗 |
| TDDL | 客户端 | 阿里开源 | 文档少 |

## 五、索引设计原则

### 5.1 索引类型

**主键索引（Primary Key）**：
- 唯一标识
- 自动创建
- 聚簇索引（InnoDB）

**唯一索引（Unique）**：
- 保证唯一性
- 可以有多个

**普通索引（Index）**：
- 加速查询
- 可以有多个

**联合索引（Composite）**：
- 多个字段组合
- 遵循最左前缀原则

**全文索引（Full-Text）**：
- 文本搜索
- MySQL 5.6+ 支持

### 5.2 索引设计原则

**原则 1：选择性高的字段建索引**

```sql
-- 好：user_id 选择性高（每个用户唯一）
CREATE INDEX idx_user_id ON seckill_order(user_id);

-- 差：status 选择性低（只有 3 个值：PENDING / SUCCESS / FAILED）
CREATE INDEX idx_status ON seckill_order(status);  -- 不推荐
```

**原则 2：WHERE / ORDER BY / JOIN 的字段建索引**

```sql
-- 查询：WHERE user_id = ? ORDER BY create_time DESC
CREATE INDEX idx_user_time ON seckill_order(user_id, create_time);
```

**原则 3：联合索引遵循最左前缀**

```sql
-- 联合索引：(activity_id, user_id, create_time)
CREATE INDEX idx_activity_user_time ON seckill_order(activity_id, user_id, create_time);

-- 可以使用索引的查询：
WHERE activity_id = ?                              -- ✓
WHERE activity_id = ? AND user_id = ?              -- ✓
WHERE activity_id = ? AND user_id = ? AND create_time > ?  -- ✓

-- 无法使用索引的查询：
WHERE user_id = ?                                  -- ✗（跳过了 activity_id）
WHERE create_time > ?                              -- ✗（跳过了 activity_id 和 user_id）
```

**原则 4：避免冗余索引**

```sql
-- 冗余：已有 (activity_id, user_id)，不需要单独的 (activity_id)
CREATE INDEX idx_activity ON seckill_order(activity_id);  -- 冗余
CREATE INDEX idx_activity_user ON seckill_order(activity_id, user_id);
```

**原则 5：控制索引数量**

- 单表索引 < 5 个
- 索引越多，写入越慢
- 索引占用存储空间

### 5.3 秒杀系统索引设计

**订单表（seckill_order）**：

```sql
CREATE TABLE seckill_order (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  user_id BIGINT NOT NULL,
  activity_id BIGINT NOT NULL,
  order_no VARCHAR(32) NOT NULL,
  status TINYINT NOT NULL DEFAULT 0,
  create_time DATETIME NOT NULL,
  update_time DATETIME NOT NULL,
  
  -- 索引设计
  UNIQUE KEY uk_activity_user (activity_id, user_id),  -- 防止重复购买
  INDEX idx_user_time (user_id, create_time),          -- 用户订单列表
  INDEX idx_order_no (order_no)                        -- 订单号查询
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**索引说明**：
1. `uk_activity_user`：唯一索引，防止同一用户重复购买同一活动
2. `idx_user_time`：联合索引，支持用户订单列表查询（按时间倒序）
3. `idx_order_no`：普通索引，支持订单号查询

## 六、实战案例：秒杀系统数据模型

### 6.1 活动表（seckill_activity）

```sql
CREATE TABLE seckill_activity (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '活动ID',
  name VARCHAR(100) NOT NULL COMMENT '活动名称',
  product_id BIGINT NOT NULL COMMENT '商品ID',
  stock INT NOT NULL COMMENT '库存',
  remain_stock INT NOT NULL COMMENT '剩余库存',
  price DECIMAL(10,2) NOT NULL COMMENT '秒杀价格',
  original_price DECIMAL(10,2) NOT NULL COMMENT '原价',
  start_time DATETIME NOT NULL COMMENT '开始时间',
  end_time DATETIME NOT NULL COMMENT '结束时间',

  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  
  INDEX idx_start_time (start_time),
  INDEX idx_status (status)

```

### 6.2 订单表（seckill_order）

```sql
CREATE TABLE seckill_order (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '订单ID',

  order_no VARCHAR(32) NOT NULL COMMENT '订单号',

  product_name VARCHAR(100) NOT NULL COMMENT '商品名称',
  price DECIMAL(10,2) NOT NULL COMMENT '价格',
  status TINYINT NOT NULL DEFAULT 0 COMMENT '状态：0-待支付 1-已支付 2-已取消',
  create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  
  UNIQUE KEY uk_activity_user (activity_id, user_id) COMMENT '防止重复购买',
  INDEX idx_user_time (user_id, create_time) COMMENT '用户订单列表',
  INDEX idx_order_no (order_no) COMMENT '订单号查询'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='秒杀订单表';
```

### 6.3 支付表（seckill_payment）

```sql
CREATE TABLE seckill_payment (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '支付ID',

  user_id BIGINT NOT NULL COMMENT '用户ID',
  amount DECIMAL(10,2) NOT NULL COMMENT '支付金额',

  pay_no VARCHAR(64) COMMENT '第三方支付流水号',
  status TINYINT NOT NULL DEFAULT 0 COMMENT '状态：0-待支付 1-已支付 2-已退款',

  update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  
  UNIQUE KEY uk_order_id (order_id) COMMENT '一个订单只能有一条支付记录',
  INDEX idx_order_no (order_no),
  INDEX idx_pay_no (pay_no)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='支付表';
```

## 八、数据模型设计面试话术

### 8.1 标准流程

```
1. 识别实体
   "这个系统有哪些核心实体？用户、活动、订单..."

2. 定义属性
   "每个实体有哪些属性？活动有名称、库存、价格..."

3. 确定关系
   "实体之间是什么关系？一个活动有多个订单（1:N）"

4. 设计索引
   "哪些字段需要建索引？WHERE / ORDER BY / JOIN 的字段"

5. 考虑分库分表
   "数据量多大？是否需要分库分表？"
```

### 8.2 常见追问

| 追问 | 应对 |
|------|------|
| "为什么选择 MySQL 而不是 MongoDB？" | "需要事务支持，数据结构复杂，需要 JOIN" |
| "为什么要建唯一索引？" | "防止重复购买，保证数据一致性" |
| "如果数据量增长到 10 亿怎么办？" | "分库分表，按 Hash 分片，每张表 1000 万条" |
| "如何保证分库分表后的查询性能？" | "选择合适的分片键，避免跨库查询" |

## 九、数据模型设计速查表

### 9.1 字段类型选择

| 数据类型 | MySQL 类型 | 说明 |
|---------|-----------|------|
| 整数 | BIGINT | 主键、外键 |
| 小整数 | TINYINT | 状态、类型 |
| 字符串 | VARCHAR | 名称、描述 |
| 长文本 | TEXT | 富文本内容 |
| 金额 | DECIMAL(10,2) | 避免精度丢失 |
| 时间 | DATETIME | 精确到秒 |
| 布尔 | TINYINT(1) | 0/1 |

### 9.2 索引设计 Checklist

```
□ 主键索引：自动创建
□ 唯一索引：防止重复数据
□ WHERE 字段：加速查询
□ ORDER BY 字段：加速排序
□ JOIN 字段：加速关联
□ 联合索引：遵循最左前缀
□ 控制数量：单表 < 5 个
```

### 9.3 分库分表 Checklist

```
□ 数据量 > 1000 万？
□ QPS > 5000？
□ 选择分片键（用户ID / 时间 / Hash）
□ 确定分片数量（库数 × 表数）
□ 考虑扩容方案（翻倍扩容）
□ 处理跨库查询（避免或异步）
```

## 十、总结

数据模型设计是系统设计的核心，直接影响性能和扩展性。

**核心要点**：
1. **选型**：关系型（事务、JOIN）vs NoSQL（扩展、性能）
2. **ER 图**：实体、属性、关系
3. **索引**：WHERE / ORDER BY / JOIN 的字段，遵循最左前缀
4. **分库分表**：数据量 > 1000 万，按 Hash / 用户ID / 时间分片
5. **字段类型**：BIGINT（主键）、VARCHAR（字符串）、DECIMAL（金额）、DATETIME（时间）

**面试话术**：
- "我先识别一下核心实体..."
- "这些实体之间是 1:N 关系..."
- "需要在 WHERE 字段上建索引..."
- "数据量大需要分库分表，按 Hash 分片..."
