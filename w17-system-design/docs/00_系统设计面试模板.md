# 系统设计面试模板

> 汇总 D91-D96 方法论，形成完整的面试答题框架

## 一、系统设计面试 4 步法

```
第 1 步：需求澄清（5-10 分钟）
  ↓
第 2 步：容量估算（5 分钟）
  ↓
第 3 步：高层架构（10 分钟）
  ↓
第 4 步：深入设计（20 分钟）
```

**总时间**：45 分钟（留 5 分钟总结与追问）

## 二、第 1 步：需求澄清（5-10 分钟）

### 2.1 使用 5W1H 方法

| 问题 | 示例 |
|------|------|
| **What** | "设计一个秒杀系统，核心功能是什么？下单、支付、查询？" |
| **Who** | "谁会使用？普通用户还是商家？" |
| **When** | "什么时候使用？活动时间多长？" |
| **Where** | "在哪里使用？移动端还是 PC 端？" |
| **Why** | "为什么需要？解决什么问题？" |
| **How** | "怎么使用？用户如何参与秒杀？" |

### 2.2 确认功能需求

**模板**：

```
面试官："设计一个秒杀系统"

你："好的，我先确认一下功能范围：
1. 核心功能：用户浏览活动、下单、支付、订单查询，对吗？
2. 是否需要支持多个活动并发？
3. 是否需要支持用户限购（每人只能买一件）？
4. 是否需要支持活动预热？"
```

### 2.3 确认非功能需求

**模板**：

```
你："接下来确认一下非功能需求：
1. 预计有多少用户参与？（如 10 万用户）
2. 峰值 QPS 大概多少？（如 5000 QPS）
3. 库存有多少件？（如 100 件）
4. 对延迟有什么要求？（如 P99 < 200ms）
5. 对可用性有什么要求？（如 99.9% SLA）"
```

### 2.4 需求拆解（SMART 原则）

```
S（Specific）：具体的需求
M（Measurable）：可衡量的指标
A（Achievable）：可实现的方案
R（Relevant）：相关的业务
T（Time-bound）：有时限的目标
```

## 三、第 2 步：容量估算（5 分钟）

### 3.1 QPS 估算

**公式**：

```
QPS = DAU × 平均每用户请求数 / 86400 × 峰值系数

秒杀系统案例：
- 10 万用户参与
- 每人平均 5 次请求
- 前 10 秒集中 80% 请求

瞬时 QPS = 100,000 × 5 × 0.8 / 10 = 40,000 QPS
```

### 3.2 存储估算

**公式**：

```
存储 = 单条记录 × 记录数 × 副本数 × 增长率

秒杀系统案例：
- 单条订单：100 字节
- 成功订单：100 条
- 副本数：2
- 增长率：1.3

存储 = 100B × 100 × 2 × 1.3 = 26 KB（极小）
```

### 3.3 带宽估算

**公式**：

```
带宽 = QPS × 响应大小 × 8

秒杀系统案例：
- QPS：40,000
- 响应大小：500 字节

带宽 = 40,000 × 500B × 8 = 160 Mbps
```

### 3.4 服务器估算

**公式**：

```
服务器数 = 峰值 QPS / 单机 QPS × 冗余系数

秒杀系统案例：
- 峰值 QPS：40,000
- 单机 QPS：6711（实测）
- 冗余系数：1.5

服务器数 = 40,000 / 6711 × 1.5 = 9 台
```

## 四、第 3 步：高层架构（10 分钟）

### 4.1 画架构图

**秒杀系统架构图**：

```
用户
  ↓
CDN（静态资源）
  ↓
Nginx（限流 100 req/s/IP）
  ↓
Gateway（限流 5000 QPS）
  ↓
┌──────────────┬──────────────┬──────────────┐
│ Seckill      │ Order        │ Payment      │
│ Service      │ Service      │ Service      │
│              │              │              │
│ Redis Lua    │ Kafka 消费   │ 支付回调     │
│ 原子扣库存   │ 创建订单     │              │
└──────┬───────┴──────┬───────┴──────┬───────┘
       │              │              │
   ┌───▼───┐      ┌───▼───┐      ┌──▼──┐
   │ Redis │      │ MySQL │      │ MQ  │
   └───────┘      └───────┘      └─────┘
```

### 4.2 说明架构特点

**模板**：

```
你："我设计了一个四层过滤架构：
1. CDN + 前端：缓存静态资源，减少 50% 请求
2. Gateway + Sentinel：限流熔断，过滤 80% 无效请求
3. Redis Lua：原子扣库存，过滤 99% 请求
4. Kafka + MySQL：异步削峰，保护数据库

10 万请求 → 2 万 → 100 个 → 最终 100 个订单"
```

### 4.3 说明技术选型

| 组件 | 技术 | 原因 |
|------|------|------|
| 网关 | Spring Cloud Gateway | 限流、路由 |
| 缓存 | Redis | 高性能、原子操作 |
| 消息队列 | Kafka | 异步削峰 |
| 数据库 | MySQL | 数据持久化 |
| 注册中心 | Nacos | 服务发现 |

## 五、第 4 步：深入设计（20 分钟）

### 5.1 数据模型设计

**活动表**：

```sql
CREATE TABLE seckill_activity (
  id BIGINT PRIMARY KEY,
  name VARCHAR(100),
  stock INT,
  remain_stock INT,
  price DECIMAL(10,2),
  start_time DATETIME,
  end_time DATETIME,
  INDEX idx_start_time (start_time)
);
```

**订单表**：

```sql
CREATE TABLE seckill_order (
  id BIGINT PRIMARY KEY,
  user_id BIGINT,
  activity_id BIGINT,
  order_no VARCHAR(32),
  status TINYINT,
  create_time DATETIME,
  UNIQUE KEY uk_activity_user (activity_id, user_id),
  INDEX idx_user_time (user_id, create_time)
);
```

### 5.2 接口设计

**秒杀下单**：

```
POST /api/v1/seckill/{activityId}
Header: X-User-Id: 10001

Response:
{
  "code": 200,
  "message": "下单成功",
  "data": {
    "orderNo": "SK_1714204800001"
  }
}
```

**查询结果**：

```
GET /api/v1/seckill/result/{orderNo}

Response:
{
  "code": 200,
  "data": {
    "orderNo": "SK_1714204800001",
    "status": "PENDING",
    "activityName": "iPhone 15 秒杀",
    "price": 3999.00
  }
}
```

### 5.3 核心流程设计

**秒杀流程**：

```
1. 用户点击秒杀按钮
   ↓
2. Gateway 限流（5000 QPS）
   ↓
3. 频率限制（Redis SETNX，1 req/s/用户）
   ↓
4. 布隆过滤器（快速判断是否已购买）
   ↓
5. Redis Lua 原子扣库存
   ↓
6. 发送 Kafka 消息（异步创建订单）
   ↓
7. 返回"下单成功"
   ↓
8. Kafka 消费者创建订单
   ↓
9. 用户查询订单状态
```

### 5.4 防超卖设计

**三道防线**：

```
第 1 道：Redis Lua 原子扣减
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock <= 0 then return 0 end
redis.call('DECR', KEYS[1])
return 1

第 2 道：MySQL 乐观锁
UPDATE seckill_activity 
SET remain_stock = remain_stock - 1 
WHERE id = ? AND remain_stock > 0;

第 3 道：唯一索引
UNIQUE KEY uk_activity_user (activity_id, user_id)
```

### 5.5 高可用设计

**冗余**：
- 应用服务器：3 台
- MySQL：1 主 2 从
- Redis：Cluster（3 主 3 从）
- Kafka：3 节点

**限流熔断降级**：
- Nginx 限流：100 req/s/IP
- Gateway 限流：5000 QPS
- 熔断：失败率 > 50% 打开
- 降级：关闭非核心功能

### 5.6 扩展性设计

**水平扩展**：
- 应用服务器：无状态设计，支持水平扩展
- 数据库：分库分表（按用户 ID Hash）
- 缓存：Redis Cluster
- 消息队列：Kafka Partition

**扩容步骤**：
1. 启动新服务器
2. 注册到 Nacos
3. 加入负载均衡
4. 验证流量

## 六、常见追问及应对

### 6.1 性能相关

| 追问 | 应对 |
|------|------|
| "如果 QPS 从 5000 增长到 50000 怎么办？" | "水平扩展应用服务器，从 3 台增加到 30 台" |
| "如何优化延迟？" | "Redis 缓存热点数据、Kafka 异步削峰、CDN 加���" |
| "如何提升吞吐量？" | "增加 Kafka Partition、数据库分库分表" |

### 6.2 可靠性相关

| 追问 | 应对 |
|------|------|
| "如果 Redis 宕机怎么办？" | "Redis Cluster 自动故障转移，或降级到 MySQL" |

| "如何防止消息丢失？" | "Kafka acks=1、手动提交、重试机制" |

### 6.3 安全相关

| 追问 | 应对 |
|------|------|
| "如何防止恶意刷单？" | "IP 限流、用户限购、验证码、风控系统" |
| "如何防止超卖？" | "Redis Lua 原子扣减 + MySQL 乐观锁 + 唯一索引" |
| "如何防止重复购买？" | "布隆过滤器 + 唯一索引" |

### 6.4 扩展相关

| 追问 | 应对 |
|------|------|
| "如何从 1 台扩展到 100 台？" | "无状态设计，水平扩展" |
| "如何从 1 亿数据扩展到 10 亿？" | "分库分表，按用户 ID Hash 分片" |
| "如何保证扩容不停机？" | "灰度发布，逐步切换流量" |

## 九、常见系统设计题

### 9.1 秒杀系统

**核心挑战**：高并发、防超卖、异步削峰

**关键设计**：
- 四层过滤（CDN → Gateway → Redis → Kafka）
- Redis Lua 原子扣库存
- Kafka 异步创建订单
- 三道防线防超卖

### 9.2 短链系统

**核心挑战**：高并发读、海量存储、短链生成

**关键设计**：
- 短链生成（Base62 编码 + 自增 ID）
- Redis 缓存热点短链
- 分库分表（按 Hash 分片）
- 读写分离（读多写少）

### 9.3 Feed 流系统

**核心挑战**：海量数据、实时性、个性化

**关键设计**：
- 推模式 vs 拉模式
- 时间线存储（Redis ZSet）
- 分页查询（游标）
- 缓存预热

### 9.4 分布式缓存

**核心挑战**：高性能、高可用、数据一致性

**关键设计**：
- 一致性 Hash
- 主从复制
- 哨兵 / Cluster
- 缓存穿透 / 击穿 / 雪崩

### 9.5 文件存储系统

**核心挑战**：海量存储、高并发、数据安全

**关键设计**：
- 分布式存储（HDFS / Ceph）
- 元数据管理（MySQL / MongoDB）
- CDN 加速
- 数据备份 / 恢复

## 十、面试评分标准

| 维度 | 优秀 | 良好 | 一般 | 较差 |

| **沟通表达** | 逻辑清晰，表达流畅 | 表达清楚 | 表达不够清晰 | 表达混乱 |

## 十二、总结

系统设计面试考察的是综合能力，需要系统性准备。

**核心要点**：
1. **4 步法**：需求澄清 → 容量估算 → 高层架构 → 深入设计
2. **时间分配**：20% + 10% + 25% + 45%
3. **沟通表达**：边画图边讲解，逻辑清晰
4. **权衡取舍**：没有完美方案，只有合适方案
5. **实战经验**：基于真实项目（秒杀系统 QPS 6711）

**面试心态**：
- 不要紧张，把面试官当成合作伙伴
- 不要追求完美，先把核心设计讲清楚
- 不要害怕追问，追问是好事，说明面试官感兴趣
- 不要放弃，即使卡壳也要继续，展示思考过程

**最后建议**：
- 多练习，至少练习 10 个经典题目
- 多总结，每次练习后总结不足
- 多复盘，看看别人的设计方案
- 多实战，基于真实项目积累经验

祝你面试顺利！🎉
