# 系统设计模拟面试

> 2 场完整的系统设计模拟面试，展示 4 步法的实际应用

## 模拟面试 1：设计短链系统

### 面试官："请设计一个短链系统，类似 bit.ly"

### 第 1 步：需求澄清（5 分钟）

**我**："好的，我先确认一下需求。

**功能需求**：
1. 核心功能是生成短链和短链跳转，对吗？
2. 是否需要统计点击数据？
3. 是否需要支持自定义短链？
4. 是否需要设置过期时间？

**非功能需求**：
1. 预计有多少用户？每天生成多少短链？
2. 峰值 QPS 大概多少？
3. 短链需要保存多久？
4. 对延迟有什么要求？"

**面试官**："日活 1000 万，每天生成 100 万个短链，保存 5 年，P99 延迟 < 100ms。"

### 第 2 步：容量估算（5 分钟）

**我**："让我粗算一下容量。

**QPS 估算**：
- 生成短链：100 万 / 86400 ≈ 12 QPS
- 点击短链：假设读写比 20:1，约 240 QPS
- 峰值 QPS：240 × 3 = 720 QPS

**存储估算**：
- 单条记录：300 字节（短码 + 长链 + 元数据）
- 5 年数据：100 万 × 365 × 5 = 18 亿条
- 存储容量：300B × 18 亿 × 2（副本）× 1.3（增长）= 1.4 TB

**服务器估算**：
- 应用服务器：720 / 5000 = 1 台（考虑冗余 2 台）
- 数据库：需要分库分表，12 个 MySQL 实例
- 缓存：Redis Cluster，14 个节点（7 主 7 从）"

### 第 3 步：高层架构（10 分钟）

**我**："我设计一个三层架构。"

```
用户
  ↓
CDN（静态资源）
  ↓
Nginx（负载均衡）
  ↓
┌──────────────┬──────────────┐
│   生成短链   │   短链跳转   │
│              │              │
│  1. 生成ID   │  1. 查Redis  │
│  2. Base62   │  2. 查MySQL  │
│  3. 存MySQL  │  3. 302跳转  │
│  4. 存Redis  │  4. 异步统计 │
└──────┬───────┴──────┬───────┘
       │              │
   ┌───▼───┐      ┌───▼───┐
   │ MySQL │      │ Redis │
   │ 分库  │      │Cluster│
   │ 分表  │      │       │
   └───────┘      └───────┘
```

**技术选型**：
- 短链生成：Snowflake ID + Base62 编码
- 数据库：MySQL 分库分表（12 个库 × 16 张表）
- 缓存：Redis Cluster（热点数据）
- 负载均衡：Nginx

### 第 4 步：深入设计（20 分钟）

**我**："接下来详细设计核心模块。

**1. 短链生成算法**：
```java
// Snowflake ID + Base62 编码
long id = snowflake.nextId();  // 1234567890
String shortCode = Base62.encode(id);  // "abc123"
```

**2. 数据模型**：
```sql
CREATE TABLE short_url (
  id BIGINT PRIMARY KEY,
  short_code VARCHAR(8) NOT NULL,
  long_url VARCHAR(2048) NOT NULL,
  url_md5 VARCHAR(32) NOT NULL,
  click_count INT DEFAULT 0,
  create_time DATETIME,
  UNIQUE KEY uk_short_code (short_code),
  INDEX idx_url_md5 (url_md5)
);

-- 分库分表：按 short_code Hash 分片
-- 12 个库 × 16 张表 = 192 张表
```

**3. 缓存策略**：
```java
// 热点数据缓存（20% 短链占 80% 访问）
String longUrl = redisTemplate.opsForValue().get("short:" + shortCode);
if (longUrl == null) {
    longUrl = shortUrlService.getByShortCode(shortCode);
    redisTemplate.opsForValue().set("short:" + shortCode, longUrl, 7, TimeUnit.DAYS);
}
```

**4. 高可用设计**：
- MySQL：1 主 2 从，自动故障转移
- Redis：Cluster 模式，7 主 7 从
- 应用服务器：2 台，负载均衡

**5. 扩展性设计**：
- 应用服务器：无状态，支持水平扩展
- 数据库：分库分表，翻倍扩容
- 缓存：Redis Cluster，动态扩容"

### 总结

**面试官**："很好，如果 QPS 增长到 10000 怎么办？"

**我**："可以通过以下方式优化：
1. 增加应用服务器（2 台 → 10 台）
2. 数据库读写分离（1 主 2 从 → 1 主 4 从）
3. Redis Cluster 扩容（14 节点 → 28 节点）
4. CDN 加速静态资源

预计可以支持 QPS 10000+。"

## 模拟面试 2：设计秒杀系统

### 面试官："请设计一个秒杀系统"

### 第 1 步：需求澄清（5 分钟）

**我**："好的，我先确认一下需求。

**功能需求**：
1. 核心功能是用户下单、支付、订单查询，对吗？
2. 是否需要支持多个活动并发？
3. 是否需要用户限购？
4. 是否需要活动预热？

**非功能需求**：
1. 预计有多少用户参与？
2. 峰值 QPS 大概多少？
3. 库存有多少件？
4. 对延迟和可用性有什么要求？"

**面试官**："10 万用户，库存 100 件，峰值 QPS 5000，P99 < 200ms，99.9% SLA。"

### 第 2 步：容量估算（5 分钟）

**我**："让我粗算一下。

**QPS 估算**：
- 假设前 10 秒集中 80% 请求
- 瞬时 QPS = 10 万 × 5 × 0.8 / 10 = 40000 QPS

**存储估算**：
- 成功订单：100 条（库存）
- 单条记录：100 字节
- 总存储：10 KB（极小）

**服务器估算**：
- 应用服务器：40000 / 6711（实测）= 6 台
- 考虑冗余：6 × 1.5 = 9 台"

### 第 3 步：高层架构（10 分钟）

**我**："我设计一个四层过滤架构。"

```
用户（10 万请求）
  ↓
CDN + 前端（过滤 50%）→ 5 万请求
  ↓
Gateway + Sentinel（限流 5000 QPS）→ 5000 请求
  ↓
Redis Lua（原子扣库存）→ 100 个请求
  ↓
Kafka + MySQL（异步创建订单）→ 100 个订单
```

**核心设计**：
- 四层过滤：CDN → Gateway → Redis → Kafka
- Redis Lua：原子扣库存
- Kafka：异步削峰
- 三道防线：Redis + MySQL 乐观锁 + 唯一索引

### 第 4 步：深入设计（20 分钟）

**我**："详细设计核心流程。

**1. 秒杀流程**：
```java
@PostMapping("/seckill/{activityId}")
public Result<String> seckill(@PathVariable Long activityId, 
                               @RequestHeader("X-User-Id") Long userId) {
    // 1. 限流（Gateway 5000 QPS）
    // 2. 频率限制（Redis SETNX，1 req/s/用户）
    // 3. 布隆过滤器（快速判断是否已购买）
    // 4. Redis Lua 原子扣库存
    Long result = redisTemplate.execute(luaScript,

        return Result.fail("库存不足");
    }

    kafkaTemplate.send("seckill-order", message);
    
    return Result.success("下单成功");
}
```

**2. 防超卖设计（三道防线）**：
```java
// 第 1 道：Redis Lua 原子扣减
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock <= 0 then return 0 end
redis.call('DECR', KEYS[1])
return 1

// 第 2 道：MySQL 乐观锁
UPDATE seckill_activity 
SET remain_stock = remain_stock - 1 
WHERE id = ? AND remain_stock > 0;

// 第 3 道：唯一索引
UNIQUE KEY uk_activity_user (activity_id, user_id)
```

**3. 高可用设计**：
- 应用服务器：9 台，负载均衡
- MySQL：1 主 2 从，自动故障转移
- Redis：Cluster 模式，3 主 3 从
- Kafka：3 节点，副本机制

**4. 性能优化**：
- 连接池预热
- JVM 参数调优（G1 GC）
- 异步化（Kafka 削峰）
- 缓存预热"

### 总结

**面试官**："很好，如果要优化到 QPS 30000 怎么办？"

**我**："基于我的实战经验（w09-seckill 项目 QPS 6711），可以分三阶段优化：

**短期（1 周，¥0）**：
- JVM 调优 → QPS 8000

**中期（1 月，8 台）**：
- 分机部署 → QPS 15000

**长期（3 月，28 台）**：
- K8s 集群 → QPS 30000+

具体方案包括：
1. 服务分机部署（消除资源竞争）
2. Redis Cluster 扩容
3. 数据库读写分离
4. CDN 加速
5. 异步化优化"

## 面试技巧总结

### 1. 时间分配

- 需求澄清：5-10 分钟

- 高层架构：10 分钟

- **先整体后局部**：先讲架构，再讲细节

- **真实数据**：基于项目经验（QPS 6711）
