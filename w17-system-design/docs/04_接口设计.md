# 接口设计

> 系统设计第四步：把数据模型转化为 RESTful API

## 一、为什么接口设计很重要？

接口是系统对外的契约，设计不好会导致：
1. **难以理解**：接口命名混乱，参数不清晰
2. **难以维护**：版本混乱，兼容性差
3. **安全问题**：缺少鉴权、限流、签名
4. **性能问题**：接口粒度不合理，N+1 查询

**好的接口设计**：
- 符合 RESTful 规范
- 命名清晰、语义明确
- 幂等性保证
- 版本控制
- 安全可靠

## 二、RESTful API 设计规范

### 2.1 HTTP 方法

| 方法 | 语义 | 幂等性 | 安全性 | 示例 |
|------|------|--------|--------|------|
| GET | 查询 | ✓ | ✓ | GET /api/activities/1 |
| POST | 创建 | ✗ | ✗ | POST /api/orders |
| PUT | 更新（全量） | ✓ | ✗ | PUT /api/orders/1 |
| PATCH | 更新（部分） | ✗ | ✗ | PATCH /api/orders/1 |
| DELETE | 删除 | ✓ | ✗ | DELETE /api/orders/1 |

**幂等性**：多次调用结果相同
**安全性**：不会修改数据

### 2.2 URL 设计规范

**规范 1：使用名词复数**

```
✓ GET /api/activities        # 查询活动列表
✓ GET /api/activities/1      # 查询单个活动
✗ GET /api/getActivity       # 不要用动词
✗ GET /api/activity          # 不要用单数
```

**规范 2：使用小写 + 连字符**

```
✓ GET /api/seckill-activities
✗ GET /api/SeckillActivities  # 不要用驼峰
✗ GET /api/seckill_activities # 不要用下划线
```

**规范 3：资源嵌套不超过 2 层**

```
✓ GET /api/activities/1/orders           # 查询活动的订单
✗ GET /api/users/1/orders/2/payments/3  # 嵌套太深
```

**规范 4：使用查询参数过滤**

```
✓ GET /api/orders?status=PENDING&page=1&size=10
✗ GET /api/orders/pending/page/1/size/10
```

### 2.3 状态码规范

| 状态码 | 含义 | 使用场景 |
|--------|------|---------|
| 200 | OK | 成功 |
| 201 | Created | 创建成功 |
| 204 | No Content | 删除成功 |
| 400 | Bad Request | 参数错误 |
| 401 | Unauthorized | 未认证 |
| 403 | Forbidden | 无权限 |
| 404 | Not Found | 资源不存在 |
| 409 | Conflict | 资源冲突（如重复购买） |
| 429 | Too Many Requests | 限流 |
| 500 | Internal Server Error | 服务器错误 |
| 503 | Service Unavailable | 服务不可用 |

## 三、接口幂等性设计

### 3.1 什么是幂等性？

**定义**：多次调用产生的结果与单次调用相同

**为什么需要幂等性？**
- 网络超时重试
- 用户重复点击
- 消息队列重复消费

### 3.2 幂等性实现方案

**方案 1：唯一索引**

```sql
-- 订单表：(activity_id, user_id) 唯一索引
CREATE UNIQUE INDEX uk_activity_user ON seckill_order(activity_id, user_id);

-- 第一次插入：成功
INSERT INTO seckill_order (activity_id, user_id, ...) VALUES (1, 10001, ...);

-- 第二次插入：失败（唯一索引冲突）
INSERT INTO seckill_order (activity_id, user_id, ...) VALUES (1, 10001, ...);
-- ERROR 1062: Duplicate entry '1-10001' for key 'uk_activity_user'
```

**方案 2：Token 机制**

```java
// 1. 客户端请求 Token
GET /api/seckill/token?activityId=1
Response: {"token": "abc123"}

// 2. 客户端携带 Token 下单
POST /api/seckill/1
Header: X-Idempotent-Token: abc123

// 3. 服务端验证 Token（Redis SETNX）
String key = "idempotent:token:" + token;
Boolean success = redisTemplate.opsForValue().setIfAbsent(key, "1", 5, TimeUnit.MINUTES);
if (!success) {
    throw new DuplicateRequestException("重复请求");
}
```

**方案 3：状态机**

```java
// 订单状态：PENDING → PAID → COMPLETED
// 只有 PENDING 状态才能支付

UPDATE seckill_order 
SET status = 'PAID' 
WHERE id = ? AND status = 'PENDING';

// 第一次：更新成功（affectedRows = 1）
// 第二次：更新失败（affectedRows = 0，因为状态已经是 PAID）
```

**方案 4：分布式锁**

```java
String lockKey = "seckill:lock:" + activityId + ":" + userId;
RLock lock = redissonClient.getLock(lockKey);

try {
    // 尝试加锁（等待 0 秒，锁定 10 秒）
    boolean locked = lock.tryLock(0, 10, TimeUnit.SECONDS);
    if (!locked) {
        throw new DuplicateRequestException("重复请求");
    }
    
    // 执行业务逻辑
    createOrder(activityId, userId);
} finally {
    lock.unlock();
}
```

### 3.3 秒杀系统幂等性设计

```java
@PostMapping("/seckill/{activityId}")
public Result<String> seckill(@PathVariable Long activityId, 
                               @RequestHeader("X-User-Id") Long userId) {

    if (bloomFilter.mightContain(activityId + ":" + userId)) {

    }
    
    // 3. 发送 Kafka 消息（异步创建订单）
    kafkaTemplate.send("seckill-order", message);
    
    // 4. 添加到布隆过滤器
    bloomFilter.put(activityId + ":" + userId);
    
    return Result.success("下单成功");
}
```

## 四、版本控制策略

### 4.1 为什么需要版本控制？

- API 需要演进（新增字段、修改逻辑）
- 老版本客户端需要兼容
- 避免破坏性变更

### 4.2 版本控制方案

**方案 1：URL 路径版本**

```
GET /api/v1/activities
GET /api/v2/activities
```

**优点**：清晰、易于理解
**缺点**：URL 变化，需要维护多个版本

**方案 2：请求头版本**

```
GET /api/activities
Header: API-Version: 1
```

**优点**：URL 不变
**缺点**：不够直观

**方案 3：查询参数版本**

```
GET /api/activities?version=1
```

**优点**：灵活
**缺点**：容易被忽略

**推荐**：使用 URL 路径版本（方案 1）

### 4.3 版本演进策略

**向后兼容的变更**（不需要升级版本）：
- 新增字段
- 新增接口
- 新增可选参数

**破坏性变更**（需要升级版本）：
- 删除字段
- 修改字段类型
- 修改接口语义

**示例**：

```java
// v1：返回价格（单位：元）
GET /api/v1/activities/1
{
  "price": 3999.00
}

// v2：返回价格（单位：分，破坏性变更）
GET /api/v2/activities/1
{
  "price": 399900
}
```

**示例**：

| 100001 | 参数错误 | 400 |
| 100002 | 活动不存在 | 404 |
| 100003 | 库存不足 | 409 |
| 100004 | 重复购买 | 409 |
| 100005 | 活动未开始 | 400 |
| 100006 | 活动已结束 | 400 |
| 200001 | 订单不存在 | 404 |
| 200002 | 订单已支付 | 409 |

- 10：秒杀活动

**错误类型**：
- 00：参数错误

## 六、限流 / 鉴权 / 签名

### 6.1 限流

**目的**：防止恶意刷单、保护系统

**方案 1：IP 限流**

```java
@RateLimiter(key = "#request.remoteAddr", rate = 100, duration = 1, unit = TimeUnit.SECONDS)
@GetMapping("/activities")
public Result<List<Activity>> list(HttpServletRequest request) {
    // ...
}
```

**方案 2：用户限流**

```java
@RateLimiter(key = "#userId", rate = 1, duration = 1, unit = TimeUnit.SECONDS)
@PostMapping("/seckill/{activityId}")
public Result<String> seckill(@PathVariable Long activityId, 
                               @RequestHeader("X-User-Id") Long userId) {
    // ...
}
```

**方案 3：接口限流**

```java
@RateLimiter(key = "'seckill'", rate = 5000, duration = 1, unit = TimeUnit.SECONDS)
@PostMapping("/seckill/{activityId}")
public Result<String> seckill(...) {
    // ...
}
```

### 6.2 鉴权

**方案 1：Token 鉴权（JWT）**

```java
// 1. 登录获取 Token
POST /api/auth/login
{
  "username": "user1",
  "password": "123456"
}

Response:
{
  "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."
}

// 2. 携带 Token 访问接口
GET /api/orders
Header: Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

**方案 2：Session 鉴权**

```java
// 1. 登录创建 Session
POST /api/auth/login
Response: Set-Cookie: JSESSIONID=abc123

// 2. 携带 Cookie 访问接口
GET /api/orders
Cookie: JSESSIONID=abc123
```

### 6.3 签名

**目的**：防止参数篡改

**签名算法**：

```java
// 1. 参数排序
Map<String, String> params = new TreeMap<>();
params.put("activityId", "1");
params.put("userId", "10001");
params.put("timestamp", "1714204800000");

// 2. 拼接字符串
String str = "activityId=1&timestamp=1714204800000&userId=10001";

// 3. 加上密钥
String signStr = str + "&key=secret123";

// 4. MD5 签名
String sign = MD5(signStr); // "abc123def456..."

// 5. 请求携带签名
POST /api/seckill/1?userId=10001&timestamp=1714204800000&sign=abc123def456...
```

**服务端验证**：

```java
// 1. 获取参数
Map<String, String> params = request.getParameterMap();
String clientSign = params.remove("sign");

// 2. 重新计算签名
String serverSign = calculateSign(params);

// 3. 比对签名
if (!serverSign.equals(clientSign)) {
    throw new SignatureException("签名错误");
}
```
