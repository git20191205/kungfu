LocalMessageTableDemo 日志分析

运行环境：JDK 1.8.0_441, Windows 11, MySQL localhost:3306

整体结果：本地消息表完整流程演示成功 — 下单+写消息同事务 → 定时扫描 → 发送 → 幂等消费。

重点观察 1 — 同事务保证原子性：

  日志原文：
    1. INSERT t_order_d35: ORD_1745115783615
    2. INSERT t_local_msg: msgId=uuid
    3. COMMIT → 订单和消息在同一个事务中提交

  分析：
    - 订单表和消息表在同一个 MySQL 实例
    - 用同一个 Connection + 同一个事务
    - COMMIT 要么两个都成功，要么两个都回滚
    - 这就是本地消息表的核心：用本地事务保证「业务+消息」的原子性

    对比 RocketMQ 事务消息：
    - RocketMQ: 半消息 → 执行业务 → 确认/回滚
    - 本地消息表: 业务+消息同事务 → 定时扫描发送
    - 本地消息表更简单，不依赖 MQ 的事务特性

重点观察 2 — 定时扫描机制：

  日志原文：
    扫描 PENDING 消息：
    msgId=uuid
    body={"orderNo":"ORD_xxx","action":"DEDUCT_STOCK",...}
    → 发送消息... ✓ 成功
    → 更新状态: PENDING → SENT

  分析：
    生产环境中的定时扫描：
    - 频率：每 5-10 秒扫描一次
    - 条件：status='PENDING' AND retry_count < max_retry
    - 发送失败：retry_count + 1，下次继续扫描
    - 超过最大重试：标记为 FAILED，人工介入

    消息表膨胀问题：
    - SENT/CONSUMED 状态的消息定期归档或删除
    - 保留最近 7 天的消息用于排查
    - 历史消息归档到冷存储

重点观察 3 — 幂等消费的必要性：

  为什么消息可能重复？
  1. 定时扫描发送成功，但更新状态前宕机 → 下次重复发送
  2. MQ 投递成功，但消费者 ACK 前宕机 → MQ 重投
  3. 网络超时，生产者不确定是否成功 → 重试发送

  幂等方案：
  - 用 msg_id 做唯一键（最简单）
  - 业务层面幂等（如：扣库存前检查是否已扣过）
  - 数据库唯一索引兜底

结论：
  1. 本地消息表是最简单的分布式事务方案，不依赖外部组件
  2. 核心原理：业务操作 + 消息写入在同一个本地事务中
  3. 定时扫描保证消息最终被发送（最终一致性）
  4. 消费端必须做幂等处理（消息可能重复）
  5. 适用场景：跨服务调用、异步通知、事件驱动
