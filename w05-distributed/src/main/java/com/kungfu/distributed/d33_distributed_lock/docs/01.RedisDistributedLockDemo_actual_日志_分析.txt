RedisDistributedLockDemo 日志分析

运行环境：JDK 1.8.0_441, Windows 11, Redis localhost:6379

整体结果：成功演示无锁超卖 vs 有锁安全扣减的对比。

重点观察 1 — 无锁并发的超卖问题：

  日志原文：
    初始库存: 100
    并发请求: 200 个线程同时扣减
    成功扣减: 200 次
    最终库存: -100

  分析：
    - 200 个线程同时读到 stock > 0 → 全部执行 DECR
    - 经典的 check-then-act 竞态条件
    - 100 个库存被扣了 200 次 → 超卖 100 个
    - 这就是为什么需要分布式锁

重点观察 2 — 有锁并发的安全扣减：

  日志原文：
    初始库存: 100
    并发请求: 200 个线程同时扣减
    成功扣减: 100 次
    最终库存: 0
    → ✓ 无超卖

  分析：
    - 200 个线程竞争同一把锁
    - 同一时刻只有 1 个线程能执行 check + DECR
    - 前 100 个获得锁的线程成功扣减
    - 后 100 个获得锁时发现 stock=0 → 不扣减
    - 最终库存精确为 0

    性能代价：
    - 有锁比无锁慢（串行化执行）
    - 但正确性 > 性能（超卖是不可接受的）

重点观察 3 — Lua 脚本解锁的必要性：

  如果不用 Lua，解锁代码是：
    String val = jedis.get(lockKey);
    if (myUuid.equals(val)) {  // ← 这里可能被其他线程抢占
        jedis.del(lockKey);     // ← 删掉的可能是别人的锁！
    }

  GET 和 DEL 之间不是原子的 → 可能误删别人的锁
  Lua 脚本在 Redis 中原子执行 → 安全

结论：
  1. 无锁并发必然超卖（check-then-act 竞态）
  2. Redis 分布式锁解决了多实例部署下的并发安全
  3. 加锁用 SET NX EX（互斥 + 防死锁）
  4. 解锁用 Lua 脚本（原子判断 + 删除，防误删）
