CAPAndBASEDemo 日志分析

整体结果：成功演示 CAP 三选二的 CP/AP 决策差异 + BASE 最终一致性模拟。

逐步分析：

  ┌──────────────────────────┬──────────────────────────────────────┬────────┐
  │ 模块                     │ 预期结果                             │ 状态   │
  ├──────────────────────────┼──────────────────────────────────────┼────────┤
  │ CAP 理论表格             │ 三维度 + 三选二 + 代表系统           │ ✅     │
  │ CP 系统模拟              │ Node-B 分区 → 写入失败 → 回滚       │ ✅     │

  │ 最终一致性模拟           │ 异步同步 T1→T2→T3 达到一致          │ ✅     │
  └──────────────────────────┴──────────────────────────────────────┴────────┘

重点观察 1 — CP vs AP 的决策差异：

  CP 系统（如 ZooKeeper）：
    - 3 节点写入，Node-B 超时 → 整体写入失败
    - 客户端收到错误，需要重试
    - 代价：可用性下降（分区期间无法写入）
    - 收益：任何时刻读任何节点都是一致的

  AP 系统（如 Eureka）：
    - 3 节点写入，Node-B 超时 → 其他节点成功即返回
    - 客户端立即收到成功响应
    - 代价：读 Node-B 可能读到旧数据
    - 收益：系统始终可用，不会因为部分节点故障而拒绝服务

  ★ 面试关键洞察：
    不是「选了 AP 就完全没有一致性」，而是「暂时不一致，最终会一致」
    不是「选了 CP 就完全不可用」，而是「分区期间不可用，分区恢复后立即可用」

重点观察 2 — 最终一致性的时间窗口：

  日志原文：
    T1: 写入 Node-A = 42 → 状态: A=42, B=0, C=0 ← 不一致！
    T2: 异步同步 → Node-B = 42 → 状态: A=42, B=42, C=0
    T3: 异步同步 → Node-C = 42 → 状态: A=42, B=42, C=42 ← 最终一致！

  分析：
    - T1 到 T3 之间存在「不一致窗口」
    - 在 Demo 中是 200ms + 400ms（模拟延迟）
    - 生产环境中这个窗口取决于：网络延迟 + 同步机制 + 队列积压
    - Redis 主从同步：通常 < 1ms
    - 跨机房同步：通常 10-50ms
    - 消息队列最终一致：通常 100ms-几秒

  ★ 设计决策：
    如果业务能容忍几百毫秒的不一致 → 选 AP + 最终一致（大多数互联网场景）
    如果业务不能容忍任何不一致 → 选 CP + 强一致（金融转账）

重点观察 3 — 实际系统的 CAP 选择：

  ┌──────────────────┬──────┬──────────────────────────────────────┐
  │ 系统             │ 选择 │ 原因                                  │
  ├──────────────────┼──────┼──────────────────────────────────────┤
  │ ZooKeeper        │ CP   │ 配置中心/注册中心需要强一致           │
  │ Etcd             │ CP   │ K8s 依赖它存储集群状态               │
  │ Eureka           │ AP   │ 服务发现允许短暂不一致               │
  │ Nacos            │ AP/CP│ 可配置：临时实例AP，持久实例CP        │
  │ Redis Cluster    │ AP   │ 异步复制，主从切换可能丢数据         │
  │ MySQL 主从       │ AP   │ 异步/半同步复制                       │
  │ MySQL Group Repl │ CP   │ Paxos 协议保证强一致                  │
  └──────────────────┴──────┴──────────────────────────────────────┘

结论：
  1. CAP 是分布式系统设计的第一性原理，所有架构决策都基于此
  2. P 必须保证 → 实际是 CP 和 AP 的权衡
  3. BASE 是 AP 系统的理论基础：接受暂时不一致，追求最终一致
  4. 大多数互联网系统选 AP（可用性优先），金融系统选 CP（一致性优先）
