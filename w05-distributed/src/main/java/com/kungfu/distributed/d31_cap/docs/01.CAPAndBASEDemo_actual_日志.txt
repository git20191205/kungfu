========================================
  CAP/BASE 理论
========================================

=== 一、CAP 定理 ===

  CAP = Consistency + Availability + Partition tolerance

  ┌──────────────────┬──────────────────────────────────────────────────┐
  │ 维度             │ 含义                                              │
  ├──────────────────┼──────────────────────────────────────────────────┤
  │ C (一致性)       │ 所有节点在同一时刻看到相同的数据                  │
  │ A (可用性)       │ 每个请求都能在合理时间内收到非错误响应            │
  │ P (分区容错)     │ 网络分区发生时系统仍能继续运行                    │
  └──────────────────┴──────────────────────────────────────────────────┘

  为什么只能三选二？
    网络分区（P）在分布式系统中必然发生 → P 必须保证
    → 实际选择是 CP 还是 AP

  ┌──────┬──────────────────────────┬──────────────────────────────────┐
  │ 选择 │ 代表系统                 │ 特点                              │
  ├──────┼──────────────────────────┼──────────────────────────────────┤
  │ CP   │ ZooKeeper, Etcd, HBase   │ 分区时拒绝写入，保证数据一致     │
  │ AP   │ Eureka, Cassandra, DNS   │ 分区时继续服务，允许数据不一致   │
  │ CA   │ 单机 MySQL（非分布式）   │ 不存在网络分区 → 不是分布式系统  │
  └──────┴──────────────────────────┴──────────────────────────────────┘

=== 二、模拟 CP 系统（写入需所有节点确认）===

  场景：3 节点集群，Node-B 发生网络分区
  CP 策略：写入需要所有节点确认

  写入 key=order_001, value=created...
    Node-A: ✓ 写入成功
    Node-B: ✗ 网络超时（分区中）
    Node-C: ✓ 写入成功

  → CP 决策：写入失败！回滚所有节点
  → 保证了一致性（C），但牺牲了可用性（A）
  → 客户端收到错误响应，需要重试

=== 三、模拟 AP 系统（写入只需一个节点成功）===

  场景：同样 3 节点集群，Node-B 网络分区
  AP 策略：写入只需一个节点成功即返回

  写入 key=order_001, value=created...
    Node-A: ✓ 写入成功
    Node-B: ✗ 网络超时（分区中，稍后异步同步）
    Node-C: ✓ 写入成功

  → AP 决策：写入成功！立即返回客户端
  → 保证了可用性（A），但 Node-B 数据暂时不一致
  → 分区恢复后，Node-B 通过异步同步达到最终一致

  读取对比：
    读 Node-A: order_001 = created  ✓
    读 Node-B: order_001 = (不存在) ✗ ← 数据不一致！
    → 这就是 AP 系统的代价

=== 四、BASE 理论 ===

  BASE = Basically Available + Soft state + Eventually consistent

  ┌──────────────────┬──────────────────────────────────────────────────┐
  │ 维度             │ 含义                                              │
  ├──────────────────┼──────────────────────────────────────────────────┤
  │ 基本可用         │ 允许部分功能降级（如返回缓存数据、排队等待）      │
  │ 软状态           │ 允许中间状态存在（如订单「支付中」状态）          │
  │ 最终一致         │ 经过一段时间后，所有节点数据最终一致              │
  └──────────────────┴──────────────────────────────────────────────────┘

  ACID vs BASE：
  ┌──────────┬──────────────────────┬──────────────────────┐
  │          │ ACID                 │ BASE                 │
  ├──────────┼──────────────────────┼──────────────────────┤
  │ 一致性   │ 强一致               │ 最终一致             │
  │ 可用性   │ 可能阻塞             │ 基本可用             │
  │ 适用场景 │ 银行转账             │ 电商下单             │
  │ 代表     │ MySQL 事务           │ 分布式消息队列       │
  └──────────┴──────────────────────┴──────────────────────┘

=== 五、模拟最终一致性 ===

  初始状态: A=0, B=0, C=0
  T1: 写入 Node-A = 42
      状态: A=42, B=0, C=0 ← 不一致！

  T2: 异步同步 → Node-B = 42
      状态: A=42, B=42, C=0 ← 仍不一致
  T3: 异步同步 → Node-C = 42
      状态: A=42, B=42, C=42 ← 最终一致！✓

  → 经过短暂的不一致窗口，所有节点最终达到一致
  → 这就是「最终一致性」的含义

========================================
  面试速记
========================================

  1. CAP 三选二，但 P 必须保证 → 实际选 CP 或 AP
  2. CP 代表：ZooKeeper（分区时拒绝写入）
  3. AP 代表：Eureka（分区时继续服务，数据可能不一致）
  4. BASE 是对 ACID 的妥协：基本可用 + 软状态 + 最终一致
  5. 大多数互联网系统选 AP + 最终一致（如电商、社交）
  6. 金融系统选 CP + 强一致（如银行转账）
