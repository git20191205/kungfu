重点观察 — 读写并发的时间戳铁证：

  日志：
    [+ 26ms] 读线程-3 读取: 初始值
    [+ 26ms] 读线程-4 读取: 初始值
    [+ 26ms] 读线程-1 读取: 初始值
    [+ 26ms] 读线程-0 读取: 初始值
    [+ 26ms] 读线程-2 读取: 初始值
    [+248ms] 写线程-0 写入: ...
    [+356ms] 写线程-1 写入: ...

  三条铁证：
    1. 读读共享：5 个读线程时间戳全部是 +26ms → 完全并发
    2. 读写互斥：写线程-0 在 +248ms → 等所有读线程释放读锁后才开始
    3. 写写互斥：写线程-1 在 +356ms → 间隔 ~108ms → 写线程-0 释放后才获锁

  对比 synchronized/ReentrantLock：
    如果用 synchronized，5 个读线程也要排队
    → 5 × 模拟耗时 = 5 倍时间。读写锁让读并发，时间 = 1 倍。

重点观察 — 锁降级的 6 步流程：

  日志步骤：
    Step 1: 获取写锁
    Step 2: 修改数据 "旧数据" → "新数据"
    Step 3: 获取读锁（降级）— 写锁未释放时可以获取读锁
    Step 4: 释放写锁 — 此时仍持有读锁，其他写线程无法介入
    Step 5: 使用数据（持有读锁保证可见性）: 新数据
    Step 6: 释放读锁

  核心理解 — 为什么需要锁降级？
    如果直接释放写锁再获取读锁（Step 4 和 Step 3 互换），
    在释放写锁和获取读锁之间会有一个空隙：
      释放写锁 → （此时另一个写线程可能修改数据！）→ 获取读锁
    读到的就不是自己刚写入的值了。

    锁降级保证了"写完立即读"的一致性：
      持有写锁 → 加读锁 → 释放写锁 → 读数据 → 释放读锁
      从未有任何一个瞬间是"无锁"状态，其他写线程无法介入。

重点观察 — 锁升级死锁的原理：

  死锁链条：
    线程 A: readLock() ✓ → writeLock() ... 等 B 释放读锁
    线程 B: readLock() ✓ → writeLock() ... 等 A 释放读锁
    → 循环等待 → 死锁！

  这就是 ReentrantReadWriteLock 不支持锁升级的根本原因：
  写锁要求独占，但当前线程持有读锁时，其他线程可能也持有读锁，
  写锁永远等不到"所有读锁都释放"的那一刻（因为自己也占着一个）。

重点观察 — StampedLock 乐观读的两种路径：

  路径一（无干扰 → 零开销读取）：
    tryOptimisticRead() → stamp=256
    读数据 → x=1.0, y=2.0
    validate(256) = true → 直接使用，未加任何锁！

  路径二（有干扰 → 降级为悲观读）：
    tryOptimisticRead() → stamp=256
    读数据 → x=1.0, y=2.0
    [写线程修改了数据]
    validate(256) = false → 数据已脏！
    readLock() → 加悲观读锁
    重新读取 → x=10.0, y=20.0
    unlockRead() → 释放

  关键性能点：
    乐观读 tryOptimisticRead() 不是 CAS，只是读一个 volatile stamp 值
    → 零 CAS 操作、零锁竞争、零内存屏障开销
    这就是 StampedLock 在读多写少场景中性能最好的原因。

  注意事项（面试常问）：
    - StampedLock 不可重入 → 同一线程不能嵌套加锁
    - StampedLock 不支持 Condition → 需要条件等待则用 ReentrantReadWriteLock
    - 乐观读必须先拷贝到局部变量再 validate → 否则读到的值可能不一致

重点观察 — 四种锁的演进路线：

  synchronized → ReentrantLock → ReentrantReadWriteLock → StampedLock
  ┌──────┐    ┌──────────────┐    ┌──────────────────────┐    ┌─────────────┐
  │全互斥│ →  │可中断/公平锁│ →  │读读共享              │ →  │乐观读零开销 │
  └──────┘    └──────────────┘    └──────────────────────┘    └─────────────┘

  每一步演进都是为了"减少不必要的互斥"：
    - synchronized：读和读也互斥（浪费）
    - ReadWriteLock：读和读不互斥了，但读还是要加锁
    - StampedLock：读连锁都不用加了（乐观读）
