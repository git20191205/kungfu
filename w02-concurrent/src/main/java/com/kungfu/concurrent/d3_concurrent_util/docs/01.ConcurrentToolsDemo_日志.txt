========================================
  并发工具类 — CountDownLatch / CyclicBarrier / Semaphore
========================================

=== 一、CountDownLatch — 等待多个任务完成 ===
  场景：微服务健康检查，3 个服务全部就绪后系统才启动

  [主线程] 等待所有服务就绪... await()
  [257ms] UserService 健康检查通过 ✓  (耗时 235ms)  → countDown()
  [294ms] PaymentService 健康检查通过 ✓  (耗时 271ms)  → countDown()
  [517ms] OrderService 健康检查通过 ✓  (耗时 494ms)  → countDown()
  [517ms] 所有服务健康检查通过，系统启动！
  → 总耗时取决于最慢的服务（并行检查），而非三者之和

=== 二、CyclicBarrier — 多线程到齐后一起出发 ===
  场景：赛跑 — 3 个选手到齐后一起出发，跑 2 轮展示可重用

  [第1轮] Runner-2 准备就绪 → await()
  [第1轮] Runner-3 准备就绪 → await()
  [第1轮] Runner-1 准备就绪 → await()
  >>> 所有选手就位，GO！<<<
  [第1轮] Runner-1 起跑中...
  [第1轮] Runner-2 起跑中...
  [第1轮] Runner-3 起跑中...
  [第2轮] Runner-3 准备就绪 → await()
  [第2轮] Runner-1 准备就绪 → await()
  [第2轮] Runner-2 准备就绪 → await()
  >>> 所有选手就位，GO！<<<
  [第2轮] Runner-2 起跑中...
  [第2轮] Runner-3 起跑中...
  [第2轮] Runner-1 起跑中...
  → CyclicBarrier 可重用：同一个 barrier 跑了 2 轮

=== 三、Semaphore — 限流/资源池 ===
  场景：停车场 — 3 个车位，5 辆车

  Car-1 到达停车场，等待车位... (可用车位: 3)
  Car-1 停入车位 ← acquire()  (剩余车位: 2)
  Car-2 到达停车场，等待车位... (可用车位: 2)
  Car-2 停入车位 ← acquire()  (剩余车位: 1)
  Car-3 到达停车场，等待车位... (可用车位: 1)
  Car-3 停入车位 ← acquire()  (剩余车位: 0)
  Car-4 到达停车场，等待车位... (可用车位: 0)
  Car-5 到达停车场，等待车位... (可用车位: 0)
  Car-3 驶出车位 → release()  (剩余车位: 1)
  Car-4 停入车位 ← acquire()  (剩余车位: 0)
  Car-1 驶出车位 → release()  (剩余车位: 1)
  Car-5 停入车位 ← acquire()  (剩余车位: 0)
  Car-2 驶出车位 → release()  (剩余车位: 1)
  Car-4 驶出车位 → release()  (剩余车位: 2)
  Car-5 驶出车位 → release()  (剩余车位: 3)
  → 虽然有 5 辆车，但同一时刻最多只有 3 辆在停车场内

=== 四、三者对比表 ===

  ┌──────────┬──────────────────┬──────────────────┬──────────────────┐
  │   特性   │  CountDownLatch  │  CyclicBarrier   │    Semaphore     │
  ├──────────┼──────────────────┼──────────────────┼──────────────────┤
  │   作用   │ 等待N个任务完成  │ N个线程互相等待  │ 控制并发访问数量 │
  ├──────────┼──────────────────┼──────────────────┼──────────────────┤
  │ 计数方式 │ countDown()减1   │ await()自动减1   │ acquire()/release│
  ├──────────┼──────────────────┼──────────────────┼──────────────────┤
  │ 可重用   │ 否（一次性）     │ 是（自动重置）   │ 是（许可可归还） │
  ├──────────┼──────────────────┼──────────────────┼──────────────────┤
  │ 线程关系 │ 一等多           │ 多等多           │ 限流，无等待关系 │
  ├──────────┼──────────────────┼──────────────────┼──────────────────┤
  │ 底层实现 │ AQS 共享模式     │ ReentrantLock    │ AQS 共享模式     │
  │          │                  │ + Condition      │                  │
  └──────────┴──────────────────┴──────────────────┴──────────────────┘

=== 面试速记 ===
  1. CountDownLatch：一次性门闩，countDown() 减到 0 后 await() 放行，不可重用
  2. CyclicBarrier：可循环屏障，所有线程 await() 到齐后一起放行，可重用
  3. Semaphore：信号量/令牌桶，acquire() 获取许可，release() 归还，用于限流
  4. CountDownLatch 基于 AQS 共享模式，CyclicBarrier 基于 ReentrantLock + Condition
  5. Semaphore 基于 AQS 共享模式，支持公平/非公平两种策略
  6. 三者都不能替代 synchronized，它们解决的是线程间协调，而非互斥
