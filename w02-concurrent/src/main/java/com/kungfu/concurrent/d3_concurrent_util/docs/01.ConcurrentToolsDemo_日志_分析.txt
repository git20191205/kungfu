重点观察 — CountDownLatch 并行效果：

  各服务完成时间：
    UserService:    235ms
    PaymentService: 271ms
    OrderService:   494ms（最慢）

  总耗时 517ms ≈ 最慢服务 494ms + 少量调度开销（23ms）
  如果串行检查，总耗时 = 235 + 271 + 494 = 1000ms
  并行效率提升：(1000 - 517) / 1000 ≈ 48%

  → 完美体现了 CountDownLatch "一等多" 的核心价值：总耗时取决于最慢任务

重点观察 — CyclicBarrier 可重用：

  第 1 轮：Runner-2 → Runner-3 → Runner-1 到齐 → GO！
  第 2 轮：Runner-3 → Runner-1 → Runner-2 到齐 → GO！

  两轮到达顺序不同（线程调度随机），但都成功触发屏障动作
  → 同一个 barrier 对象跑了 2 轮，验证了 "Cyclic"（可循环）特性
  → 对比 CountDownLatch：减到 0 后无法重置，只能用一次

重点观察 — Semaphore 限流行为：

  时间线：
    Car-1,2,3 依次 acquire → 车位从 3→2→1→0
    Car-4,5 到达时车位为 0 → 阻塞等待
    Car-3 release → Car-4 立即 acquire
    Car-1 release → Car-5 立即 acquire
    最后 Car-2,4,5 依次 release → 车位恢复到 3

  全程车位数始终 ≥ 0 且 ≤ 3，限流约束未被破坏
  → Semaphore 的 acquire/release 对称性保证了资源不超卖

结论：所有实验效果清晰，代码无问题。
