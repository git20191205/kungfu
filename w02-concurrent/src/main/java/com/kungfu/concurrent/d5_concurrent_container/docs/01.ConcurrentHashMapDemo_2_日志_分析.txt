重点观察 — 1.7 vs 1.8 核心差异：

  锁粒度演进：
    1.7: Segment 级（默认 16 个 Segment）→ 并发度 = 16
    1.8: 桶头节点级 → 并发度 = 数组长度（可达数百上千）

  put 操作：
    1.7: hash → 定位 Segment → ReentrantLock → 头插法
    1.8: hash → 桶为空 CAS 无锁写入 / 非空 synchronized 锁头节点 → 尾插法

  冲突处理升级：
    1.7: 只有链表 → O(n) 查询
    1.8: 链表 ≥8 且数组 ≥64 → 红黑树 → O(log n) 查询

重点观察 — size() 的巧妙设计：

  1.8 的 baseCount + CounterCell[] 本质是 LongAdder 思想：
    - 低竞争：CAS 直接更新 baseCount（一次操作完成）
    - 高竞争：分散到 CounterCell 数组各槽位，减少 CAS 冲突
    - 读 size()：baseCount + 所有 Cell 之和

  这比 1.7 的"尝试 2 次不加锁，失败则全锁"策略更优：
    - 写操作永远无锁（CAS）
    - 读操作也无锁（求和）
    - 代价是 size() 结果可能有微小偏差（弱一致性）

结论：修复后所有实验效果清晰，代码无问题。HashMap 不安全演示保留了数据丢失效果，
      超时保护解决了死循环卡死问题。
