重点观察 — 4 种拒绝策略对比：

  测试条件：core=2, max=2, queue=2，提交 5 个任务
  容量 = max + queue = 4，第 5 个任务必然触发拒绝

  ┌──────────────────────┬──────────────┬──────────────────────────────┐
  │       拒绝策略        │   实际执行    │         对业务的影响          │
  ├──────────────────────┼──────────────┼──────────────────────────────┤
  │ AbortPolicy          │ 1,2,3,4       │ 任务5 直接异常，调用方感知   │
  ├──────────────────────┼──────────────┼──────────────────────────────┤
  │ CallerRunsPolicy     │ 1,2,3,4,5     │ 任务5 由 main 执行，无丢失   │
  │                       │              │ → 天然反压：main 被阻塞      │
  ├──────────────────────┼──────────────┼──────────────────────────────┤
  │ DiscardPolicy        │ 1,2,3,4       │ 任务5 静默丢失，调用方无感知 │
  ├──────────────────────┼──────────────┼──────────────────────────────┤
  │ DiscardOldestPolicy  │ 1,3,4,5       │ 任务2 被丢弃（队列最老）     │
  └──────────────────────┴──────────────┴──────────────────────────────┘

重点观察 — DiscardPolicy 日志看似未触发拒绝：

  日志显示 "任务4 提交成功" 和 "任务5 提交成功"，看似没触发拒绝。

  真相：这是时序问题。
  - 任务1 被核心线程立即取走执行（"任务1 执行中" 出现在 "任务4 提交" 之前）
  - 等到任务4、5 提交时，核心线程已腾出位置
  - 但最终只有 1,2,3,4 四个任务实际执行 → 任务5 确实被静默丢弃了

重点观察 — 执行流程（小节三）：

  core → queue → max → reject 完整流程：

    任务1 → poolSize=1（core 未满，创建核心线程）
    任务2 → poolSize=2（core 已满）
    任务3 → queueSize=1（入队）
    任务4 → queueSize=2（队列已满）
    任务5 → poolSize=3（扩展非核心线程）
    任务6 → poolSize=4（max 已达上限）
    任务7 → 拒绝！（pool + queue 均满）

  完美体现了"先核心、后队列、再扩容、最后拒绝"的执行优先级。

重点观察 — DiscardOldestPolicy 丢弃追踪：

  日志执行了：任务1、任务3、任务4、任务5
  未出现 "任务2 执行中" → 任务2 被丢弃

  过程还原：
    任务1 → 核心线程1
    任务2 → 核心线程2（或入队）
    任务3 → 入队 [队头: 任务2(或3)]
    任务4 → 入队 [队列满]
    任务5 → 触发 DiscardOldestPolicy → 丢弃队头（任务2）→ 任务5 入队

结论：所有实验效果清晰，代码无问题。
