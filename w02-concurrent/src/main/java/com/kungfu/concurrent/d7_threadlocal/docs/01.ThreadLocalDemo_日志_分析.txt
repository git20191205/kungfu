重点观察 — 错误用法日志的"反常"：

  日志片段：
    pool-1-thread-1 设置了值，但没有 remove()
    pool-1-thread-2 复用线程，读到残留值: null

  问题：既然"没有 remove"，为什么 thread-2 读到 null 而非残留值？

  真相：thread-2 ≠ thread-1（名字不同 → 不是同一个线程）
    - 线程池可能是 core=2 的池，所以 thread-1 和 thread-2 是两个独立线程
    - thread-2 从没设置过值 → 读到 null 属于正常
    - 如果线程池 core=1，thread-1 执行完再次执行（线程复用），
      才会读到残留值

  → 此处的"残留值: null"是代码输出文字的误导，实际并没有演示到真正的残留。
     教学效果略弱于预期，但并不影响结论的正确性（即"必须 remove()"）。
     可在后续优化中改用 core=1 的线程池来稳定复现残留。

重点观察 — 内存泄漏根因（小节三知识点）：

  ┌─────────────┬──────────────┬──────────────────────────────┐
  │   引用位置  │   引用类型   │         GC 回收?             │
  ├─────────────┼──────────────┼──────────────────────────────┤
  │ Entry.key   │ WeakReference│ 是（ThreadLocal 变量置 null后）│
  ├─────────────┼──────────────┼──────────────────────────────┤
  │ Entry.value │ 强引用       │ 否（需主动 remove）           │
  ├─────────────┼──────────────┼──────────────────────────────┤
  │ Thread      │ 强引用持 Map │ 否（线程池中长期存在）        │
  └─────────────┴──────────────┴──────────────────────────────┘

  泄漏链：Thread(存活) → Map → Entry → value(强引用) → 大对象
       即使 key 已被 GC，value 仍无法回收 → 内存泄漏

  解决方案：用完必须 remove()！

重点观察 — InheritableThreadLocal 的线程池陷阱：

  日志：
    任务1 继承到: 第一次任务的值
    任务2 读到: 第一次任务的值 (线程被复用，不会重新继承新值！)

  核心原理：
    - 继承时机：Thread.init() 时从 parent 复制 inheritableThreadLocals
    - 继承发生在"线程创建时"，而非"任务执行时"
    - 线程池的线程只创建一次，后续任务复用线程 → 不会重新继承

  → 这是线程池 + InheritableThreadLocal 的经典坑，阿里的 TransmittableThreadLocal
    （TTL）就是为解决这个问题而生。

重点观察 — 用户上下文场景：

  3 个请求虽然执行顺序是 [请求2 → 请求1 → 请求3]（线程调度不确定），
  但每个请求的 Service/DAO 层读到的都是各自请求的用户 → 零串扰。

  这正是 Spring RequestContextHolder、Spring Security SecurityContextHolder、
  MyBatis SqlSessionHolder 的实现基础。

结论：所有实验效果清晰，代码基本无问题。小节三的"错误用法"演示因线程池配置
      未能稳定复现残留值，建议后续优化改用 core=1 的池。
