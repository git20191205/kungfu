重点观察 — 残留值复现的关键证据：

  实验一日志：
    [任务1] pool-1-thread-1 set: 敏感数据-用户A
    [任务2] pool-1-thread-1 get: 敏感数据-用户A    ← 同一线程名！

  两个关键点：
    1. 线程名相同（pool-1-thread-1）→ 证明是同一个 Thread 对象
    2. 任务2 从未 set 过值，却读到了 "敏感数据-用户A" → 这就是残留值泄露

  这在真实 Web 场景中意味着：
    - 用户B 的请求复用了处理用户A 的线程
    - 用户B 能读到用户A 的隐私数据（userId、token、权限等）
    - 这不是理论风险，而是线上真实发生过的安全事故

重点观察 — 线程池编号的小细节：

  三个实验分别输出了 pool-1、pool-2、pool-3 的线程名，
  这是因为每次 Executors.newFixedThreadPool() 创建新池时，
  JVM 内部的 poolNumber 原子计数器会自增。

  印证了：ThreadLocal 的隔离性是"Thread 对象级"的，
  不同线程池之间的 Thread 完全独立，不会互相干扰。

结论：修复完美生效。core=1 保证线程复用，残留值 100% 稳定复现。
