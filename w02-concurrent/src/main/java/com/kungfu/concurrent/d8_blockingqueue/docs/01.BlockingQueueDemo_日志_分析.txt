重点观察 — 四组 API 的边界行为：

  ┌──────────────┬──────────────────────┬──────────────────────────────────┐
  │   方式       │      队列满时        │        队列空时                  │
  ├──────────────┼──────────────────────┼──────────────────────────────────┤
  │ 抛异常       │ IllegalStateException│ NoSuchElementException           │
  ├──────────────┼──────────────────────┼──────────────────────────────────┤
  │ 返回特殊值   │ offer→false          │ poll→null                        │
  ├──────────────┼──────────────────────┼──────────────────────────────────┤
  │ 阻塞         │ put 阻塞等待        │ take 阻塞等待                    │
  ├──────────────┼──────────────────────┼──────────────────────────────────┤
  │ 超时         │ offer→false(500ms后) │ poll→null(500ms后)               │
  └──────────────┴──────────────────────┴──────────────────────────────────┘

  实际开发中的选择：
    - 抛异常：几乎不用（调用方需 try-catch，代码难看）
    - 返回值：非阻塞场景（快速失败，不想等）
    - 阻塞：生产者-消费者模型（put/take 是核心方法）
    - 超时：线程池内部（offer + 拒绝策略配合）

重点观察 — Array vs Linked 性能对比：

  ┌──────────────────────┬──────────────────┬────────────────────┬──────────┐
  │                      │ ArrayBlockingQueue│ LinkedBlockingQueue│   倍率   │
  ├──────────────────────┼──────────────────┼────────────────────┼──────────┤
  │ 耗时（100000 元素）  │       38 ms      │       21 ms        │ 1.8x 快 │
  ├──────────────────────┼──────────────────┼────────────────────┼──────────┤
  │ 锁                   │ 1 把（互斥）     │ 2 把（并行）       │          │
  └──────────────────────┴──────────────────┴────────────────────┴──────────┘

  为什么 Linked 更快？
    ArrayBlockingQueue：一把 ReentrantLock → put 和 take 互斥
      生产者 put 时，消费者 take 必须等锁释放

    LinkedBlockingQueue：两把锁 putLock + takeLock → put 和 take 可并行
      生产者操作链表尾部，消费者操作链表头部，互不干扰

  但 Array 也有优势：
    - 内存预分配，无 GC 压力（链表每个节点都是一次 new）
    - 有界强制（LinkedBlockingQueue 默认 Integer.MAX_VALUE ≈ 无界，有 OOM 风险）

重点观察 — SynchronousQueue 交付时序：

  日志时序：
    [生产者] 准备交付: 任务A (将阻塞直到消费者取走)
    [消费者] 收到: 任务A
    [生产者] 已交付: 任务A

  每一轮都是严格的三步：
    1. 生产者 put → 阻塞
    2. 消费者 take → 消费者拿到，生产者解除阻塞
    3. 生产者继续下一轮

  核心理解：SynchronousQueue 容量为 0，不是"存 1 个就满了"，
  而是"压根不存"——put 和 take 必须配对出现，实现直接交付（hand-off）。

  这就是 CachedThreadPool 的工作方式：
    提交任务 → SynchronousQueue.offer → 没有线程 take → 创建新线程

重点观察 — DelayQueue 时间精度：

  ┌────────┬──────────────┬──────────────┬──────────────┐
  │ 任务   │ 预期延时     │ 实际延时     │    误差      │
  ├────────┼──────────────┼──────────────┼──────────────┤
  │ 任务A  │   500 ms     │   503 ms     │   +3 ms      │
  ├────────┼──────────────┼──────────────┼──────────────┤
  │ 任务B  │  1000 ms     │  1016 ms     │  +16 ms      │
  ├────────┼──────────────┼──────────────┼──────────────┤
  │ 任务C  │  1500 ms     │  1514 ms     │  +14 ms      │
  └────────┴──────────────┴──────────────┴──────────────┘

  误差 3~16ms 属于正常范围（线程调度 + 系统时钟精度）。
  DelayQueue 内部使用 PriorityQueue 按 getDelay() 排序，
  take() 时 peek 队头元素，如果 delay > 0 则 await(delay)。

  典型应用场景：
    - 订单超时取消（30 分钟未付款自动关闭）
    - 缓存过期清理（Redis 本地缓存的兜底方案）
    - 重试队列（失败后延时 N 秒重试）

重点观察 — 队列类型决定线程池行为：

  这张对应关系表揭示了一个重要设计思想：
  线程池的"性格"不是由 ThreadPoolExecutor 本身决定的，
  而是由传入的 BlockingQueue 类型决定的。

  验证串联（回顾 D4 — ThreadPoolDemo）：
    - FixedThreadPool 用 LinkedBlockingQueue(无界) → 永不触发 max → 永不拒绝 → OOM 风险
    - CachedThreadPool 用 SynchronousQueue(零容量) → 每个任务都要线程 → 线程爆炸风险
    - 阿里规范推荐 ThreadPoolExecutor + ArrayBlockingQueue(有界) → 最可控

结论：所有实验效果清晰，代码无问题。四组 API、双锁并行、直接交付、延时排序
      四个核心概念全部通过实验验证，队列与线程池的关系形成了与 D4 的知识串联。
