========================================
  阻塞队列 — 生产者消费者的基石
========================================

=== 一、BlockingQueue 接口方法对比 ===

  ┌──────────────┬──────────────┬──────────────┬──────────────┐
  │   方式       │   插入       │   移除       │   检查       │
  ├──────────────┼──────────────┼──────────────┼──────────────┤
  │ 抛异常       │ add(e)       │ remove()     │ element()    │
  ├──────────────┼──────────────┼──────────────┼──────────────┤
  │ 返回特殊值   │ offer(e)     │ poll()       │ peek()       │
  ├──────────────┼──────────────┼──────────────┼──────────────┤
  │ 阻塞         │ put(e)       │ take()       │ —            │
  ├──────────────┼──────────────┼──────────────┼──────────────┤
  │ 超时         │ offer(e,t,u) │ poll(t,u)    │ —            │
  └──────────────┴──────────────┴──────────────┴──────────────┘

  【第一组：抛异常 — add / remove / element】
    add("A"), add("B") 成功，队列已满
    add("C") → 抛出 IllegalStateException: Queue full
    element() → A (查看队头，不移除)
    remove()  → A
    remove()  → B
    remove()  → 抛出 NoSuchElementException (队列为空)

  【第二组：返回特殊值 — offer / poll / peek】
    offer("X") → true
    offer("Y") → true
    offer("Z") → false (队列满，返回 false)
    peek()     → X (查看队头，不移除)
    poll()     → X
    poll()     → Y
    poll()     → null (队列空，返回 null)

  【第三组：阻塞 — put / take】
    put() 在队列满时阻塞，take() 在队列空时阻塞
    (生产者-消费者模型的核心方法，后续 SynchronousQueue 中详细演示)

  【第四组：超时 — offer(e,t,u) / poll(t,u)】
    offer("TIMEOUT", 500ms) → true (队列未满，立即成功)
    offer("TIMEOUT2", 500ms) → true (队列未满，立即成功)
    offer("TIMEOUT3", 500ms) → false (队列满，等待 500ms 后返回 false)
    poll(500ms) → null (队列空，等待 500ms 后返回 null)

=== 二、ArrayBlockingQueue vs LinkedBlockingQueue ===

  特性对比：
  ┌──────────────────┬─────────────────────────┬─────────────────────────────┐
  │     特性         │  ArrayBlockingQueue     │  LinkedBlockingQueue        │
  ├──────────────────┼─────────────────────────┼─────────────────────────────┤
  │ 底层结构         │ 数组                    │ 链表                        │
  ├──────────────────┼─────────────────────────┼─────────────────────────────┤
  │ 是否有界         │ 必须指定容量（有界）     │ 可选（默认 Integer.MAX_VALUE）│
  ├──────────────────┼─────────────────────────┼─────────────────────────────┤
  │ 锁的数量         │ 一把锁（ReentrantLock） │ 两把锁（putLock + takeLock） │
  ├──────────────────┼─────────────────────────┼─────────────────────────────┤
  │ 吞吐量           │ 较低（put/take 互斥）   │ 较高（put/take 可并行）      │
  ├──────────────────┼─────────────────────────┼─────────────────────────────┤
  │ 内存             │ 预分配数组，无 GC 压力  │ 动态分配节点，有 GC 压力     │
  └──────────────────┴─────────────────────────┴─────────────────────────────┘

  吞吐量测试（100000 个元素，容量 1024）：
    ArrayBlockingQueue  耗时: 38 ms
    LinkedBlockingQueue 耗时: 21 ms

  说明：LinkedBlockingQueue 使用两把锁（putLock + takeLock），
  生产者和消费者可以并行操作，通常吞吐量更高。

=== 三、SynchronousQueue — 直接交付 ===

  SynchronousQueue 特点：
    - 容量为 0，不存储任何元素
    - put() 阻塞直到另一个线程调用 take()
    - 实现生产者与消费者之间的直接交付（hand-off）
    - Executors.newCachedThreadPool() 使用它

    [生产者] 准备交付: 任务A (将阻塞直到消费者取走)
    [消费者] 收到: 任务A
    [生产者] 已交付: 任务A
    [生产者] 准备交付: 任务B (将阻塞直到消费者取走)
    [消费者] 收到: 任务B
    [生产者] 已交付: 任务B
    [生产者] 准备交付: 任务C (将阻塞直到消费者取走)
    [消费者] 收到: 任务C
    [生产者] 已交付: 任务C

  可以观察到：生产者 put 后必须等消费者 take 才能继续，实现了直接交付。

=== 四、DelayQueue — 延时任务 ===

  DelayQueue 特点：
    - 元素必须实现 Delayed 接口（getDelay + compareTo）
    - take() 只能取出延时到期的元素
    - 内部按到期时间排序（PriorityQueue）
    - 应用场景：定时任务、缓存过期、订单超时

    放入 3 个任务：延时 500ms、1000ms、1500ms
    取出顺序按延时到期时间排列：

    取出: 任务A (延时500ms) (已过 503ms)
    取出: 任务B (延时1000ms) (已过 1016ms)
    取出: 任务C (延时1500ms) (已过 1514ms)

=== 五、BlockingQueue 与线程池的关系 ===

  ┌──────────────────────┬──────────────────────┬──────────────────────────────┐
  │   队列类型           │   对应线程池          │   行为特点                   │
  ├──────────────────────┼──────────────────────┼──────────────────────────────┤
  │ LinkedBlockingQueue  │ FixedThreadPool      │ 无界队列，不会创建非核心线程  │
  ├──────────────────────┼──────────────────────┼──────────────────────────────┤
  │ SynchronousQueue     │ CachedThreadPool     │ 直接交付，总是创建新线程      │
  ├──────────────────────┼──────────────────────┼──────────────────────────────┤
  │ ArrayBlockingQueue   │ 自定义线程池          │ 有界队列，可触发拒绝策略      │
  ├──────────────────────┼──────────────────────┼──────────────────────────────┤
  │ DelayedWorkQueue     │ ScheduledThreadPool  │ 按延时排序，定时执行          │
  └──────────────────────┴──────────────────────┴──────────────────────────────┘

  关键理解：
    1. 无界队列（LinkedBlockingQueue）→ maximumPoolSize 形同虚设
       任务永远能入队，永远不会触发创建非核心线程，也不会触发拒绝策略
       风险：任务堆积可能导致 OOM

    2. 直接交付（SynchronousQueue）→ 每个任务都交给线程处理
       没有空闲线程就创建新线程（直到 maximumPoolSize）
       适合大量短生命周期的异步任务

    3. 有界队列（ArrayBlockingQueue）→ 最可控的方式
       核心线程满 → 入队 → 队列满 → 创建非核心线程 → 都满 → 拒绝策略
       阿里巴巴规范推荐使用 ThreadPoolExecutor + 有界队列

========================================
  面试速记
========================================

  1. BlockingQueue 四组 API：抛异常/返回值/阻塞/超时
  2. ArrayBlockingQueue 有界 + 一把锁; LinkedBlockingQueue 可选有界 + 两把锁
  3. SynchronousQueue 容量为 0，Executors.newCachedThreadPool 使用它
  4. DelayQueue 按延时时间排序，用于定时任务
  5. 线程池的行为很大程度取决于使用的队列类型
