重点观察 — ABA 问题对比：

  AtomicInteger（无保护）：
    线程2: 1→2→1（值回到原样）
    线程1: CAS(1,3)=true → ABA 未被发现，更新成功

  AtomicStampedReference（版本号保护）：
    线程2: 值 1→2→1，但版本号 1→2→3
    线程1: CAS(值=1, 版本=1) → 值匹配但版本不匹配 → false

  关键理解：ABA 问题在大多数计数场景无害，但在以下场景致命：
    - 链表/栈的 CAS 操作（节点被删除又插入，指针语义已变）
    - 金融等需要追踪完整变更历史的场景

重点观察 — LongAdder 性能对比：

  ┌──────────────┬──────────────┬──────────────┬──────────────┐
  │              │  AtomicLong  │  LongAdder   │     倍率     │
  ├──────────────┼──────────────┼──────────────┼──────────────┤
  │ 耗时         │    117 ms    │     22 ms    │   5.3x 快    │
  ├──────────────┼──────────────┼──────────────┼──────────────┤
  │ 结果正确性   │  10,000,000  │  10,000,000  │   都精确     │
  └──────────────┴──────────────┴──────────────┴──────────────┘

  本机 28 核 CPU，10 线程竞争下 LongAdder 已快 5.3 倍。
  线程数越多、竞争越激烈，差距越大（可达 10x 以上）。

  原因：AtomicLong 10 个线程全部 CAS 同一个 value → 大量自旋浪费
        LongAdder 将竞争分散到 base + Cell[] → 每个线程各写各的 Cell

  适用场景选择：
    - 需要精确实时值（如余额）→ AtomicLong
    - 只需最终汇总值（如计数器、统计）→ LongAdder

知识串联：

  CAS 是并发编程的底层基石：
    volatile → 可见性
    CAS → 原子性（无锁）
    synchronized/Lock → 互斥性（有锁）

  CAS 的应用处处可见：
    - AtomicInteger/LongAdder 直接使用 CAS
    - ConcurrentHashMap 1.8 空桶写入用 CAS
    - AQS 的 state 更新用 CAS
    - synchronized 偏向锁→轻量级锁升级用 CAS

结论：所有实验效果清晰，代码无问题。
