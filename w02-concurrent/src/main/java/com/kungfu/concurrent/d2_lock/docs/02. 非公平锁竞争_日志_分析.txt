02. 非公平锁竞争复现 日志分析
========================================

整体结果：全部通过，非公平锁插队现象稳定复现。

重点分析 — 二补充（10 线程 × 3 轮）：

  公平锁获取顺序：
    F-0, F-1, F-2, F-0, F-3, F-1, F-4, F-2, F-8, F-6, ...
    → 各线程交替出现，没有同一线程连续获取

  非公平锁获取顺序：
    U-0, U-0, U-0, U-2, U-2, U-2, U-4, U-4, U-4, ...
    → 每个线程连续获取 3 次后才轮到下一个

┌──────────────┬────────────────────────────────┬──────────────────────────────────┐
│     指标     │             公平锁             │             非公平锁             │
├──────────────┼────────────────────────────────┼──────────────────────────────────┤
│ 获取顺序     │ 交替轮转（近似 FIFO）          │ 同一线程连续 3 次获取            │
├──────────────┼────────────────────────────────┼──────────────────────────────────┤
│ 插队次数     │ 0 次                           │ 20 次（30次中有20次连续重复）    │
├──────────────┼────────────────────────────────┼──────────────────────────────────┤
│ 插队率       │ 0%                             │ 67%                              │
└──────────────┴────────────────────────────────┴──────────────────────────────────┘

为什么非公平锁会连续 3 次获取？

  线程 U-0 执行流程：
    1. lock() → CAS 成功 → 获取锁 → 执行 → unlock()
    2. unlock() 后立刻再次 lock() → 此时队列中的线程正在被 unpark 唤醒
    3. unpark 唤醒需要：内核态→用户态切换 + 线程调度，耗时 ~几微秒
    4. 而 U-0 的 CAS 只需 ~几纳秒，远快于队列线程被唤醒
    5. 所以 U-0 连续 3 轮都能抢在队列线程之前获取锁

  时间线示意：
    U-0 unlock → U-0 CAS(成功) → U-0 执行第2轮
                  ↑ 此时队列中 U-1 还在被 unpark 唤醒中...

  这就是非公平锁"吞吐量高"的原因：
    - 避免了线程切换的开销（同一线程连续执行，CPU 缓存命中率高）
    - 代价是可能造成饥饿（排队线程长时间得不到锁）

与原始实验（5线程×1次）的对比：

  原始实验：5 个线程各获取 1 次锁，竞争窗口只有 1 轮
    → 恰好有序（0→1→2→3→4），未复现乱序

  补充实验：10 个线程各获取 3 次锁，竞争窗口放大
    → 稳定复现 67% 插队率，效果一目了然

结论：补充实验设计合理，非公平锁插队现象稳定复现，无需进一步调整。

新增方法 demonstrateUnfairLockReorder() 已加入 ReentrantLockDemo.java，运行结果：

  - 公平锁：F-0, F-1, F-2, F-0, F-6, F-1... — 近似轮转，每个线程交替获取
  - 非公平锁：U-0, U-0, U-0, U-1, U-1, U-1... — 同一线程连续 3 次获取（20 次插队），因为刚释放锁的线程立即 CAS 再次获取，抢在队列中排队线程被 unpark 之前成功"插队"

  核心原理：非公平锁 lock() 时先直接 CAS 尝试获取，不检查队列。而公平锁会先调用 hasQueuedPredecessors() 检查队列中是否有等待线程，有则乖乖排队。
