   ReentrantLock 与 AQS 原理 Demo
========================================

=== 一、ReentrantLock vs synchronized 对比 ===

  ┌──────────┬────────────────┬────────────────────┐
  │   特性   │  synchronized  │   ReentrantLock    │
  ├──────────┼────────────────┼────────────────────┤
  │ 实现方式 │ JVM 内置关键字 │ Java API (AQS)     │
  ├──────────┼────────────────┼────────────────────┤
  │ 锁获取   │ 自动获取       │ 手动 lock/unlock   │
  ├──────────┼────────────────┼────────────────────┤
  │ 公平性   │ 非公平         │ 可选公平/非公平    │
  ├──────────┼────────────────┼────────────────────┤
  │ 条件变量 │ 单一 wait/notify│ 多个 Condition     │
  ├──────────┼────────────────┼────────────────────┤
  │ 可中断   │ 不可中断       │ lockInterruptibly  │
  ├──────────┼────────────────┼────────────────────┤
  │ 自动释放 │ 是（退出同步块）│ 否（需 finally）   │
  └──────────┴────────────────┴────────────────────┘

=== 二、公平锁 vs 非公平锁 ===

  【公平锁 ReentrantLock(true)】— 按 FIFO 顺序获取锁：
    公平锁 - Fair-Thread-0 获取到锁
    公平锁 - Fair-Thread-1 获取到锁
    公平锁 - Fair-Thread-2 获取到锁
    公平锁 - Fair-Thread-3 获取到锁
    公平锁 - Fair-Thread-4 获取到锁

  【非公平锁 ReentrantLock(false)】— 允许插队，顺序不保证：
    非公平锁 - Unfair-Thread-0 获取到锁
    非公平锁 - Unfair-Thread-1 获取到锁
    非公平锁 - Unfair-Thread-2 获取到锁
    非公平锁 - Unfair-Thread-3 获取到锁
    非公平锁 - Unfair-Thread-4 获取到锁

=== 三、可中断锁 ===

  Thread-A: 获取到锁，将持有 3 秒...
  Thread-B: 尝试以 lockInterruptibly() 获取锁...
  Main: 中断 Thread-B
  Thread-B: 等待锁时被中断！捕获 InterruptedException
  Thread-B: 可以执行清理逻辑或退出，而不会一直阻塞
  Thread-A: 释放锁

=== 四、Condition 条件变量 — 精确唤醒 ===

  精确唤醒链：A → B → C → A

  Thread-A: 第 1 轮执行（flag=1），唤醒 B
  Thread-B: 执行（被 A 唤醒，flag=2），唤醒 C
  Thread-C: 执行（被 B 唤醒，flag=3），唤醒 A
  Thread-A: 第 2 轮执行（被 C 唤醒），精确唤醒验证成功！

  执行顺序：A(第1轮) → B → C → A(第2轮)  ✓ 精确唤醒成功

=== 五、AQS 原理 ===

  AQS（AbstractQueuedSynchronizer）是 JUC 锁和同步工具的基础框架。

  【state 变量】
  ┌────────────────────────────────────────────────────────┐
  │ volatile int state                                     │
  │                                                        │
  │ - ReentrantLock: 0=未锁定, >=1=锁定(值表示重入次数)    │
  │ - Semaphore:     state 表示可用许可数                   │
  │ - CountDownLatch: state 表示剩余计数                    │
  └────────────────────────────────────────────────────────┘

  【CLH 队列结构（双向链表）】

         ┌──────────┐    ┌──────────┐    ┌──────────┐
  head → │  Node    │ ←→ │  Node    │ ←→ │  Node    │ ← tail
         │ (哨兵)   │    │ Thread-1 │    │ Thread-2 │
         │ status=0 │    │ SIGNAL   │    │ SIGNAL   │
         └──────────┘    └──────────┘    └──────────┘

  - head 指向哨兵节点（持有锁的线程或虚拟节点）
  - tail 指向最后一个等待节点
  - 每个 Node 通过 prev/next 双向链接
  - waitStatus: SIGNAL(-1) 表示后继节点需要被唤醒

  【acquire（加锁）流程】
  ┌─────────────────────────────────────────────────────┐
  │ 1. tryAcquire(arg) 尝试 CAS 修改 state             │
  │    ├─ 成功 → 设置 exclusiveOwnerThread，返回       │
  │    └─ 失败 → 进入步骤 2                             │
  │ 2. addWaiter() 将当前线程封装为 Node 加入 CLH 队列  │
  │ 3. acquireQueued() 自旋：                           │
  │    ├─ 前驱是 head → 再次 tryAcquire()               │
  │    │  ├─ 成功 → 设置自己为 head，返回               │
  │    │  └─ 失败 → park 挂起                           │
  │    └─ 前驱非 head → park 挂起                       │
  └─────────────────────────────────────────────────────┘

  【release（解锁）流程】
  ┌─────────────────────────────────────────────────────┐
  │ 1. tryRelease(arg) 修改 state                       │
  │    ├─ state=0 → 清除 exclusiveOwnerThread           │
  │    └─ state>0 → 仍持有锁（重入未完全释放）          │
  │ 2. 若 state=0，unpark 唤醒 head 的后继节点          │
  │ 3. 被唤醒的线程从 acquireQueued() 中继续自旋        │
  └─────────────────────────────────────────────────────┘

========================================
  面试速记
========================================

  1. ReentrantLock vs synchronized：
     - 都是可重入锁，ReentrantLock 功能更丰富
     - ReentrantLock 支持公平锁、可中断、多条件变量
     - synchronized 由 JVM 优化（偏向锁→轻量级锁→重量级锁）

  2. 公平锁 vs 非公平锁：
     - 公平锁：FIFO 顺序，不会饥饿，吞吐量较低
     - 非公平锁（默认）：允许插队，吞吐量高，可能饥饿

  3. AQS 核心三要素：
     - state（volatile int）：同步状态
     - CLH 队列（双向链表）：等待线程排队
     - CAS + park/unpark：线程安全的阻塞与唤醒

  4. Condition 精确唤醒：
     - 一个 Lock 可创建多个 Condition
     - await/signal 比 wait/notify 更灵活精确
     - 典型场景：生产者-消费者、顺序执行控制

  5. ReentrantLock 使用注意：
     - 必须在 finally 中 unlock()，防止死锁
     - lock() 不要写在 try 块内（避免未获取锁却执行 unlock）
     - 推荐用法：lock.lock(); try { ... } finally { lock.unlock(); }
