重点观察 — 非公平锁未体现乱序：

  公平锁：  0 → 1 → 2 → 3 → 4（严格 FIFO，符合预期）
  非公平锁：0 → 1 → 2 → 3 → 4（本次恰好有序）

  原因分析：
  - 5 个线程启动间隔极短，线程到达 lock() 的时间差不够大
  - 非公平锁的"插队"发生在：新线程恰好在锁刚释放时 CAS 成功，抢在队列中排队线程之前
  - 只有 5 个线程 × 1 次获取，竞争窗口太小，乱序概率低

  是否需要调整：
  - 不影响正确性，非公平锁"允许"乱序而非"保证"乱序
  - 如需稳定复现乱序效果，可以增加线程数(如20个)并让每个线程多次获取锁
  - 当前效果已足够说明 API 用法，建议暂不调整

重点观察 — 可中断锁时序：

  Thread-A 持有锁 → Thread-B lockInterruptibly() 阻塞
  → main 中断 Thread-B → Thread-B 立即捕获 InterruptedException
  → Thread-A 3秒后释放锁

  关键对比：如果用普通 lock()，Thread-B 无法响应中断，只能死等。
  lockInterruptibly() 允许外部中断等待，避免线程永久阻塞。

重点观察 — Condition 精确唤醒链：

  flag=1 → A 执行，signal conditionB → flag=2
  flag=2 → B 执行，signal conditionC → flag=3
  flag=3 → C 执行，signal conditionA → flag=1
  flag=1 → A 第2轮执行，验证成功

  核心价值：synchronized 只有一个 wait/notify，无法控制唤醒哪个线程。
  ReentrantLock 的多个 Condition 实现了精确唤醒，这是两者最重要的差异之一。

结论：所有实验效果清晰，代码无问题。非公平锁乱序未复现属于正常概率现象，不影响教学目的。
